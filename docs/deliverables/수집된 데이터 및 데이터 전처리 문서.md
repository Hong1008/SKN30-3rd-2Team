# WorkShield 데이터 수집 및 전처리 파이프라인 명세서

본 문서는 WorkShield 프로젝트(프리랜서 및 수급사업자 용역계약 표준 대비 검토 시스템)의 근간이 되는 데이터의 수집 출처와 전처리(Preprocessing) 파이프라인을 체계적으로 명세하기 위해 작성되었습니다. 향후 검토 대상 계약서의 도메인이 확장되거나 파싱(Parsing) 알고리즘이 고도화될 경우, 본 명세서가 시스템 확장의 기준이 됩니다.

---

## 1. 개요 (Overview)

### 1.1 데이터 관리의 핵심 원칙 (Core Principles)
> **"단일 진실 원천(Single Source of Truth)은 Git으로 형상 관리되는 정규화된 JSON 파일이며, 모든 검색 인덱스(SQLite, Chroma DB)는 스크립트를 통해 온디맨드(On-demand)로 재생성되어야 한다."**

WorkShield 시스템은 운영 및 평가 환경의 완벽한 재현성(Reproducibility)을 보장하기 위해, 대용량 바이너리 형태의 DB 파일을 Git 형상 관리 대상에서 제외합니다. 대신, 휴먼 리더블(Human-readable)하고 코드 리뷰(Diff)가 용이한 정규화된 `JSON` 파일만을 추적합니다. 이를 통해 팀 내 모든 개발자는 `just build-db` 명령어 단 1회 실행으로 원본 JSON에서부터 완벽하게 동일한 벡터 검색 환경을 로컬에 자동 구축할 수 있습니다.

## 2. 데이터 수집 (Data Collection)

본 프로젝트는 법률적 검토의 정확도와 공신력을 확보하기 위해, 국가 기관에서 공식 제정 및 배포한 표준계약서와 가이드라인을 준거 데이터(Ground Truth)로 채택하였습니다.

### 2.1 수집 대상 및 선정 기준 (Criteria for Collection)
본 시스템의 목적은 하도급 계약 및 프리랜서 위수탁 계약에 있어 구조적 열위에 놓인 당사자('을')의 권익을 보호하는 것입니다. 따라서 사적 기업의 임의적 양식을 배제하고, 가장 공정하고 법률적 기준이 되는 국가 기관의 문서를 수집하였습니다. `data/01_raw/`에 적재된 원본 데이터의 명세는 다음과 같습니다.

1. **공정거래위원회 제정 표준하도급계약서 (총 4종)**
   - `221228_상용소프트웨어_공급개발구축업.hwp` (SI 하도급, 2022년)
   - `221228_상용소프트웨어_유지관리업종.hwp` (SM 하도급, 2022년)
   - `251221_상용소프트웨어공급개발구축업종(비밀유지계약서_통합_및_안전등_추가).hwp` (SI 2025년 최신 개정판)
   - `251221_상용소프트웨어유지관리업종(비밀유지계약서_통합_및_안전_추가).hwp` (SM 2025년 최신 개정판)
   - *채택 사유*: 소프트웨어 산업 내 빈번하게 발생하는 위수탁 관계를 규율하는 핵심 법률문서이며, 비밀유지계약(NDA) 통합 및 안전 보건 관리 조항이 추가된 2025년 최신 법령 트렌드를 반영하기 위함입니다.

2. **과학기술정보통신부 제정 SW종사자 표준계약서 (총 2종)**
   - `201231_SW종사자_표준도급계약서.hwp` (프리랜서 위수탁용)
   - `201231_SW종사자_기간제,단시간__표준근로계약서.hwp` (단기 근로용)
   - *채택 사유*: 하도급법의 보호를 받기 어려운 1인 프리랜서 개발자 및 디자이너들의 지식재산권(저작권), 과업 범위, 대금 지급 지연 방지 등을 명확히 규정한 기준 문서입니다.

3. **독소조항 큐레이션 데이터 (`data/toxic/`)**
   - `경기도_프리랜서_가이드라인.pdf`, `프리랜서_불공정계약유형.md`
   - *채택 사유*: 단순히 표준계약서와 조항을 대조하는 것만으로는 적발하기 힘든 교묘한 불공정 특약(독소조항)의 패턴을 직접적으로 탐지하기 위해, 실제 분쟁 판례 및 지자체 가이드라인에서 발췌한 실무적 데이터를 수집했습니다.

### 2.2 원본 데이터 (Raw Data) 보존 정책
수집된 모든 원본 문서는 `data/01_raw/` 디렉토리에 불변(Immutable) 상태로 영구 보관됩니다. 이는 향후 파싱 알고리즘이 고도화되거나 전처리 요구사항이 변경되었을 때, 데이터 오염 없이 원본(HWP, DOCX, PDF)으로부터 전체 파이프라인을 재가동할 수 있는 베이스라인(Baseline) 역할을 수행합니다.

## 3. 데이터 전처리 파이프라인 (Data Preprocessing Pipeline)

수집된 원본 문서를 RAG(Retrieval-Augmented Generation) 시스템과 임베딩 모델이 고속으로 처리할 수 있는 정규화된 벡터 데이터로 변환하기 위해 3단계의 전처리 파이프라인을 거칩니다.

### 3.1 [Step 1] 문서 구조 파싱 및 텍스트 데이터 추출 (`kordoc` 채택)
대한민국의 정부 및 공공기관 표준 문서는 압도적으로 **HWP(아래아한글)** 포맷으로 유통됩니다. PyPDF나 Tika 등 범용 텍스트 추출기를 활용할 경우, 계약서 해석의 핵심인 조항 번호(제1조 등)와 항/호 기호(①, 1, 가 등)의 계층적 위계가 붕괴되며, 표 내부에 삽입된 텍스트의 순서가 소실되는 치명적 한계가 존재합니다.
- **채택 기술**: 국내 관공서 문서 파싱에 최적화된 **`kordoc`** CLI 도구를 채택하여, HWP 문서 특유의 계층적 구조와 표 형태를 보존한 채 마크다운(`.md`) 텍스트로 변환합니다.
- 추출된 마크다운 결과물은 `data/02_converted/`에 보관되며, 파싱 중 분리된 이미지 객체는 `images/` 디렉토리에 분리 저장됩니다.

### 3.2 [Step 2] 조항 단위 분할 및 스키마 기반 정규화 (Chunking & Normalization)
계약서 텍스트 전체를 단일 단위로 벡터화(Vectorization)할 경우, 특정 주제(예: 대금 지급)에 대한 검색 시 문맥적 연관성이 결여된 다른 조항(예: 손해배상)이 검색되는 노이즈(False Positive)가 급증합니다. 이를 방지하기 위해 정밀한 단위 분할(Chunking)이 필수적입니다.
- **조(Article) 단위 분리 (`StandardClause`)**: 마크다운 텍스트의 "제N조(제목)" 헤더를 정규식으로 인식하여 독립된 조항으로 분할합니다.
- **항(Paragraph) 단위 세분화 (`StandardSubChunk`)**: 본 시스템의 이탈 판별 알고리즘(`deviation.py`)은 실무에서 극단적으로 축약된 사용자 조항과 방대한 원본 조항 간의 길이 편차를 보정하기 위해 **항 단위 정렬(Sub-chunk Alignment)** 기법을 수행합니다. 이를 지원하기 위해 조항 내부를 ①, ②, 1. 등 세부 계층 기호 단위로 추가 분할합니다.
- 분할된 텍스트는 Pydantic 기반의 데이터 스키마 검증을 거쳐, `data/03_normalized/` 디렉토리에 각 도메인별 JSON 파일(예: `standard_clauses.si_subcontract.2025.json`)로 영속화됩니다.

### 3.3 [Step 3] 데이터 영속화 및 하이브리드 벡터 인덱싱 (Data Loading & Indexing)
실시간 런타임 환경에서 발생하는 파일 I/O 병목을 제거하고, 고속 검색 쿼리를 지원하기 위해 관계형 DB와 벡터 DB에 데이터를 적재합니다.
- **관계형 DB 적재 (`contract.sqlite3`)**: `just migrate` 스크립트를 통해 정규화된 JSON 데이터를 SQLite 데이터베이스로 마이그레이션합니다. 이를 통해 계약 연도, 도메인, 조항 번호 등 메타데이터 기반의 하드 필터링(Hard Filtering) 쿼리가 가능해집니다.
- **하이브리드 벡터 인덱싱 (`chroma.sqlite3`)**: `bge-m3` 다국어 임베딩 모델을 활용하여 조항 텍스트의 고차원 의미론적 벡터(Semantic Vector)를 추출합니다. 이후 단어의 의미적 유사성(Dense)과 키워드 매칭(Sparse)의 장점을 결합하기 위해 Chroma DB에 인덱스를 구축합니다. 이 과정은 `just build-index` 스크립트에 의해 완전히 자동화되어 있습니다.

## 4. 알고리즘 평가용 골든 데이터셋 (Golden Dataset) 구축

시스템의 이탈 탐지(Pure RAG) 성능을 LLM-judge의 개입 없이 정량적이고 결정론적(Deterministic)으로 검증하기 위해 자체적인 평가 데이터셋을 구축하였습니다.

### 4.1 데이터셋 구성 (v1 vs v2)
1. **`v1` 데이터셋 (기계적 변조)**: 표준조항 텍스트에서 명시적인 단어를 삽입하거나 삭제하는 등 형태적 변형이 주를 이루는 기초 테스트셋입니다.
2. **`v2` 데이터셋 (실무형 변조)**: 실제 계약 체결 과정에서 빈번히 발생하는 고난도 변형을 다룹니다. 방대한 분량의 원본 조항을 1~2문장으로 극단적으로 축약하거나, 문맥적 의미는 보존하되 어휘와 구문을 완전히 변형시킨 패러프레이징(Paraphrasing) 케이스들로 구성됩니다.

### 4.2 정답(Labeling) 분류 체계
골든셋의 각 문항은 법률적 관점에서 아래 5가지 클래스 중 하나로 라벨링(`gold_deviation`) 되어 있으며, 평가 파이프라인은 이 정답 라벨과의 일치율(Precision, Recall, F1-Score)을 산출합니다.
- **`NONE`**: 표준조항과 내용적 취지 및 법적 효력이 완벽히 동일한 무결점 조항
- **`MISSING`**: 수급사업자 보호를 위해 반드시 포함되어야 할 필수 표준조항의 누락
- **`EXTRA`**: 표준계약서 규정 외에 사용자(원사업자)가 임의로 추가 삽입한 조항
- **`CHANGED`**: 대응되는 표준조항이 존재하나, 당사자의 책임이나 권리가 불리/유리하게 변조된 조항
- **`TOXIC`**: 프리랜서 가이드라인 등에서 정의한 '대표적인 불공정 약관(독소조항)'의 의미론적 패턴을 띠는 조항

## 5. 향후 고도화 과제 (Future Scope)

현재의 데이터 파이프라인은 디지털 텍스트 변환이 보장된 1차 본계약 전자 문서(HWP, PDF 등) 처리에 최적화되어 있습니다. MVP 기획서 및 데이터 정책에 명시된 2차 고도화(Phase 2) 단계에서는 다음과 같은 데이터 파이프라인 확장을 목표로 합니다.

1. **조항 의존성 그래프(Clause Relation Graph) 구축 (MVP 고도화 A)**
   - **목표**: 평면적인 조항 검색을 넘어, "이 조항이 이탈할 경우 연쇄적으로 위험해지는 다른 조항"의 연관 관계를 추론하기 위한 그래프 데이터베이스를 구축합니다.
   - **계획**: 현재 초기 스키마만 정의된 `clause_relations.json` 데이터셋을 본격적으로 큐레이션하여 조항 간 엣지(Edge) 데이터를 확충하고 인접 리스트(Adjacency List) 형태의 지식 그래프(Knowledge Graph) 데이터를 확보합니다.

2. **OCR 전처리 파이프라인 도입 (MVP 비목표 4번 해제)**
   - **목표**: 1차 스코프에서 제외된 "물리적 날인이 포함되어 스캔된 형태의 이미지 PDF 계약서"로부터 텍스트를 추출하고 레이아웃을 복원합니다.
   - **계획**: VQA/OCR 기술을 연동하여 이미지 내의 표와 항/호 위계를 보존하여 마크다운으로 변환하는 전처리 프로세스를 추가합니다.

3. **부속 합의서 스키마 편입 (`hold/` 데이터 활성화)**
   - **목표**: 현재 1차 스코프에서 보류되어 `hold/` 디렉토리에 임시 보관 중인 부속 문서들의 탐지 커버리지를 확대합니다.
   - **계획**: 비밀유지 계약서(NDA), 직접지급 합의서, 연동계약서 등의 형태를 정규화 스키마에 통합하여 데이터 생태계를 본계약 외곽으로 확장합니다.
