# RAG에서 검색 관련성과 의미 동치성을 분리하는 법

> 프로젝트 저장소: [WorkShield — SKN30 3차 프로젝트](https://github.com/SKNETWORKS-FAMILY-AICAMP/SKN30-3rd-2Team)  
> 이 글은 위 프로젝트에서 표준계약서 대비 검토 후보를 검색하는 RAG MCP를 만들며 실제로 겪은 실패와 설계 변경을 정리한 글이다. 법률 내용을 해설하거나 조항의 위법성·유불리를 판단하려는 글이 아니라, 검색 시스템이 책임질 수 있는 질문의 범위를 다룬다.

다음 두 조항을 비교해 보자.

이 글에서 RAG(Retrieval-Augmented Generation, 검색 증강 생성)는 먼저 비교할 표준조항 후보를 찾는 검색 시스템을 뜻한다. `reranker`는 검색 후보를 다시 정렬하는 모델이고, `grounding`(근거 연결)은 검색한 근거를 결과에 연결하는 작업이다.

본문에서 `query`는 질의, `corpus`는 검색 대상 문서 모음(코퍼스), `sub-chunk`는 긴 조항을 나눈 작은 단위(하위 청크), `threshold`는 임계값으로 표현한다. `recall`은 정답을 놓치지 않은 비율인 재현율, `precision`은 검색 결과 중 정답 비율인 정밀도다. TP(true positive, 참 양성)는 실제 양성을 찾은 경우, FP(false positive, 거짓 양성)는 아닌 것을 양성으로 찾은 경우, FN(false negative, 거짓 음성)는 양성을 놓친 경우다.

```text
표준조항
성과물과 지식재산권은 수급인과 도급인의 공동소유로 한다.

사용자 계약
성과물과 지식재산권 일체는 별도의 대가 없이 도급인에게
전부 영구적으로 귀속된다.
```

두 문장에는 `성과물`, `지식재산권`, `귀속`, `도급인`이라는 핵심 어휘가 반복된다. 같은 주제를 다루고 있으므로 검색 시스템이 두 조항을 가까운 후보로 가져오는 것은 자연스럽다.

하지만 비교할 조항을 잘 찾았다는 사실과 두 조항의 내용이 같다는 결론은 전혀 다른 문제다.

WorkShield의 초기 구현은 이 둘을 충분히 구분하지 못했다. 검색과 재정렬에서 얻은 관련성 점수에 후보 선택뿐 아니라 본문 보존 여부까지 책임지게 했다. 그 결과 검색에는 성공했지만 의미 차이는 놓치거나, 반대로 정상적인 말바꿈을 변경으로 분류하는 일이 반복됐다.

이 글에서 다루는 질문은 다음과 같다.

> RAG는 두 문장의 의미가 같다고 판단해야 하는가, 아니면 그 판단에 필요한 비교 근거를 찾아야 하는가?

이 글에서 사용하는 `NONE`, `EXTRA`, `NO_MATCH`, `MISSING`은 모두 **법률적 판단이 아니라 검색 상태**다. 이후 상태값은 이 의미로만 사용한다.

## 처음에는 검색과 의미 판정을 하나의 흐름으로 보았다

초기 목표는 사용자 계약의 각 조항을 표준조항과 비교해 `NONE`, `CHANGED`, `EXTRA`, `NO_MATCH`, `MISSING` 같은 상태를 반환하는 것이었다.

당시 `NONE`은 대응하는 표준조항을 찾았고 내용도 충분히 보존됐다는 의미에 가까웠다. `CHANGED`는 같은 표준조항과 대응하지만 본문에 중요한 차이가 있다는 의미였다.

이를 구현하려고 다음 신호를 차례로 쌓았다.

```text
후보 검색과 reranker 점수
        ↓
서브청크 coverage 확인
        ↓
문자열 유사도 비교
        ↓
부정어·숫자·당사자 변경 규칙
        ↓
NONE 또는 CHANGED
```

겉으로 보면 각 단계가 앞 단계의 약점을 보완하는 구조다. 실제로는 서로 다른 질문을 하나의 결론에 몰아넣고 있었다.

- 검색은 “무엇과 비교할 것인가?”를 묻는다.
- coverage는 “표준의 구성요소가 얼마나 대응하는가?”를 묻는다.
- 문자열 비교는 “표현이 얼마나 비슷한가?”를 묻는다.
- 의미 동치성은 “의무·권리·예외·주체가 보존됐는가?”를 묻는다.

마지막 질문에 필요한 정보가 앞의 신호에 반드시 들어 있는 것은 아니다.

## 실패 1: 관련성이 높아도 의미는 반대일 수 있었다

글의 첫 사례는 실제 v3 골든셋의 `v3-sw-11`에서 가져왔다. 표준조항은 지식재산권 공동소유를 기술하고, 사용자 조항은 한쪽에 무상·영구 귀속되도록 바꿨다.

검색 관점에서는 성공 사례다. 지식재산권 조항을 입력했을 때 엉뚱한 대금이나 계약기간 조항이 아니라 지식재산권 표준조항을 찾았기 때문이다.

의미 보존 관점에서는 별도의 비교가 필요하다. 공동소유와 일방 귀속은 같은 주제 안에서 권리 관계가 달라진 경우다. 부정어가 추가되지 않아도, 숫자가 바뀌지 않아도, 등장하는 당사자 집합이 같아도 관계는 반전될 수 있다.

비슷한 현상은 독소 패턴 검색에서도 나타났다.

| 정상 또는 중립 문구 | 높은 점수로 잘못 연결된 주의 패턴 | 당시 점수 |
| --- | --- | ---: |
| 추가작업에는 추가대금을 지급한다 | 무상 추가작업 | 0.986 |
| 비밀유지 기간은 업무 종료 후 1년이다 | 무기한 비밀유지 | 0.991 |
| 지식재산권은 공동소유한다 | 전부·무상 귀속 | 0.991 |
| 법령과 상호 합의로 계약을 해석한다 | 일방적 계약 해석 | 0.99998 |

모델이 완전히 엉뚱한 주제를 고른 것은 아니다. 오히려 주제를 정확히 찾았기 때문에 점수가 높았다. 문제는 시스템이 이 점수를 극성, 기간, 대가, 권한 주체까지 같다는 증거처럼 소비한 데 있었다.

> 관련성 점수의 역할을 넘어선 해석은 점수가 아니라 소비 코드에서 만들어진다.

## 실패 2: 문자열 유사도는 독립적으로 작성된 계약에 약했다

reranker만으로 의미 차이를 가르기 어렵다고 판단한 뒤 문자열 일치율을 보조 신호로 사용했다. 그러나 실제 계약서는 표준계약서를 그대로 복사하지 않는다. 같은 의무라도 어순, 표현, 조항 구조가 크게 달랐다.

실계약 측정에서 문자열 유사도의 천장은 약 `0.545`였다. 기준을 높이면 정상적인 말바꿈을 `CHANGED`로 분류했고, 기준을 낮추면 실제 변경도 `NONE`으로 통과했다.

문자열 비교는 복사 여부나 서식 차이를 확인하는 데는 유용하다. 그러나 독립적으로 작성된 계약 조항의 의미 보존 여부를 맡기기에는 질문과 신호가 맞지 않았다.

> 결국 문자열 유사도는 “얼마나 비슷하게 쓰였는가”를 보는 신호이지, “의미가 보존됐는가”를 판단하는 신호가 아니었다.

## 실패 3: 규칙을 추가할수록 책임 경계가 흐려졌다

다음 시도는 의미 차이의 표면적 징후를 규칙으로 찾는 것이었다.

- 부정 표현이 추가되거나 삭제됐는가
- 숫자와 기간이 달라졌는가
- 당사자가 바뀌었는가

v3 골든셋을 실제 함수에 넣자 곧바로 구체적인 실패가 나타났다.

첫째, `아니되` 패턴은 계약서에서 흔한 “아니 된다”를 잡지 못했다. 활용형의 유니코드 문자가 예상과 달랐다.

둘째, 숫자 뒤 목적격 조사 `을`을 계약 당사자 `乙`로 오인했다. “10000분의 1을 곱하여” 같은 문구가 당사자 변경으로 분류됐다.

셋째, “없다”, “없이” 같은 표현은 규칙 목록에 들어 있지 않았다.

이 사례들은 패치할 수 있었다. 실제로 앞의 두 건은 고쳤다. 하지만 세 번째 누락을 발견했을 때 개별 표현을 더 추가하는 작업을 멈췄다.

한국어의 모든 부정·양태 표현을 열거하는 문제와, 표면 마커가 없는 권리 관계 반전을 이해하는 문제는 다르다. 전자는 규칙의 완성도 문제지만 후자는 규칙 입력 자체에 필요한 관계 정보가 없다.

강한 말바꿈일수록 문장 정렬 점수가 낮아져 규칙 검사 자체가 실행되지 않는 역설도 있었다. 신호에 없는 정보를 규칙의 개수로 만들려고 하고 있었다.

> 규칙은 표면 마커를 보완할 수는 있었지만, 문장 안의 권리 관계 자체를 이해하게 만들 수는 없었다.

## 실패 4: 골든셋도 시스템이 답할 수 없는 질문을 주고 있었다

v2 골든셋에서 `NONE`으로 라벨된 44건 중 35건은 60~90자 정도의 짧은 사용자 문장을 여러 항으로 구성된 긴 표준조항에 연결하고 있었다. 일부 표준조항은 28개 서브청크, 2,898자였다.

당시 coverage 규칙은 표준조항의 모든 항이 사용자 조항에 대응돼야 `NONE`을 반환했다. 짧은 문장 하나가 28개 항을 모두 포함할 수는 없다. 이 35건은 어떤 임계값을 선택해도 정답에 도달할 수 없었다.

실제로 임계값을 바꿀 때마다 결과는 “전부 CHANGED”와 “전부 NONE” 사이를 오갔다. 튜닝이 부족한 것이 아니라 시스템이 만족할 수 없는 데이터 계약을 주고 있었다.

v3부터는 조 전체 텍스트를 사용하고, 현재 코드가 맞히기 쉽게 문장을 순화하지 않았다. 코드가 틀리면 문장을 고치는 대신 실패 이유를 기록했다.

> 평가 데이터가 시스템이 답할 수 없는 질문을 주고 있다면, 임계값 튜닝은 품질 개선이 아니라 모순에 숫자를 맞추는 일이 된다.

이 과정을 거치며 문제를 다음처럼 다시 나눴다.

| 질문 | 필요한 신호 | 1차 RAG의 책임 |
| --- | --- | --- |
| 검색 후보가 존재하는가? | 검색 결과 유무 | 담당 |
| 가장 가까운 표준조항은 무엇인가? | 후보 순위와 reranker(후보 재정렬기) 점수 | 담당 |
| 대응 후보로 취급할 최소 신호를 넘었는가? | 정규화된 점수와 임계값 | 제한적으로 담당 |
| 계약 전체에서 표준조항이 보이지 않는가? | 전체 조항의 매칭 집합 | 담당 |
| 두 조항의 의미가 보존됐는가? | 권리·의무·주체·예외 관계 해석 | 확정하지 않음 |
| 차이가 사용자에게 어떤 의미인가? | 원문 비교와 도메인 해석 | 서버 밖 후속 검토 |

## 변경 후: RAG는 비교 근거를 찾는다

판정 책임을 줄인 뒤 1차 흐름은 다음과 같이 단순해졌다.

```text
사용자 조항
  ↓
표준조항·서브청크 후보 검색
  ↓
reranker로 후보 재정렬
  ↓
최고 후보 보존
  ↓
후보 유무와 임계값으로 상태 생성
  ↓
사용자 원문·표준 원문·점수·출처 반환
```

규칙 기반 치명변경 탐지, 문자열 유사도 폴백, 의미 coverage 2-pass와 `CHANGED` 상태는 제거했다.

현재 후보 선택 함수는 임계값을 적용하지 않는다.

```python
def select_best_match(candidates):
    if not candidates:
        return None, 0.0

    return max(candidates, key=lambda candidate: candidate[1])
```

이 함수가 답하는 질문은 하나뿐이다.

> 현재 후보 중 무엇이 가장 가까운가?

최고점 후보를 대응 조항으로 인정할지는 별도 함수가 결정한다.

```python
def classify_clause_deviation(
    matched_standard,
    score: float,
    match_threshold: float,
):
    if matched_standard is None:
        return Deviation.EXTRA

    return Deviation.NONE if score >= match_threshold else Deviation.EXTRA
```

임계값 미달인 `EXTRA`에도 최고 후보를 남긴다. 외부 클라이언트가 “무엇과 비교해 다른지” 설명하기 위해 같은 검색을 다시 실행할 필요가 없기 때문이다.

## 네 상태는 서로 다른 방향의 질문이다

현행 상태는 표준 대비 검토 후보를 구성하기 위한 검색 상태다.

| 상태 | 클라이언트가 해석할 의미 | 비교 방향 |
| --- | --- | --- |
| `NONE` | 대응하는 표준조항 후보를 찾음 | 사용자 조항 → 표준조항 |
| `EXTRA` | 근접 후보는 있지만 대응 신호가 임계값 미만 | 사용자 조항 → 표준조항 |
| `NO_MATCH` | 검색 후보 자체를 찾지 못함 | 사용자 조항 → 표준조항 |
| `MISSING` | 계약 전체에서 대응된 적 없는 표준조항 | 표준조항 → 사용자 계약 전체 |

`NONE`이라는 이름은 “차이 없음”으로 읽히기 쉽다. 그러나 현재 계약에서의 의미는 잠정적인 매칭 성공이다. 이름과 실제 의미가 어긋난다는 점은 이 API 계약의 명백한 부채다.

`NO_MATCH`와 `MISSING`도 다르다. `NO_MATCH`는 계약서에 실제로 존재하는 사용자 조항에서 비교 후보를 찾지 못한 상태다. `MISSING`은 방향을 뒤집어, 표준에는 있지만 사용자 계약 전체에서 대응된 적이 없는 조항을 찾은 결과다.

`EXTRA`의 근접 후보는 `MISSING` 계산에서 “대응됨”으로 세지 않는다. 임계값에 미달한 후보까지 표준조항을 커버한 것으로 처리하면, 사용자 조항은 `EXTRA`인데 해당 표준조항의 누락 가능성은 사라지는 모순이 생긴다.

## 결과 포맷은 판정보다 근거를 전달한다

실제 MCP 결과의 핵심 형태는 다음과 같다.

```json
{
```

여기서 `confidence=0.99`는 의미 동치 확률이 아니라 구현상 매칭에 사용한 관련성 신호다.

`NONE`과 `IP_TOTAL_FREE`가 동시에 존재하는 것도 모순이 아니다. 두 값은 서로 다른 검색 축에서 왔다.

- `NONE`: 대응하는 지식재산권 표준조항 후보를 찾았다.
- `IP_TOTAL_FREE`: 별도의 주의 문구 코퍼스와 유사한 표현도 찾았다.

1차 결과는 두 신호와 원문 쌍을 반환한다. 서버 밖 클라이언트는 필요할 때 이 근거를 읽고 권리·의무·주체·예외의 차이를 설명한다.

## 평가는 모델 점수가 아니라 질문별로 나눴다

책임을 나눈 뒤에는 평가도 질문별로 나눴다.

| 평가 질문 | 단위와 지표 |
| --- | --- |
| 정답 표준조항을 후보에 넣었는가? | 조항별 재현율(`Recall@k`), 평균 역순위(`MRR`) |
| NONE과 EXTRA의 경계를 나눴는가? | 조항별 혼동행렬, FP/FN |
| 표준조항 누락 후보를 찾았는가? | 계약 단위 MISSING(표준조항 누락) 재현율(recall) |
| 두 원문의 차이를 올바르게 설명했는가? | 1차 검색 평가와 분리 |

상태 오류는 정상 대응 조항을 낮은 점수로 놓치는 `UNDER_MATCH`(정상 조항을 놓침)와, 표준에 없는 조항을 높은 점수로 받아들이는 `OVER_MATCH`(추가 조항을 과도하게 매칭)로 다시 나눴다.

실제 `v3-si-45`는 표준에 없는 개발실 공간·테스트 환경 제공 조항이었지만 최신 표준 후보와 `0.99980`의 높은 점수로 매칭됐다. 검색 모델 입장에서는 프로젝트 수행 환경이라는 가까운 주제를 찾았지만, EXTRA/NONE 경계에서는 과매칭이었다.

이 사례는 임계값을 조금 올리는 것으로 해결하기 어렵다. 정상적인 말바꿈 중에는 반대로 점수가 낮은 사례도 있어, 한쪽 오류를 줄이면 다른 쪽 오류가 늘어날 수 있기 때문이다. `MISSING`도 단일 문장이 아니라 계약 전체의 `NONE` 집합을 기준으로 따로 측정해야 한다.

1차 회귀에서는 LLM-judge를 사용하지 않고 검색·상태·누락 신호를 결정론적으로 평가했다. 의미 차이 설명은 이 점수에 섞지 않았다.

## 변경 전과 변경 후

| 구분 | 변경 전 | 변경 후 |
| --- | --- | --- |
| reranker 점수 | 관련성과 의미 보존을 함께 판정 | 후보 재정렬과 대응 신호 |
| 문자열·규칙·coverage | 본문 변경을 확정하는 보조 판정 | 판정 경로에서 제거 |
| `CHANGED` | 1차가 본문 차이를 확정 | 제거 |
| `NONE` | 내용이 보존됐다는 확정 | 대응 후보를 찾았다는 잠정 상태 |
| `EXTRA` | 대응 후보가 사실상 소실 | 최고 근접 후보를 함께 반환 |
| 외부 클라이언트 | 결과 표시 위주 | 원문 쌍을 읽고 차이를 설명 |

바뀐 것은 모델이 아니라 점수와 상태에 부여한 의미였다.

## 이 설계에도 남아 있는 한계

가장 먼저 바꾸고 싶은 이름은 `NONE`이다. 현재 의미는 “차이 없음”이 아니라 “표준 후보와 매칭됨”에 가깝다. 현행 동결 계약에서는 기존 enum을 유지했지만, 새로 설계한다면 `MATCHED`나 `CANDIDATE_FOUND`처럼 이름 붙였을 것이다.

이름만 바꾼다고 검색 오류가 사라지는 것은 아니다. `NONE/EXTRA` 경계는 여전히 관련성 임계값에 의존한다. 의미적으로 가까운 비표준 조항은 과매칭되고, 강한 말바꿈은 under-match될 수 있다. 잘못된 `NONE`은 계약 전체의 `MISSING` 계산에도 영향을 준다.

이 한계를 출력 계약과 문서에 드러내는 것이 중요했다. 빈 결과나 낮은 점수를 “문제 없음”으로 바꾸지 않고, 각 상태가 답하는 질문을 그대로 전달하도록 했다.

## 내가 배운 설계 원칙

- 점수의 값보다 먼저 그 점수가 답하는 질문을 정의한다.
- 후보 검색과 의미 판정을 같은 상태값에 넣지 않는다.
- 실패 상태에도 다음 계층이 사용할 비교 근거를 남긴다.
- 검색, 상태 분류, 누락 탐지, 의미 설명을 따로 평가한다.
- 신호에 없는 정보를 규칙의 개수로 만들어내려 하지 않는다.

RAG의 역할은 모든 답을 확정하는 것이 아니라, 다음 판단에 필요한 근거를 신뢰할 수 있는 형태로 제공하는 데 있다.
