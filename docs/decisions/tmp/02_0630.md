# 결정 카드 — 2026-06-30

## 제품 경계·판정 설계

### D18. Z_Lever를 철회하고 서브청크 방식으로 전환
- **결정:** element 정렬·법률 레버·심각도 결정표를 포함한 Z_Lever를 전면 폐기하고 G_sub_chunk를 채택한다.
- **이유:** “어디가 어떻게 불리한지” 해석은 1차의 역할을 넘어 2차 LLM에 속한다. 계약 변경·구조 복잡도도 효용 대비 과했다.
- **대체:** 서브청크 검색, 부모 Max roll-up, 커버리지 체크.
- **상태:** 채택 (06-30)

### D19. 삭제 탐지는 커버리지 체크로 분리
- **결정:** 임계값 상향으로 항 삭제를 잡으려는 가정을 버리고, 부모 매칭 뒤 표준 서브청크마다 사용자 조항 안의 대응 여부를 확인한다. 미커버 시 `CHANGED`로 상향한다.
- **이유:** 일부 항이 삭제돼도 전체 조항 유사도는 매우 높게 남는다. 커버리지는 삭제 탐지에 직접 대응하는 결정론적 방법이다.
- **범위:** 조 내부만 탐색한다. 계약 전체 교차 탐색·재배치 해석은 2차에 남긴다.
- **상태:** 채택 (06-30)

### D20. 부모 검색에는 Max roll-up
- **결정:** 거대 조항의 부모 후보 확보 점수는 서브청크 점수의 Max를 사용하고 Mean은 기각한다.
- **이유:** 사용자가 한 항만 언급해도 정확한 부모를 놓치지 않아야 한다. 삭제 판단은 별도 커버리지 단계가 맡는다.
- **상태:** 채택 (06-30)

### D21. 서브청크의 초기 분할 규칙과 가법 저장소
- **결정:** 500자 초과 또는 항·호 기호 3개 이상 조항을 분할한다. `standard_sub_chunks` SQLite 테이블과 `StandardSubChunk` 모델만 신설한다.
- **이유:** 기존 계약·MCP 시그니처를 바꾸지 않고 구조 조회는 SQLite, 벡터 검색은 Chroma의 역할로 분리한다.
- **상태:** 채택; 초기값은 측정 후 조정 가능 (06-30)

### D22. 고도화 G의 평가는 Coverage Recall
- **결정:** 항 삭제 탐지의 주 지표를 Coverage Recall로 둔다.
- **이유:** Recall@k는 부모 검색 지표이고, 서브청크 삭제 탐지의 기여를 측정하지 못한다.
- **상태:** 채택 (06-30)

## 배포·모델 운영

### D23. 시연 배포는 Cloud Run 단일 컨테이너
- **결정:** MCP·보조 CLI·embedded SQLite/Chroma를 단일 Docker 컨테이너로 묶어 Cloud Run에 배포한다. 모델 추론은 외부 API를 호출한다.
- **이유:** 시연 단계에서 VM과 모델 서버 분리의 운영 복잡도·비용이 과하다.
- **기각:** VM 분리, 자체 호스팅 infinity-emb.
- **상태:** 채택 (06-30)

### D24. 시연기의 임베딩/리랭커 전략
- **결정:** 임베딩은 HF Serverless API, 리랭커는 no-op으로 처리한다. bge-m3와 경량 한국어 모델은 Hit@1/Hit@3로 비교해 이후 확정한다.
- **이유:** 해당 리랭커 전용 API 부재와 시연 규모의 낮은 리랭킹 효용 때문이다. bge-m3는 현재 dense만 활용해 모델 고유의 sparse/ColBERT 강점은 쓰지 못한다.
- **전환 조건:** 실서비스 또는 정확도 검증 시 별도 모델 서버/모델 교체를 검토한다.
- **상태:** 채택 / 모델 선택 후속 (06-30)

### D25. 다중 벡터 컬렉션은 현 구조를 유지
- **결정:** `collection_name` 기반 Chroma 다중 컬렉션 지원을 그대로 활용한다.
- **이유:** 표준조항과 독소패턴은 추가 구조 변경 없이 격리 가능하다. ColBERT는 Chroma 구조상 불가하므로 시연 범위 밖으로 둔다.
- **상태:** 채택 (06-30)

## 헥사고날 경계·의존성 주입

### D26. 인프라 포트를 adapter로 이동
- **결정:** `Embedder`·`Retriever`와 `Reranker`·`Database`는 `adapter/port.py`에 두고, contracts에는 도메인 포트만 남긴다.
- **이유:** 모델·벡터 DB에 의존하는 계약을 도메인 계약과 분리한다. 고유 어댑터 기능에 별도 포트를 늘리지는 않는다.
- **상태:** 채택 (06-30)

### D27. VectorManager 생성자 주입과 단일 조립 지점
- **결정:** VectorManager가 embedder를 생성자로 주입받고, `adapter/__init__.py`에서 구현체를 조립한다.
- **이유:** 테스트에서 실제 모델 로드를 막고 mock 주입을 가능하게 하며, 외부 소비자의 import 방식은 유지한다.
- **상태:** 채택 (06-30)

### D28. 운영 환경의 RunPod 구현체 선택
- **결정:** local 이외 환경에서는 RunPod `/runsync` 기반 API Embedder/Reranker를 사용하며, `APP_ENV` 분기는 adapter 조립 지점에 한정한다.
- **이유:** pipe/server/core가 환경·외부 I/O를 알지 않게 한다. 콜드스타트를 고려해 호출 타임아웃은 60초로 둔다.
- **상태:** 채택 (06-30)

## 데이터 분류·코퍼스 완전성

### D29. Category 점수는 앵커별 max-pool과 제목 우선
- **결정:** 앵커 평균벡터 대신 앵커별 벡터의 최대 유사도를 쓰고, 제목 단독 및 제목+본문 쿼리 중 높은 점수를 쓴다.
- **이유:** 평균벡터는 다의적 카테고리의 신호를 희석하고, 법률 본문 일반어휘는 제목의 사건 신호를 가린다.
- **상태:** 채택 (06-30)

### D30. 카테고리 판정은 낮은 임계값·마진 보조
- **결정:** 실측을 바탕으로 점수 임계값 0.40, 마진 0.02를 사용한다.
- **이유:** 실제 조항의 점수 분포는 합성 테스트보다 낮고, 미분류의 더 강한 신호는 절대점수보다 1·2위 점수의 근접성이다.
- **상태:** 채택 (06-30)

### D31. 표준 조항은 절대 드롭하지 않고 GENERAL로 수용
- **결정:** 정규화에서 SKIP을 제거하고, 특정 범주가 아닌 표준 조항은 `Category.GENERAL`로 명시적으로 수용한다.
- **이유:** 표준 코퍼스에서 누락된 조항은 사용자 계약의 MISSING을 영구히 탐지할 수 없게 하는 조용한 실패다.
- **영향:** GENERAL은 임베딩 경쟁에 참여하지 않는 fallback이며, grounding·그래프에서 단일 술어로 제외한다.
- **상태:** 채택; enum 변경은 승인됨 (06-30)

### D32. 다중 카테고리 대신 단일 라벨과 구체성 우선순위
- **결정:** 마진 내 복수 후보에는 결정론적 구체성 우선순위를 적용해 단일 라벨만 부여한다.
- **이유:** MVP의 모델·매칭·평가 복잡도를 억제하면서 모호함을 재현 가능하게 해소한다.
- **기각:** 다중 라벨.
- **상태:** 채택 (06-30)

## 성능·신뢰성

### D33. 하이브리드 검색과 BM25 내부 최적화
- **결정:** dense/BM25 검색을 병렬화하고, BM25는 상위 k만 heap으로 선별한다. 인덱스 빌드는 락 범위를 줄이고, 빈 결과를 캐시하며 Kiwi 배치 토큰화를 쓴다.
- **이유:** 독립 검색의 지연을 겹치고, 불필요한 전체 정렬·빈 컬렉션 재조회·전역 락 대기를 줄인다.
- **상태:** 채택 (06-30)

### D34. 적재 병목은 코드가 아닌 CPU 임베딩
- **결정:** 적재 경로의 배치 오버랩과 캐시 무효화 통합은 적용하되, 40초/반복의 근본 원인은 CPU에서의 대형 BGE-m3 추론으로 판단한다.
- **이유:** GPU 부재와 모델/미니배치 측정 결과가 대부분의 시간을 `encode`가 차지함을 보였다.
- **후속:** GPU·경량 모델·API 임베더 중 해소 방안을 모델 비교 결과와 연계해 결정한다.
- **상태:** 원인 확정 / 해소 방향 보류 (06-30)
