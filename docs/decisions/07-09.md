## 0. 세션 결론 (한 줄)

`ToxicPattern`enum엔 `contract_types` 같은 계약유형 제약이 불필요함을 확인(순수 어휘 문제이지 개념
유효성 문제가 아님) → N 카드 방향 2(코퍼스 하도급 어휘 이중화, 40→90건 + `INDEFINITE_CONFIDENTIALITY`
신규 작성, `korean-law` CLI로 판례 근거 확보)를 구현·`just build-db`·`pytest` 검증까지 마쳤으나,
사용자가 실행한 `v3_result.md` 재평가에서 SI/SM 독소 recall이 전혀 개선되지 않음(TP=0 유지)을
발견했다. 직접 재현한 결과, 골든셋 `user_clause`가 `"### 제N조(제목)\n본문"` 형태로 마크다운 헤더를
포함하는데 `toxic_patterns.json` 코퍼스는 헤더 없는 평문이라 이 비대칭이 리랭커 점수를 0.99대에서
0.0003대로 붕괴시키는 것이 근본 원인임을 확인했다(SI·SW 양쪽에서 재현). 표준조항 코퍼스는 쿼리와
동일하게 헤더를 포함해 대칭이라 이 문제가 없었다 — 이탈(deviation) 탐지가 정상이었던 이유이기도 하다.
사용자 결정에 따라 이 원인은 N 카드 범위를 벗어난 별개 카드([P_text_normalization.md](../tasks/P_text_normalization.md))로
분리하고, 이번 세션에서는 구현하지 않았다.

## 1. `ToxicPattern.contract_types` 필요성 검토

`Category.contract_types`는 "이 카테고리 개념이 해당 계약유형에 아예 존재하지 않는다"(예:
`WORKING_HOURS`는 도급/용역엔 없음)는 **구조적 제약**을 표현하며, `korean_law_grounder.py`·
`server.py`(list_categories 필터)·`pipe/normalize.py`(스코핑 검증) 3곳에서 실제로 쓰인다.

`ToxicPattern`의 9개 독소 개념(경업금지·IP무상귀속·대금지연 등)은 SW_FREELANCE·SI_SUBCONTRACT·
SM_SUBCONTRACT 어디서든 개념적으로 동일하게 성립 — "이 유형엔 이 독소가 없다"는 구조적 제약이 아니라
**순수 어휘(당사자 용어) 문제**임을 확인했다. 코드 근거: `review_pipe.py`의 독소 검색은 이미
`type_filter=None`으로 계약유형 무관 검색을 전제하고 있고, `toxic_patterns.json`·`ToxicPatternRecord`·
`toxic_patterns` 테이블 어디에도 `contract_type` 컬럼이 없다(3곳 일관). 결론: **`contract_types`
필드 불필요** — 오히려 도입해 필터링에 쓰면 코퍼스에 특정 유형 어휘가 없을 때 후보가 0으로 떨어져
역효과.

## 2. N 카드 구현 — 코퍼스 하도급 어휘 이중화 (방향 2 확정)

사용자가 "독소 개념 수가 적으니 방향 2(코�퍼스 증식)가 맞다"고 확정. 구현 범위:

- 기존 8개념 40건 각각에 당사자 어휘(도급인→원사업자, 수급인→수급사업자, 보수→하도급대금 등)만
  치환한 하도급 어휘 버전 40건 추가(`pattern_id`는 `-sub-` 접미사로 구분, 원본 id·내용 강도 불변).
- 조사 중 발견: `ToxicPattern` enum엔 9개 값이 있는데 코퍼스엔 `INDEFINITE_CONFIDENTIALITY`(불특정
  기간 비밀유지) 예문이 0건 — 그런데 `v3_sm_subcontract.json`·`v3_sw_freelance.json` 골든셋은 이미
  이 패턴을 `gold_toxic`으로 요구해 recall이 무조건 0으로 확정되는 상태였음. 사용자가 이 개념도
  범위에 포함하기로 결정하고, `korean-law` CLI로 법적 근거를 조사하도록 지시.
  - 조사 결과: 서울고법 2011라1853(전직금지가처분) 판결이 민법 제103조 위반 여부를 판단할 때
    "제한기간의 과도성"을 핵심 요소로 삼는다는 점, 상법 제41조(영업양도인의 경업금지)가 유사 취지의
    제한에도 법정 상한(20년)을 두는 입법례라는 점을 근거로, "기간의 정함 없음/영구"를 독소 요소로
    명문화한 SW 어휘 5건 + 하도급 어휘 5건(총 10건) 작성.
- 최종 코퍼스: 40 → **90건**(8개념 이중화 80 + 신규 개념 10).
- 스키마(`ToxicPatternRecord`·SQL DDL·`build_toxic_index`)는 변경하지 않음(§1 결론에 따라 불필요).

## 4. 근본 원인 진단 — 쿼리·코퍼스 간 마크다운 헤더 포맷 비대칭

`v3_result.md`을 M_eval_dual_track.md(이탈 채점 지표 재정의) 소관 아니냐는 사용자 질문에 대해, M은
`deviation`/`CHANGED` 관련 지표 정의만 다루고(07-07 §7에서 `CHANGED` 자체가 이미 폐지돼 M의 전제도
낡은 상태) 독소 지표와는 무관함을 먼저 확인. 이어서 N 카드가 실제로 효과가 있는지 직접 재현:

- 로컬(실제로는 `.env`의 `APP_ENV=prod` 기본값 때문에 전 과정이 Runpod API 경유)에서 `v3-si-31`
  (`NONCOMPETE_EXCESS`, EXTRA) 조항 하나만 단독으로 검색·리랭킹했을 때 신규 `-sub-` 어휘 항목이
  최고점(0.998)으로 정확히 매칭 — 코퍼스 수정 자체는 유효함을 1차 확인.
- 그러나 골든셋 45건 전체를 `review_contract`로 배치 검토(eval과 동일한 호출 경로)하면 동일 케이스가
  빈 `toxic_patterns=[]`로 나옴 — 로컬 재현으로 격리.
- 단계별 분해 결과 `hybrid_search_many`(검색)는 정상(`-sub-01`이 최상위 후보)이나 `reranker.rerank_many`
  (재정렬) 점수가 0.998이 아니라 0.00036으로 나오는 지점 특정.
- 두 호출의 유일한 차이는 **쿼리 텍스트**: 단독 테스트는 손으로 옮겨 적은 본문만, 배치 테스트는 골든셋
  원본 그대로(`"### 제43조(경업금지 및 인력 유출 방지)\n"` 헤더 포함)였음. 동일 문서쌍에 헤더 유무만
  바꿔 직접 대조:

  | 계약유형 | 케이스 | 헤더 포함 쿼리 | 헤더 제거(본문만) |
  | --- | --- | --- | --- |
  | SI_SUBCONTRACT | v3-si-31 (`NONCOMPETE_EXCESS`) | 0.00036 | 0.9979 |
  | SW_FREELANCE | v3-sw-07 (`INDEFINITE_CONFIDENTIALITY`) | 0.009 | 0.9999 |

  SI뿐 아니라 SW에서도 동일 크기로 재현 — 세 계약유형 공통의 근본 원인.
- `standard_clauses.*.json`(표준조항 코퍼스)도 `"### 제1조(목적) ..."`처럼 동일한 헤더 포맷을 쓰고
  있어 쿼리와 대칭 — 이탈(deviation) 탐지 F1이 정상(SI 0.903·SM 0.909·SW 1.000)이었던 이유를 설명한다.
  `toxic_patterns.json`만 처음부터(N 카드 이전의 기존 40건부터) 헤더 없는 평문으로 작성돼 있어
  비대칭이 발생했다.

## 5. 범위 분리 결정 (사용자 결정)

사용자가 "독소 쿼리에서만 헤더를 제거하는 국소 패치"보다 넓게, DB에 저장되는 데이터(표준조항·서브청크·
독소패턴)의 마크다운 헤더·특수문자 정규화와 사용자 입력(런타임 파싱 후 텍스트)의 정규화를 **동일한
과정으로 공유**하도록 전반적으로 검토해야 한다고 판단 — 이번 세션에서 구현하지 않고
[P_text_normalization.md](../tasks/P_text_normalization.md)로 분리해 질문 목록만 정리했다.
[N_toxic_corpus_bias.md](../tasks/N_toxic_corpus_bias.md)의 "recall 재측정" DoD 항목은 P 카드가
선행돼야 실제로 검증 가능하다고 표시해뒀다.

## 6. P 카드가 세션 중 이미 병행 구현됨 — `v3_result.md` 재검토

§5에서 "이번 세션 미구현"으로 정리한 직후, 사용자가 `v3_result.md` 재분석을 요청. 확인해보니
`git status`상 `core/splitter.py`(`normalize_for_search` 신설)·`pipe/build_index.py`·
`pipe/review_pipe.py`·`adapter/vector_manager.py`·`adapter/port.py`가 이미 (커밋 안 된 상태로)
수정돼 있었다 — P 카드 해법이 세션과 병행해 별도로 구현된 상태. 헤더·항호 기호를 벗기고 제목만
본문 앞에 병합하는 정규화 함수가 `build_index.py`(코퍼스 색인)와 `review_pipe.py`(쿼리) 양쪽에
대칭으로 배선돼 있었다. `docs/tasks/M_eval_dual_track.md`에도 별도로 §5(RRF `dense_weight`/
`bm25_weight` 파라미터화, `review_contract`에는 미배선)가 추가돼 있었으나 기본값이 하위호환이라
이번 결과 변화의 원인은 아님을 확인.

이 상태로 재실행된 `v3_result.md`(12:26:42)를 원인 분석한 결과:
- **독소**: recall은 개선(TP 0→2/1/5)됐으나 **precision이 붕괴**(FP 1~2건→7~14건) — 정규화로
  점수 분포 자체가 바뀌었는데 `toxic_threshold=0.6`을 그대로 둬서 과잉 트리거가 발생한 것으로 추정.
- **이탈**: SI/SM/SW 전부 F1 악화(예: SI 0.903→0.667). `review_contract` 경로는 이미 대칭 정규화라
  이것만으론 설명 안 돼 추가 조사.
- **A-1(검색 ablation)**: `hybrid_rerank` recall@5 0.920→0.893으로 하락.

## 7. A-1 회귀 원인 특정 — `run_eval.py`에 정규화 미적용 (사용자 가설 확인)

사용자가 "`run_eval.py`에 이번 정규화 로직이 안 들어가서 그런 것 아니냐"고 제기 → 코드 확인 결과
**A-1에 한해 정확했다**: `build_cases_by_variant()`(run_eval.py:178)는 `review_contract`를 거치지
않고 `queries = [g["user_clause"] for g in scored]`(원문 그대로)로 `bm25_search_many`/
`dense_search_many`/`hybrid_search_many`를 직접 호출한다 — 코퍼스는 이미 정규화돼 있는데 쿼리만
비정규화라 비대칭 재발. 로컬 측정으로 확인(SI, hybrid_rerank recall@5): 원문 쿼리 0.867 → 정규화
쿼리 0.900.

그러나 이탈(A-2/A-3)은 `review_golden_clauses()` → `review_contract()` 경로라 이미 대칭 정규화
상태였고, 이 가설로는 설명되지 않았다. 로컬 재현(SI, 45건 배치)으로 오분류 6건을 특정했는데
(`v3-si-41·44·45` 등, EXTRA인데 NONE으로 오판), 매칭된 표준조항 id가 `si_subcontract-2022-art36`·
`si_subcontract-2022-art52`처럼 **2022년판**이었다. DB 확인 결과 `standard_clauses`에
SI_SUBCONTRACT 2022년판(53건)·2025년판(56건)이 `version` 구분 없이 같은 검색 풀에 공존 —
`_load_standards`·`review_contract`의 `type_filter`가 `contract_type`만 걸러 `version`은 전혀
스코핑하지 않는, 이번 세션과 무관한 기존 구조적 결함으로 판단. (부수 관찰: 로컬 재현 6건 vs 실제
eval 9건 — Runpod 추론에 실행 간 미세한 비결정성이 있는 것으로 보임, 향후 결과 비교 시 유의.)

## 8. 조치 — A-1 즉시 수정, 버전 스코핑은 새 카드로 분리 (사용자 결정)

더 근본적인 문제라 별도 카드로 분리 — 이번 이슈 + version의 쓰임새 검토 둘 다 카드에 포함."

- `src/eval/run_eval.py::build_cases_by_variant`에 `core.splitter.normalize_for_search` 적용해
  쿼리를 정규화(§7 실측대로 코퍼스와 대칭 회복). `uv run pytest tests/ -m "not integration"` 102
  passed(정규화 관련 신규 테스트 포함 총량 증가), 회귀 없음 확인.
- [Q_standard_version_scoping.md](../tasks/Q_standard_version_scoping.md) 신설 — §7의 재현 데이터·
  `version` 필드가 어디서도 소비되지 않는다는 사실을 그대로 옮기고, "버전이 이 시스템에서 뭘
  의미해야 하는가"(최신판 전용 vs 특정 시점 표준 비교 지원)부터 미결 질문으로 남겼다. 코드는
  변경하지 않음.

## 후속(미결) 항목

- ⬜ **독소 임계값 재보정** — §6에서 발견한 precision 붕괴(FP 급증). `toxic_threshold=0.6`이
  정규화 이후 점수 분포에 안 맞을 가능성 — held-out으로 재보정 필요.
- ⬜ **P 카드 검증** — 병행 구현된 정규화가 이탈 F1까지 악화시킨 이유가 §7의 버전 혼재만으로 완전히
  설명되는지, 정규화 자체의 다른 부작용은 없는지 Q 카드 해결 후 재확인.
- ⬜ **Q 카드 착수** — `version` 의미 확정 + 스코핑 구현.
- ⬜ **N 카드 recall 재측정** — P·Q 카드 정리 후 `just eval`로 SI/SM(+SW `INDEFINITE_CONFIDENTIALITY`)
  개선 여부 확인.
- ⬜ M/O 카드는 07-08 후속 항목에서 이미 미착수 상태 유지.
