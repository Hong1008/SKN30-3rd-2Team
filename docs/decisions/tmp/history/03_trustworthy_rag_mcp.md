# 잘 찾는 도구에 신뢰 기준을 세우다

> WorkShield 프로젝트 발전사 3/3  
> 결과의 원인을 재현하고, 실패한 개선을 기준선으로 되돌릴 수 있게 만든 기록이다.

독소패턴 예문을 40건에서 90건으로 늘렸는데도, SI·SM 계약에서 참 양성(TP)은 여전히 0건이었다.

분명히 코퍼스는 늘었고 인덱스도 다시 만들었다. 단일 문장을 손으로 넣으면 새 예문이 잘 검색됐다. 그런데 골든셋 전체를 평가 경로로 흘리면 같은 사례가 탐지되지 않았다.

두 번째 글까지의 WorkShield는 검색 점수에 과도한 의미를 부여하지 않도록 역할을 정리했다. 1차 RAG MCP는 표준조항과 독소패턴 후보를 검색하고 명시 상태를 반환한다. 의미 차이는 필요할 때 서버 밖 클라이언트가 원문을 읽고 설명한다.

하지만 경계를 정했다고 결과가 자동으로 신뢰할 수 있게 되는 것은 아니었다. 후반부의 과제는 최고 점수를 더 높이는 일이 아니라 다음 질문에 답하는 일이었다.

> 이 결과가 나온 이유를 재현할 수 있고, 개선안이 실패했을 때 원래 기준선으로 돌아갈 수 있는가?

## 데이터 양보다 표현 계약이 먼저였다

문제의 원인은 문장 내용이 아니라 형식이었다.

```text
골든셋 질의:
### 제43조(경업금지 및 인력 유출 방지)
본문...

독소패턴 코퍼스:
본문만 있는 평문...
```

같은 문장쌍에서 헤더 유무만 바꾸자 SI 사례의 재정렬 점수는 `0.00036 → 0.9979`로 바뀌었다. SW 사례도 `0.009 → 0.9999`로 변했다.

표준조항 검색이 상대적으로 정상인 이유도 설명됐다. 사용자 질의와 표준조항 코퍼스는 모두 `### 제N조(제목)` 형식을 사용했지만, 독소패턴만 평문이었다. 질의와 코퍼스의 표현이 비대칭이었던 것이다.

해법은 독소 검색에만 헤더 제거 코드를 붙이는 것이 아니었다. 검색용 정규화 규칙을 하나 만들고, 코퍼스를 색인하기 직전·런타임 사용자 질의를 검색하기 직전·평가가 검색기를 직접 호출하는 경로가 모두 공유하게 했다. 원본 문서는 보존하고, 검색 표현만 같은 규칙으로 변환했다.

> RAG에서 데이터가 많다는 사실보다, 질의와 코퍼스가 같은 표현 규칙을 따른다는 사실이 먼저다.

## version은 데이터 필드가 아니라 제품 정책이었다

정규화 이후 이탈 오류를 다시 살펴보다가 이상한 후보를 발견했다. SI 2025년 표준계약서를 기준으로 만든 v3 골든 케이스가 SI 2022년 표준조항과 매칭되고 있었다.

스키마에는 처음부터 `version`이 있었다. 하지만 검색은 계약유형만 필터링했고 version은 사용하지 않았다. 데이터에 필드가 있다는 사실과 제품이 그 필드의 의미를 소비한다는 사실은 달랐다.

정책을 명확히 했다.

- 계약유형마다 최신 표준판 하나만 활성 검색 대상으로 둔다.
- 과거판은 삭제하지 않고 보관 영역으로 옮긴다.
- 활성 영역에 같은 계약유형의 version이 둘 이상이면 데이터 재구성 과정을 실패시킨다.
- 특정 체결 시점의 과거 표준 비교는 현재 MCP 범위 밖으로 둔다.

이 문제는 검색 튜닝 문제가 아니었다. “어느 표준판과 비교하는가”라는 제품 정책이 데이터 적재 단계에 반영되지 않은 문제였다.

## MCP도 호출할 수 있다는 것만으로 운영 도구가 되지 않는다

RAG의 데이터 경계를 정리하는 동안 MCP의 실행 경계도 손봤다. 1차 파이프라인은 LLM 없이 표준 대비 후보를 만들고, 조문번호가 명시된 제한된 근거만 원문으로 붙였다. 더 넓은 외부 조회 도구는 필요한 클라이언트가 별도로 조합할 수 있게 했다.

지원 범위를 확인하는 도구도 추가했다. 파일명이나 LLM 추측이 아니라 파싱된 본문의 규칙으로 지원 범위 안·계약유형 불확실·범위 밖을 구분했다.

외부 도구의 출력도 제한해야 했다. 포괄적인 조회와 잘못된 본문 파싱 때문에, 수정 전에는 문서에 부착된 법령 근거 항목이 총 82,080개까지 늘었다. 조문 범위와 결과 상한을 적용한 뒤에는 101개가 됐고, 정상 문서 평균 처리시간도 66.1초에서 19.1초로 줄었다.

이 수치는 법률 검색 품질을 증명하는 지표가 아니다. 외부 도구도 세션, 캐시, 출력 상한, 지원 범위를 관리해야 MCP 안에서 운영 가능한 구성요소가 된다는 사례다.

익숙하지 않은 전문 도메인을 다루면서 우리는 전문지식을 더 많이 흉내 내기보다, 지원 범위와 원문 근거, 실패 상태를 명시하는 쪽으로 책임 경계를 좁혔다.

## 진단도 실제 판정의 증거여야 했다

오류를 분석하려면 최종 결과와 함께 상위 후보와 점수를 남겨야 했다. 처음에는 `review_contract`가 끝난 뒤 평가 코드가 검색과 재정렬을 다시 실행해 진단 자료를 만들었다.

그러나 이것은 실제 판정을 설명하는 증거가 아니었다. 최종 후보가 진단의 상위 3개 후보에 없는 사례가 7건 발견됐다. 진단은 실제 판정 뒤에 다시 실행한 검색에서 나왔고, 외부 재정렬 모델의 별도 호출 결과는 최종 실행과 달랐다.

평가 코드를 바꿔 실제 review 실행 중에 나온 후보와 점수를 수집하게 했다. 이후 최종 후보와 진단 후보가 불일치한 사례는 0건이 됐다. 후보가 없을 때도 빈 값 대신 `NO_CANDIDATE`나 `SEARCH_MISS`를 기록했다.

> 관측 데이터는 판정을 재현한 결과가 아니라, 판정을 만든 바로 그 실행에서 나와야 한다.

## 하나의 골든셋으로 모든 질문에 답하지 않았다

처음에는 검색, 이탈, 독소, 임계값을 하나의 골든셋에서 함께 보려 했다. 이후에는 데이터셋마다 답할 질문을 분리했다.

- **v4**는 정상 조항과 독소 예문이 표면상 비슷한 경우를 검증하는 반례 데이터셋이다. 독소 개선안을 독립적으로 확인하는 용도로 동결했다.
- **v5**는 SI·SM·SW 계약유형별 이탈 오류를 비교하는 기준선이다. 같은 주제 그룹이 튜닝용 데이터와 독립 검증용 데이터에 함께 들어가지 않게 분리했다.

임계값 기본값을 유지한 직접 근거는 v3의 고정 독립 검증이었다. 튜닝에서는 좋아 보인 후보가 독립 검증에서 같은 우위를 재현하지 못했다. 그래서 `match_threshold=0.50`, `toxic_threshold=0.60`을 바꾸지 않았다.

v5는 다른 사실을 보여 줬다. SI·SM에서는 정상 조항을 놓치는 오류가 많았고, SW에서는 추가 조항을 과도하게 매칭하는 오류가 많았다. 하나의 전역 임계값으로 모든 계약유형을 동시에 개선하기 어렵다는 뜻이었다.

## 좋아진 수치만으로는 제품에 넣지 않았다

평가 기반 개선에서 가장 위험한 순간은 독립 검증 결과를 본 다음이다. 실패 사례를 확인한 뒤 조건이나 임계값을 조금 고치면 같은 데이터에 빠르게 맞출 수 있다. 하지만 그 데이터는 더 이상 독립 검증셋이 아니다.

그래서 실험 전에 가설, 튜닝 통과 기준, 독립 검증 채택 기준을 기록했다. 평가 도구도 특정 사례만 실행하고, 동결된 산출물을 덮어쓰지 않으며, 결과를 별도 경로에 남기도록 바꿨다.

두 실험은 이 원칙을 실제로 시험했다.

| 실험 | 가설과 관찰된 개선 | 기준을 통과하지 못한 이유 | 결정 |
| --- | --- | --- | --- |
| S | 독소 후보에 제목을 붙이면 구분력이 좋아질 것이라 가정. FP는 줄었다. | FN이 9→12로 늘고 F1이 0.563→0.462로 하락 | 독립 검증 미실행, 폐기 |
| C1 | 표준 후보와 항 단위 후보의 부모가 다르면 추가 검토 후보로 처리. FN은 4→2, F1은 0.706→0.800 | FP가 1→2로 늘어 사전 기준 초과 | 보류, 기준선 복귀 |

실험 S는 튜닝에서 실패했으므로 독립 검증 데이터를 쓰지 않았다. C1은 튜닝을 통과해 독립 검증을 한 번 실행했지만, 일부 수치가 좋아져도 전체 기준을 통과하지 못했으므로 기본 동작에 남기지 않았다.

이 결정은 좋은 아이디어를 버린 일이 아니라, 개선의 정의를 실행 전에 정한 대로 지킨 일이다.

## 바이브 코딩에서 인간이 맡았던 일

이 프로젝트의 코드는 AI의 도움을 받아 빠르게 만들어졌다. 내부 구현을 모든 순간에 한 줄씩 이해하고 작성한 것은 아니다. 오히려 그래서 사람이 놓치지 말아야 할 일이 더 분명해졌다.

- 이 시스템이 답해야 할 질문과 답하지 말아야 할 질문을 정한다.
- 모델 점수와 상태의 의미를 계약으로 고정한다.
- 원천 데이터와 파생 인덱스를 구분한다.
- 평가셋이 실제로 측정 가능한 정답인지 검토한다.
- 튜닝과 독립 검증을 격리한다.
- 일부 수치가 좋아져도 사전 기준에 실패하면 채택하지 않는다.
- 실패한 실험을 기본 동작에서 제거하고 기준선으로 돌아간다.

AI가 코드를 만드는 비용은 낮아졌지만, 구현 가능한 아이디어 가운데 무엇을 제품에 남길지 결정하는 비용은 낮아지지 않았다. 특히 전문성이 필요한 도메인에서는 문제 정의와 책임 경계가 기술 선택보다 앞에 있었다.

## 프로젝트가 도착한 곳

최종 WorkShield는 계약 내용을 법적으로 판단하는 시스템이 아니다. 사용자 계약을 표준계약서와 비교해 다음 신호와 근거를 반환하는 RAG MCP다.

- 잠정 매칭된 표준조항
- 추가 검토 후보
- 검색 후보 없음
- 계약 전체에서 대응되지 않은 표준조항
- 독립적으로 검색된 독소패턴 후보
- 외부 클라이언트가 이어서 사용할 원문과 근거

처음의 문제 정의는 크게 바뀌지 않았다. 바뀐 것은 그 문제를 해결한다고 믿었던 구현의 범위였다. 규칙과 점수를 더하는 것보다 어떤 정보까지 책임질 수 있는지 좁히고, 실패를 숨기지 않고, 독립 데이터로 변경을 검증하는 일이 더 중요했다.

이 발전사에서 가장 가치 있었던 산출물은 특정 모델이나 프레임워크가 아니었다.

> RAG가 근거를 찾고, MCP가 그 근거와 실패 상태를 전달하며, 인간이 시스템의 질문·검증 기준·중단 조건을 책임지는 구조였다.
