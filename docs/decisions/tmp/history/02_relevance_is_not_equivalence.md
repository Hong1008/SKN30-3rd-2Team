# 검색 점수는 의미가 같다는 증거가 아니었다

> WorkShield 프로젝트 발전사 2/3  
> 실제 모델과 평가 데이터를 연결하면서, 검색 점수에 부여했던 의미를 교정한 기록이다.

첫 번째 단계에서 WorkShield는 계약서 조항을 표준조항과 매칭하는 RAG의 골격을 갖췄다. 모듈별 테스트도 통과했다. 하지만 실제 데이터를 전체 파이프에 흘려보내자 결과는 처참했다.

모든 사용자 조항이 `NO_MATCH`가 되었고, 그 결과 모든 표준조항이 `MISSING`으로 나왔다.

실제로는 비교할 표준조항이 있었는데도, 시스템은 사용자 계약의 모든 조항에 “비교할 대상을 찾지 못했다”고 답하게 되는 상태였다.

검색이 어려운 문제가 아니었다. 서로 다른 의미의 점수를 같은 숫자처럼 비교한 것이 원인이었다.

## 0.033짜리 점수를 0.5와 비교했다

하이브리드 검색은 의미 검색과 키워드 검색(dense + BM25)의 순위를 순위 결합 점수(RRF)로 합쳤다. 당시 설정에서 RRF `fusion_score`의 이론상 최대값은 대략 다음과 같았다.

```text
1 / 61 + 1 / 61 ≈ 0.0328
```

그런데 파이프는 이 점수를 `match_threshold=0.5`와 직접 비교했다. 아무리 검색 순위가 좋아도 0.5에 도달할 수 없는 구조였다. 설계에서 전제했던 재정렬 모델(reranker) 단계도 실제 파이프에는 빠져 있었다.

이 버그가 테스트에서 보이지 않은 이유도 중요했다. 테스트용 가짜 검색기가 `rerank_score=0.95`를 고정해 반환하고 있었다. 테스트는 함수의 분기와 출력 형태를 검증했지만, 실제 검색기가 반환하는 점수의 의미와 범위는 검증하지 않았다.

파이프를 다음처럼 다시 연결했다.

```text
하이브리드 검색
→ 상위 후보 확보
→ reranker 재정렬
→ 재정렬 점수로 임계값 비교
→ NONE / EXTRA / NO_MATCH 분류
```

1편에서는 상태값 설명을 미뤘다. 이 시점에는 상태의 차이가 핵심이어서, 당시 의미를 분명히 할 필요가 있다.

- 검색 후보 자체가 없으면 `NO_MATCH`
- 후보는 있지만 임계값 미만이면 `EXTRA`
- 임계값 이상이면 `NONE`
- 초기에는 매칭 후보가 있지만 본문 차이가 크다고 본 경우를 `CHANGED`로 따로 분류했다.

당시 `NONE`은 단순히 후보를 찾았다는 뜻을 넘어, 내용도 충분히 대응된다는 의미로 쓰려던 상태였다. 이후 이 책임이 검색 점수에 과도했다는 사실이 드러난다.

RRF 점수는 후보를 합치는 순위 신호이고, reranker 점수는 질의와 후보 쌍을 다시 평가한 신호다. 둘 다 숫자지만 같은 계약을 갖지 않는다.

> 점수 기반 시스템에서는 값보다 먼저 그 값의 생성 과정, 범위, 소비 위치를 계약으로 고정해야 한다.

## 검색 성능과 내용 분류를 분리해서 보게 됐다

평가를 연결하면서 두 현상이 동시에 보였다.

- 정답 표준조항을 상위 후보로 가져오는 검색은 어느 정도 작동했다.
- `NONE`과 당시 존재하던 `CHANGED`를 나누는 결과는 한쪽으로 무너졌다.

처음에는 임계값을 조절하면 해결될 것으로 생각했다. 그러나 값을 바꿀 때마다 결과는 “전부 CHANGED”와 “전부 NONE” 사이를 오갔다.

문제는 임계값만이 아니었다. 첫 번째 문제는 정답지 자체가 당시 분류 규칙과 모순됐다는 점이었다.

## NONE이 될 수 없는 골든셋

v2 골든셋에서 `NONE`으로 표시된 44건 중 35건은 60~90자 정도의 짧은 사용자 문장을 여러 항으로 이루어진 긴 표준조항에 연결하고 있었다. 일부 표준조항은 28개 서브청크, 2,898자에 달했다.

당시 항 단위 대응 확인 절차는 표준조항의 모든 항이 사용자 조항에서 대응돼야 `NONE`을 반환했다. 짧은 문장 하나가 28개 항 전체를 포함할 수는 없다. 즉 35건은 어떤 임계값을 사용해도 `NONE`에 도달할 수 없는 정답이었다.

이미 앞선 리뷰에서 “사용자 조항을 조 전체 텍스트로 만들어야 한다”는 경고가 있었지만, 다음 골든셋 제작에 반영되지 않았다. 일부 케이스에는 표준조항 ID가 잘못 연결되거나, 강한 말바꿈을 `NONE`으로 볼지에 대한 기준도 일관되지 않았다.

다섯 번의 임계값 조정이 실패한 이유는 튜닝이 부족해서가 아니었다. 시스템이 만족할 수 없는 데이터 계약을 주고 숫자를 조절하고 있었다.

v3에서는 원칙을 바꿨다.

- 조 전체 텍스트를 사용한다.
- 현재 코드가 맞히기 쉽게 문장을 순화하지 않는다.
- 현실적인 계약 수정 문장과 올바른 정답 라벨을 먼저 쓴다.
- 코드가 틀리면 그 사실을 주석에 남긴다.
- 스키마와 코퍼스 참조를 검증기로 확인한다.

이 변화는 모델 개선보다 먼저 평가가 무엇을 측정하는지 복원하는 작업이었다.

## 관련성과 의미 동치성은 다른 문제다

골든셋을 고치면 평가의 모순은 사라진다. 하지만 데이터만 고쳐서는 해결되지 않는 두 번째 문제가 함께 드러났다. 검색 점수만으로 의미가 보존됐는지 판단할 수 없다는 문제였다.

`NONE`과 `CHANGED`를 가르는 질문은 “이 두 조항이 같은 주제를 다루는가?”가 아니다. “내용이 보존됐는가?”에 가깝다.

하지만 당시 1차가 가진 신호는 이 질문에 충분하지 않았다.

### 문자열 유사도

독립적으로 작성된 실제 계약서는 같은 의미라도 표현이 크게 달랐다. 당시 실계약 측정에서 문자열 유사도(`difflib`)의 천장은 약 0.545였다. 기준을 높이면 정상적인 말바꿈을 차이로 보고, 낮추면 실제 변경도 통과시켰다.

### reranker 점수

reranker는 두 문장이 같은 주제와 맥락에 있는지를 잘 찾는다. 그러나 같은 주제 안에서 의미가 반전된 경우도 높은 점수를 줄 수 있다.

예를 들어 “권리를 공동으로 보유한다”와 “한쪽이 권리를 영구적으로 보유한다”는 표현은 핵심 어휘와 주제가 매우 가깝다. 검색 관점에서는 좋은 후보지만, 내용 보존 관점에서는 별도의 해석이 필요하다.

### 부정어·숫자·당사자 규칙

마지막 방어선으로 부정어, 숫자, 당사자 변화를 정규식으로 찾는 규칙도 있었다. v3 케이스를 만들면서 다음 문제가 연이어 드러났다.

- “아니 된다”의 활용형을 놓침
- 숫자 뒤 목적격 조사 “을”을 당사자 `乙`로 오인
- “없다”, “없이” 같은 표현 미지원
- 부정어나 숫자가 없는 권리 관계 반전은 원천적으로 탐지 불가

표현을 하나씩 추가할 수는 있었다. 하지만 세 번째 예외를 발견했을 때 패치를 멈췄다. 한국어 부정 표현을 완전히 열거하는 일과, 표면 마커가 없는 의미 관계를 정규식으로 판단하는 일은 다른 문제였다.

강한 말바꿈일수록 규칙 검사를 위한 문장 정렬 점수가 낮아져 검사 자체가 꺼지는 역설도 있었다. 신호에 없는 정보를 규칙의 개수로 보충하고 있었다.

## 1차가 책임질 수 있는 신호만 남겼다

법률 판단은 처음부터 범위 밖이었다. 여기서 바뀐 것은 검색 점수와 규칙이 책임져야 한다고 생각했던 범위였다.

문제는 “표준 대비 다르게 쓰였다”는 내부 상태를 구현하는 과정에서, 검색 점수와 규칙에 의미 동치성까지 책임지게 한 것이었다. 이 책임을 다시 좁혔다.

- 규칙 기반 치명 변경 탐지를 제거했다.
- 문자열 유사도 기반 대체 경로를 제거했다.
- 항 단위 대응 확인 절차를 제거했다.
- `CHANGED` 상태를 제거했다.
- 임계값 이상의 매칭은 잠정 `NONE`으로 표현했다.
- 임계값 미만의 조항은 `EXTRA`로 두되, 최고 표준후보를 함께 반환했다.

EXTRA에 최고 후보를 남긴 것은 MCP 관점에서 특히 중요했다. 외부 클라이언트가 다시 검색하지 않고도 사용자 원문과 가장 가까운 표준 원문을 비교할 수 있기 때문이다.

현재의 역할은 다음처럼 정리됐다.

```text
1차 RAG MCP
├─ 비교할 표준조항 후보 검색·재정렬
├─ NONE / EXTRA / NO_MATCH / MISSING 상태
├─ 사용자 원문, 최고 표준후보와 출처, 점수 제공
└─ LLM 호출 없음

서버 밖 선택적 2차
└─ NONE/EXTRA의 원문 쌍을 읽고 의미 차이를 설명
```

## 두 번째 단계의 도착점

이 단계에서 얻은 결론은 “reranker를 믿을 수 없다”가 아니었다. reranker는 비교할 후보를 찾는 데 유용했다. 문제는 검색 관련성 점수를 그보다 더 큰 의미로 사용한 것이었다.

RAG는 비교할 근거를 찾는다. MCP는 그 근거와 실패 상태를 소비 가능한 계약으로 전달한다. 두 조항의 의미가 같은지에 대한 설명은 필요할 때 서버 밖 계층이 원문을 읽고 수행한다.

이제 남은 질문은 정확도를 한 번 높이는 방법이 아니었다.

> 데이터 포맷이 바뀌고, 표준계약서 버전이 늘고, 외부 모델 결과가 달라져도 같은 기준으로 시스템을 검증할 수 있는가?

다음 글에서는 독소 코퍼스를 40건에서 90건으로 늘리고도 recall이 전혀 개선되지 않았던 사건에서 시작한다. 이후 WorkShield는 모델 튜닝보다 데이터 표현 계약, 실행 trace, held-out, 그리고 실패한 실험을 되돌리는 절차를 먼저 갖추게 된다.
