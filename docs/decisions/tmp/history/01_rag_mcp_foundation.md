# 표준 대비 검토 후보를 찾는 RAG MCP의 골격을 만들다

> WorkShield 프로젝트 발전사 1/3  
> 작은 팀이 AI와 함께 개발하면서, 기능보다 먼저 경계와 데이터 계약을 고정한 기록이다.

사용자가 계약서의 대금 지급 조항을 넣으면, 시스템은 공개된 표준계약서에서 비교할 조항 후보를 찾는다. 사용자 계약에 대응 조항이 보이지 않거나, 표준에만 남은 조항이 있으면 그 사실도 숨기지 않고 돌려준다. 이후 사람이나 외부 AI가 두 원문을 다시 읽고 검토할 수 있게 하는 것이 WorkShield의 1차 역할이다.

처음부터 목표는 “이 조항은 불법이다”라고 말하는 서비스가 아니었다. 사용자 계약서를 표준계약서와 조항 단위로 비교해, 빠졌거나 추가됐거나 대응 관계가 불분명한 지점을 **검토 후보**로 찾는 것이었다. 법률 해석이나 유불리 판단은 생성하지 않고, 사람이 다시 볼 위치와 비교 근거를 제공한다.

이 목표에는 RAG와 MCP가 서로 다른 역할로 필요했다.

- RAG는 사용자 조항과 비교할 표준조항을 찾아오는 내부 엔진이다.
- MCP는 검색 결과와 실패 상태를 다른 AI나 클라이언트가 호출·소비할 수 있게 만드는 도구 계약이다.

이 글에서 가장 중요한 질문은 “무엇을 검색할 것인가?”만이 아니었다.

> 검색 결과가 어디까지 말할 수 있게 할 것인가?

## “보수”가 “하자보수”를 대금 지급으로 분류했을 때

초기에는 표준조항의 조항 분류를 키워드로 정했다. 그런데 `"보수" in text` 같은 규칙이 “하자보수”를 대금 지급 카테고리로 분류했다.

문자열 한 줄의 문제처럼 보였지만, 검색 시스템에서는 후보군을 잘못 만드는 문제였다. 조항 분류를 검색 전 필터로 쓰려 했기 때문에, 잘못된 분류는 정답 표준조항을 후보군 밖으로 밀어낼 수 있었다. 재정렬 모델이 아무리 좋아도 처음부터 검색하지 못한 후보를 복구할 수는 없다.

팀은 키워드 규칙을 카테고리별 앵커 문장과 임베딩 유사도를 비교하는 방식으로 바꿨다. 여기서도 실제 데이터가 설계를 바꿨다. 앵커를 평균 내면 “납품”과 “검수”처럼 서로 다른 의미가 희석됐다. 그래서 가장 가까운 앵커의 점수를 쓰는 방식(max-pool)을 적용하고, 제목 단독 질의와 제목+본문 질의 중 더 높은 점수를 선택했다.

합성 문장과 실제 계약서의 점수 분포도 달랐다. 합성 테스트는 앵커 표현을 그대로 포함해 높은 점수를 얻었지만, 실제 계약 문체의 조항은 훨씬 낮은 범위에 모였다. 테스트에서는 잘 맞는 것처럼 보여도, 합성 데이터에서 정한 기준을 그대로 적용하면 실제 계약서에서는 정답 후보를 스스로 거의 모두 지워버릴 수 있었다.

이 사례는 이후 내린 여러 결정의 출발점이 됐다.

> RAG의 품질은 모델을 호출하는 코드보다, 정답이 후보군에 들어올 수 있게 만드는 데이터 계약에서 먼저 결정된다.

## 협업에서는 기능보다 먼저 결과의 의미를 맞춰야 했다

프로젝트는 5명이 짧은 기간에 병렬로 구현해야 했다. AI 코딩 도구는 구현 속도를 높였지만, 서로 다른 전제를 가진 코드도 같은 속도로 늘릴 수 있었다.

이 글은 개별 구현 기록이 아니라, 프로젝트 리드 관점에서 경계 설정과 통합 과정에서 바뀐 판단을 정리한 회고다. 초기 구조와 통합 기준은 내가 설계했고, 정규화·인덱싱·검토 파이프라인·평가 하니스는 팀이 모듈 단위로 나누어 구현했다.

예를 들어 조항 모델, 검색 결과의 상태값, 데이터 스키마를 서로 다르게 해석하면 각 모듈은 개별적으로 동작해도 통합 시점에 결과의 의미가 달라질 수 있다. 그래서 첫 결정은 검색 모델이나 벡터 DB가 아니었다. 여러 모듈이 공유할 경계부터 고정했다.

```text
동결 계약: 모델·상태값·인터페이스
        ↓
순수 로직: 외부 모델과 DB를 직접 알지 않음
        ↓
외부 연동: 문서 파싱·검색·법령 조회
        ↓
실행 흐름 조립
        ↓
MCP 도구 표면
```

테스트에서는 실제 임베딩 모델 대신 가짜 구현을 주입할 수 있게 했다. 팀원이 서로 다른 모듈을 맡아도 공통 계약을 지키면 나중에 조립할 수 있게 하기 위해서다.

데이터에도 같은 원칙을 적용했다.

> 정답은 Git에 두고, 인덱스는 다시 만든다.

표준조항을 정규화한 JSON과 migration SQL을 원천으로 관리하고, SQLite와 Chroma 인덱스는 빌드 명령으로 재생성하게 했다. 바이너리 인덱스를 정답처럼 보관하면 변경 이유를 리뷰하기 어렵다. 반면 정규화된 조항 데이터는 어떤 라벨과 문장이 바뀌었는지 diff로 확인할 수 있다.

이 선택은 단순한 저장 방식이 아니었다. 검색 결과가 이상할 때 모델, 인덱스, 원천 데이터 중 어디를 의심해야 하는지 구분할 수 있는 토대가 됐다.

## 검색 결과도 실패를 숨기지 않게 했다

최초의 흐름은 다음과 같았다.

```text
계약서 파싱
→ 조항 단위 분해
→ 계약유형으로 검색 범위 축소
→ 의미 검색과 키워드 검색(dense + BM25)을 결합한 하이브리드 검색
→ 재정렬 모델(reranker)로 후보 재정렬
→ 표준조항 후보와 명시 상태 반환
```

MCP가 단순 후보 목록만 반환하면, 그 결과를 쓰는 외부 AI는 빈 목록의 이유를 알 수 없다. 그래서 실패도 상태로 드러내기로 했다.

- `NO_MATCH`: 비교할 표준조항 후보 자체를 찾지 못함
- `MISSING`: 표준에는 있지만 사용자 계약 전체에서 대응을 찾지 못함

예를 들어 사용자 조항 하나를 넣었는데 비교할 표준 후보를 찾지 못하면 `NO_MATCH`다. 반대로 표준계약서에는 있어야 할 조항이 사용자 계약 전체에서 대응되지 않으면 `MISSING`이다.

이 신호들은 법률적 결론이 아니다. 검색과 매칭 과정에서 무엇이 일어났는지를 외부 소비자에게 전달하기 위한 정보다. 특히 `NO_MATCH`를 빈 배열로 숨기지 않는 것은 중요했다. 외부 AI가 빈 결과를 보고 없는 내용을 추측하지 않게 하기 위해서다.

초기에는 본문 차이까지 별도 상태로 분류하려는 시도도 있었다. 이 상태는 다음 글에서 다룰 실제 평가와 함께 제거된다. 여기서 중요한 것은 결과를 단순히 성공·실패로 뭉개지 않고, **어떤 실패인지 계약으로 남겼다**는 점이다.

## 표준 코퍼스에서는 조항을 버리지 않는다

어떤 카테고리에도 충분히 가깝지 않은 표준조항을 예외 처리하거나 건너뛰는 방식도 검토했다. 하지만 표준 코퍼스에서 조항 하나를 버리면 런타임에서 복구할 방법이 없다. 그 조항이 사용자 계약에 빠져 있어도 `MISSING`으로 찾을 수 없기 때문이다.

그래서 특정 카테고리에 속하지 않는 조항을 위한 `GENERAL`을 만들었다. `GENERAL`은 임베딩 경쟁에 참여하는 또 하나의 의미 범주가 아니라, 표준조항을 유실하지 않기 위한 명시적 대체 경로였다.

한 조항이 여러 카테고리에 걸리는 경우에도 당시 MVP에서는 다중 라벨보다 결정론적 우선순위를 택했다. 완벽한 도메인 분류기를 만드는 것보다, 모든 표준조항을 기준선에 남기고 같은 입력을 같은 검색 범위로 보내는 것이 우선이었다.

## 긴 조항의 일부가 빠진 경우

다음 문제는 긴 조항이었다. 예를 들어 표준조항에는 납품, 검수, 하자 통지, 보수 기한이 함께 들어 있는데 사용자 계약에서는 하자 통지 항목만 빠져 있을 수 있다. 표준조항에 항이 10개 있고 1개만 빠졌다고 가정하면, 나머지 9개가 같아 조항 전체 임베딩 유사도는 여전히 매우 높다. 당시 설계 검토의 예시 계산에서는 약 0.97까지 나올 수 있다고 봤다. 단순히 임계값을 높이는 것으로는 부분 누락을 안정적으로 찾기 어려웠다.

처음에는 조항을 세부 항목(항·호)으로 나누고, 표준과 사용자 계약의 항목을 정렬한 뒤 당사자·수치·표현을 비교해 차이를 계산하는 상세 설계를 검토했다.

문제는 이 설계가 검색 보조 기능을 넘어, “어디가 어떻게 달라졌는가”를 해석하는 규칙 엔진으로 커진다는 점이었다. 동결 계약에도 여러 모델과 상태를 추가해야 했고, 문장이 하나에서 여러 개로 나뉘거나 여러 문장이 하나로 합쳐지는 경우까지 처리하려면 규칙은 빠르게 복잡해졌다.

그래서 이 상세 비교 설계는 철회했다. 대신 긴 조항을 항 단위 조각으로 나누고, 가장 높은 세부 항목 점수로 부모 조항 후보를 올리는 방식(Max roll-up)을 선택했다. “하나라도 잘 맞으면 우선 부모 조항을 확보한다”는 검색 목적에 맞춘 결정이었다.

이 차이가 중요했다. 항 단위 기술은 후보를 더 잘 찾는 데는 유용했지만, 내용 차이를 확정하는 근거가 되지는 못했다. 같은 기술이라도 **검색 장치**와 **분류 장치**가 책임지는 질문은 달랐다.

## 첫 번째 단계의 도착점

이 시점의 WorkShield에는 다음 골격이 생겼다.

- 리뷰 가능한 표준조항 원천 데이터
- dense와 BM25를 결합한 검색
- reranker를 연결할 수 있는 외부 연동 경계
- 긴 조항을 위한 항 단위 검색과 부모 조항 후보 결합
- 검색 실패를 숨기지 않는 MCP 상태 계약
- 외부 클라이언트가 결과를 이어서 사용할 수 있는 도구 표면

익숙하지 않은 전문 도메인을 다루면서, 우리는 해석을 더 많이 코드화하는 대신 표준 원문과 검색 근거를 보존하고 시스템이 말할 수 있는 범위를 좁히는 쪽을 택했다. 이 판단은 이후 실제 평가 실패를 겪으며 더 분명해진다.

하지만 아직 중요한 검증이 남아 있었다. 각 모듈의 테스트는 통과했지만, 실제 검색 점수와 실제 reranker를 전체 파이프에 연결했을 때도 같은 가정이 성립하는지는 확인되지 않았다.

다음 글은 바로 그 지점에서 시작한다. 실제 모델과 평가 데이터를 연결하자 RRF 점수 계약 결함으로 모든 조항이 `NO_MATCH`가 되었다. 그 결함을 수정한 뒤, 별도의 점수 정규화 결함에서는 반대로 거의 모든 조항이 ‘문제없이 대응됨’ 상태가 되었다.

그 과정에서 가장 크게 바뀐 것은 기술 스택이 아니었다. **검색 점수에 어떤 의미까지 부여할 수 있는가**에 대한 생각이었다.
