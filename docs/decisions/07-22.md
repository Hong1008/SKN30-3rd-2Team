# 의사결정 로그 — 2026-07-22

> 전체 MCP 도구 명세 감사와 호환성 강화 구현 결과를 기록한다. 기존 공개 계약을 크게
> 변경하지 않고 신규 권장 도구를 중심으로 사용 경로를 정리했으며, 현 상태를 MVP에서
> 실제 클라이언트 연동에 사용할 수 있는 기준선으로 확정했다.

## 0. 세션 결론

- 현재 MCP는 **MVP에서 사용 가능**하다. 추가 v2 도구는 출시 차단 조건이 아니다.
- 신규 클라이언트의 권장 전체 흐름은 다음으로 고정한다.

```text
assess_contract_scope
→ review_contract_candidates
→ 필요한 조항만 get_category_grounding
```

- 부분 검토는 `parse_contract_clauses → classify_clause_candidate`를 사용한다.
- `parse_contract`, `review_contract`, `classify_clause`, `get_grounding`은 기존 응답을
  유지하는 호환 경로이며 신규 클라이언트의 기본 경로로 사용하지 않는다.
- 추가 도구를 계속 늘리기보다 현재 명세를 기준선으로 고정하고 실제 클라이언트 연동으로
  넘어간다.

## 1. 검토와 법령 조회 책임 분리

- `review_contract_candidates`는 계약서 파싱, 표준조항 검색·재정렬·분류, `MISSING` 탐지와
  주의 문구 신호만 반환한다.
- 이 권장 검토 응답에는 `grounding` 필드를 두지 않는다.
- 법령 원문은 사용자가 필요로 하는 조항에 한해 `get_category_grounding`으로 별도 조회한다.
- `get_category_grounding`은 다음 상태를 구분하며, `OK`일 때만 최소 한 건의 법령을 반환한다.

```text
OK
NO_RESULT
UNMAPPED_CATEGORY
UPSTREAM_ERROR
TIMEOUT
```

- 기존 `review_contract`의 조건부 법령 첨부와 `get_grounding`의 자유 법령명 질의는
  하위호환을 위해 유지한다.

## 2. 공개 DTO와 도구 발견

- 신규 권장 도구는 내부 `src/contracts` 모델을 직접 공개하지 않고 `server/public_dto.py`의
  독립 DTO를 사용한다.
- 동결된 `src/contracts`의 enum·모델·port는 변경하지 않았다.
- `MatchClauseResponse`도 상태와 배열 관계를 강제한다.
  - `OK`이면 후보가 최소 한 건이다.
  - `NO_RESULT`이면 후보는 빈 목록이다.
- 신규 `get_mcp_capabilities` 도구를 추가해 권장 전체·부분 흐름과 기존 호환 도구의 대체
  관계를 구조화해 반환한다.

## 3. 입력 스키마와 파이프라인 상태 강화

- `match_clause.top_k`는 `1..10`으로 제한한다.
- `classify_clause`와 `classify_clause_candidate`의 `match_threshold`는 `0.0..1.0`으로 제한한다.
- 위 범위는 런타임뿐 아니라 FastMCP JSON Schema에도 노출한다.
- 검토 파이프라인 내부의 `CorpusUnavailableError`, `EmptyDocumentError`는 각각
  `CORPUS_UNAVAILABLE`, `EMPTY_DOCUMENT`로 보존한다.
- 벡터 검색 hit가 있지만 SQLite 표준조항과 하나도 결합되지 않는 경우 사용자 문구의
  `EXTRA`로 숨기지 않고 `CORPUS_UNAVAILABLE`과 인덱스 재생성 안내를 반환한다.
- 공개 오류 메시지에서 내부 팀원·구현 담당자 표현을 제거한다.

## 4. 외부 korean-law MCP 프록시

- 외부 `CallToolResult.isError=true`는 정상 `result` 문자열로 반환하지 않고 오류로 처리한다.
- 공유 락 대기와 실제 외부 도구 호출에 기본 30초 제한을 적용한다.
- 공개 예외에는 외부 서버의 내부 경로·토큰·상세 오류 원문을 포함하지 않는다. 상세 정보는
  서버 로그에만 남긴다.
- 기존 성공 출력인 `{result: string}` 계약은 유지한다.
- 다음 입력 제약을 FastMCP Schema와 wrapper 런타임에서 함께 검증한다.
  - `search_law.display`: `1..100`
  - `legal_research.task`, `legal_analysis.mode`, 결정 검색 `domain`, 별표 `knd`: 명시 enum
  - 법령 식별자·별표 일련번호: 6자리
  - 조문, 시행일, 별표 번호: 문서화된 형식
  - `legal_analysis`: mode별 필수 arguments
  - `search_decisions.options`: 허용 키와 값 형식
- `legal_research`, `legal_analysis` 등은 같은 MCP에 노출된 선택적 외부 프록시다. 그 결과는
  참고 자료이며 WorkShield의 결정론적 표준 대비 판정을 변경하거나 법률적 결론을 확정하지 않는다.

## 5. 표준조항 리소스 사용 범위

- `standard://{contract_type}`과 `standard://{contract_type}/{clause_id}`는 유지한다.
- 두 리소스는 표준조항 독립 탐색 또는 저장한 `clause_id`의 재조회에 사용한다.
- 권장 전체 검토 응답에는 대응 표준조항 본문이 이미 있으므로 검토 뒤 리소스를 항상 추가
  호출하지 않는다.

## 6. 보류한 개선

다음은 실제 사용을 막지 않으므로 이번 기준선에는 포함하지 않는다.

1. 파일 입력의 `file_path` 또는 `file_content + file_name` 배타 조건을 discriminated union으로
   표현하는 신규 v2 도구
2. 외부 법령·판례 프록시의 `{result: string}`을 구조화 DTO로 바꾸는 신규 v2 도구
3. `execute_tool`의 동적 외부 도구 표면을 별도 허용 목록으로 제한하는 신규 실행 도구

현재도 잘못된 파일 조합은 런타임에서 명시적 입력 오류로 반환한다. 계약 검토의 권장 법령
조회인 `get_category_grounding`도 이미 구조화된 상태를 제공한다. 위 개선은 실제 클라이언트
사용 중 반복적인 호출 오류나 문자열 파싱 문제가 확인될 때 별도 결정으로 착수한다.

- 실제 FastMCP 공개 표면: 도구 24개, 리소스 템플릿 2개
- `site/mcp-spec.json`을 현재 등록 상태로 재생성했다.
- 단위 테스트: **271 passed, 18 deselected**
- `compileall`, `git diff --check` 통과
- 활성 v5 평가 기준·임계값과 동결 held-out은 변경하거나 재실행하지 않았다.
