# 의사결정 로그 — 2026-07-10

> 07-09에 분리한 Q(표준판 혼재) 카드를 active corpus + archive 정책으로 완료하고, v3 전체·tuning·held-out
> 평가를 다시 구성해 RRF 및 이탈·독소 임계값 후보를 검증한 세션. 별도 `v3_review.md`는 작성하지 않고,
> 카드와 이 로그에 결과·결정을 기록한다.

## 0. 세션 결론

최신 표준판만 활성 검색 대상으로 삼고 SI·SM 2022판은 archive로 분리해 Q를 완료했다. 그 결과 A-1
`hybrid_rerank_5_5`는 Recall@5=0.920·MRR=0.903으로 회복됐다. 그러나 임계값 held-out에서 tuning 후보가
일관되게 우세하지 않아, `match_threshold=0.50`·`toxic_threshold=0.60` 기본값은 **변경하지 않기로** 했다.
1차 시스템은 사람의 계약 검토를 돕는 "검토 후보" 도구로 제한하며, 독소 자동 위험판정 용도의 상용 투입은
현재 지표만으로는 보류한다.

## 1. Q 완료 — active corpus + archive

### 확정 정책

- 1차 MVP는 계약유형별 **현재 최신 표준판 하나**와만 비교한다. SI·SM은 2025, SW 유형은 현재 보유한
  2020판을 활성 기준으로 사용한다.
- SI·SM 2022 표준조항·서브청크 JSON 4개는 `data/03_normalized/archive/`로 이동해 Git 이력으로 보관한다.
  `0.migrate.py`의 비재귀 glob 대상이 아니므로 SQLite·Chroma 검색 풀에는 적재되지 않는다.
- 특정 체결 시점의 과거 표준 비교는 이번 범위에서 제외한다. 필요 시 동결 MCP 시그니처에 version 입력을
  추가하는 별도 사인오프 카드로 다룬다.

### 구현·검증

- `0.migrate.py::validate_single_active_version`을 추가했다. 활성 루트에 동일 계약유형의 복수 version이
  있으면 archive 이동 방법을 포함한 `ValueError`로 마이그레이션을 중단한다.
- `just build-db` 후 활성 SQLite는 SI 2025 56건·SM 2025 63건·SW_EMPLOYMENT 2020 18건·
  SW_FREELANCE 2020 23건, 총 160건만 적재됨을 확인했다.
- 파이프·평가·서버 관련 단위 테스트 42건 및 v3 골든 스키마 117건 검증을 통과했다.
- 구버전 후보는 활성 Chroma에 존재하지 않으므로, 07-09에서 관찰된 `v3-si-41`·`v3-si-44`의 2022판
  매칭 경로는 제거됐다. `v3-si-45`는 2025판과의 경계 사례로 별도 매칭 분석 대상이다.

## 2. A-1 RRF 가중치 스윕

| 변형 | Recall@5 | MRR | 결정 |
| --- | ---: | ---: | --- |
| hybrid 5:5 | 0.920 | 0.901 | 기준 |
| hybrid rerank 5:5 | 0.920 | 0.903 | 현행 기본 유지 |
| hybrid 7:3 / rerank | 0.920 | 0.903 | 5:5와 동률 |
| hybrid 8:2·9:1 / rerank | 0.907 | 0.900 | recall 하락 |

`review_contract`의 기본 동일가중 5:5는 성능상 열세가 아니므로 그대로 유지한다. 가중치 파라미터는
평가용으로 유지하되, 프로덕션 경로에 별도 배선하지 않는다.

## 3. 임계값 보정 — 고정 held-out 도입과 결과

골든 JSON 스키마를 바꾸지 않고 `threshold_heldout.v3.json` manifest를 추가했다. held-out은 39건
(SI 15·SM 15·SW 9), tuning은 78건이며 각 유형 held-out에 이탈·독소 양성을 포함한다. `run_eval.py`는
전체합산뿐 아니라 tuning·held-out 각각의 스윕을 출력한다.

### 이탈 match_threshold

- tuning: 0.40·0.45 F1=0.792, 0.50 F1=0.778.
- held-out: 0.40·0.45·0.50 모두 F1=0.720(TP=9·FN=5)으로 동률.
- **결정:** 0.45의 개선이 held-out에서 재현되지 않았으므로 현행 **0.50 유지**.
- 0.40~0.60 전 구간에서 전체 FN=12로 동일하다. 남은 FN은 임계값이 아니라 표준과 의미적으로 가까운
  EXTRA의 과매칭 한계다.

### 독소 toxic_threshold

- tuning: 0.75 F1=0.333, TP=5·FP=13·FN=7로 가장 높았다.
- held-out: 0.75 F1=0.300, 0.85·0.90 F1=0.333으로 순위가 뒤바뀌었다.
- **결정:** held-out 독소 양성이 6건뿐이고 tuning 후보가 재현되지 않았으므로 현행 **0.60 유지**.
  0.75·0.85 어느 값도 새 기본값으로 채택하지 않는다.

## 4. 독소·이탈 오류 분석 결과

### 독소

- 전체 기본 0.60: TP=8·FP=31·FN=10, P=0.205·R=0.444·F1=0.281.
- 0.75는 TP/FN을 바꾸지 않고 FP만 31→24로 낮추지만, held-out에서 우위를 재현하지 못했다.
- 현재 결과에는 케이스별 top-3 독소 후보·pattern_id·rerank_score가 남지 않아 FP/FN을 검색 누락,
  점수 미달, 엉뚱한 패턴 매칭으로 분해할 수 없다.

### 이탈

- 전체 기본 0.50: TP=30·FP=7·FN=12·TN=68, F1=0.759.
- SI FN 6건이 전체 FN의 절반이다. `v3-si-45`, 이어 `v3-si-41`·`v3-si-44`의 Q 이후 상태를 우선
  진단한다.
- 현재 결과에는 `case_id`별 confidence·matched standard id/version/category·후보 출처가 없어
  과매칭과 검색 실패를 구분할 수 없다.

## 5. 상용 범위 판단

- **가능:** 최신 표준판 기준으로 사람 검토를 보조하는 제한적 데모·파일럿. 표면 문구는 계속

## 6. MCP 조립 루트·법률 보조도구 DI 전환

### 확정 정책

- `FastMCP`는 더 이상 `server.py` 모듈 전역에서 생성하지 않는다. `app.py::create_app()`이
  composition root로서 서버 인스턴스를 만들고 도구 등록기들을 생성자 주입한다.
- 기존 WorkShield 도구·`standard://` 리소스는 `WorkShieldTools`가 등록한다. 기존 도구 이름과
  입출력 표면은 유지한다.
- 2차 LLM이 법률 리서치·판례·조문 조회를 조합할 수 있도록 `KoreanLawWrapper`가
  `korean-law-mcp`의 9개 도구를 별도 MCP 도구로 노출한다. 1차 파이프라인은 이 도구들을
  해석·판정에 사용하지 않는다.

### 구현·검증

- `create_app()`의 MCP 등록 계약 테스트에서 WorkShield 도구, 법률 보조 9개 도구, 표준조항
  리소스 템플릿을 함께 확인한다.
- `KoreanLawMCPClient.query()`는 존재하지 않는 외부 `query` 도구 호출을 제거하고,
  정확 법령명 `search_law → get_law_text` 체인으로 변경했다. 정확 일치하지 않으면 1차에서는
  법령 본문을 붙이지 않는 `NO_RESULT` 경로로 처리한다.

## 7. 지원 범위 사전 판별

### 확정 정책

- 기존 `review_contract`의 동결 입력·출력은 변경하지 않는다. 대신 새 확장 도구
  `assess_contract_scope`를 추가한다.
- 파싱된 본문에 대한 결정론적 앵커 규칙만 사용해 `IN_SCOPE` /
  `CONTRACT_TYPE_UNCERTAIN` / `OUT_OF_SCOPE`를 반환한다. 파일명과 LLM 판단은 사용하지 않는다.
- 인접 도메인 문서는 `CONTRACT_TYPE_UNCERTAIN` 경고를 표시하되, 사용자가 `contract_type`을
  명시하면 기존 `review_contract` 검토를 허용한다. `OUT_OF_SCOPE`는 검토 대상에서 제외하도록
  안내한다.

### 골든 기준

- `src/eval/golden_b/scope_manifest.json`에 SW 문서, 농축어업·선원 문서(범위 외),
  방송·영상 문서(유형 불확실)를 구분해 기록한다.
- 농업 표준근로계약서 HWP·선원법 PDF는 이번 E2E에서 파싱 0건(`EMPTY_DOCUMENT`)이었으며,
  SW 프리랜서 표준 대비 검토 품질 지표에서 제외한다.

## 8. 법령 MCP 장기 세션·캐시

### 확정 정책

- `korean-law-mcp` CLI를 컨테이너에 설치한 사실만으로는 캐시가 재사용되지 않는다. 기존 구현은
  도구 호출마다 stdio 자식 프로세스를 생성·종료했다.
- WorkShield 앱 lifespan에서 stdio 자식 프로세스와 `ClientSession` 하나를 열고 종료 시 닫는다.
  AnyIO 컨텍스트는 같은 owner task에서 열고 닫도록 해 cancel-scope 종료 오류를 방지한다.
- 1차·2차 법률 도구는 같은 `KoreanLawMCPClient` 인스턴스를 공유한다. 세션 내 호출은 lock으로
  직렬화한다.
- 캐시는 DB를 추가하지 않고 두 계층으로 둔다.
  - 요청 캐시: `review_contract` 한 번 안에서 `(Category, ContractType)` 결과 재사용 후 즉시 폐기.
  - 프로세스 캐시: 정적 카테고리 grounding만 24시간 TTL·최대 64건 LRU로 보관. 성공·`NO_RESULT`만
    저장하며 오류와 사용자 자유 조항 질의는 저장하지 않는다.

### 검증

- 실제 공개 법령 검색에서 동일 세션 재사용과 `client.close()` 후 정상 종료를 확인했다.
- lifecycle·캐시·범위 판별·파이프 관련 테스트 42건을 통과했다.

## 9. 법령 grounding 폭증 수정 및 Golden E2E

### 원인과 결정

- `jo` 6자리 형식은 정상 동작했다. 폭증의 주원인은 조문번호 없는 `"민법 도급"` 기본 폴백과
  사회보험 등 포괄 질의가 법령 전문을 반환한 것이었다.
- 기존 파서는 본문 속 `제N조` 인용까지 조문 헤더로 오인했다.
- 1차 정적 grounding은 **조문번호가 명시된 매핑만 허용**한다. 조문번호 없는 정적 질의와 미매핑
  카테고리는 외부 호출 없이 빈 목록(`NO_RESULT` 경로)으로 처리한다.
- 실제 줄 시작 조문 헤더만 파싱하고, 빈 본문 제거·`(법령명, 조번호)` 안정 중복 제거·최대 3건 상한을
  적용한다. 법령 전문을 그대로 수백·수천 건의 근거로 노출하지 않는다.

### Golden E2E 전후 비교

| 지표 | 수정 전 | 수정 후 |
| --- | ---: | ---: |
| 정상 검토 문서 | 12 | 12 |
| 총 법령 근거 | 82,080 | 101 |
| grounding 결과당 법령 수 | 448.5 | 1.0 |
| 정상 문서 평균 처리시간 | 66.1초 | 19.1초 |
| 중앙 처리시간 | 64초 | 15.55초 |
| 빈 법령 본문 | 0 | 0 |

- 총 법령 근거는 99.88%, 전체 처리시간은 70.9% 감소했다.
- E2E 요약의 `grounding_duplicate_count`는 문서 전체의 여러 MISSING 결과에 같은 조문이 붙은 것을
  평탄화해 센 값이다. 각 grounding 결과는 법령 1건이므로, 한 결과 안의 중복 조문 문제는 해소됐다.
