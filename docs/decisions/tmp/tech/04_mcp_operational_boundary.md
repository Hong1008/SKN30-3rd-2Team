# MCP를 운영 가능한 도구로 만드는 경계

> 프로젝트 저장소: [WorkShield — SKN30 3차 프로젝트](https://github.com/SKNETWORKS-FAMILY-AICAMP/SKN30-3rd-2Team)  
> 이 글은 위 프로젝트에서 표준계약서 대비 검토 후보를 찾는 RAG MCP를 구현하며 실제로 겪은 운영 문제와 경계 설계 과정을 정리한 글이다. 법률적 판단이나 조항의 위법성을 설명하지 않고, MCP가 외부 클라이언트와 어떤 상태·범위·실패 계약을 맺어야 하는지를 다룬다.

프로젝트의 전체 발전 과정은 [잘 찾는 도구에 신뢰 기준을 세우다](https://velog.io/@hong1008/ai-camp-3rd-project-3)에서 다뤘다. 앞선 기술 글에서는 RAG가 무엇을 검색해야 하는지, 점수의 의미를 어떻게 분리하는지, 질의와 코퍼스의 표현 계약을 어떻게 맞추는지를 정리했다. 이번 글은 그 검색 엔진을 외부에서 호출하는 MCP(Model Context Protocol, 모델 컨텍스트 프로토콜) 도구의 경계를 다룬다.

처음에는 MCP 서버가 도구를 등록하고 호출에 응답하면 충분하다고 생각했다. 실제 운영 흐름에서는 그렇지 않았다. 호출은 성공했지만 응답이 비어 있었고, 지원하지 않는 문서가 검토되었으며, 외부 법령 도구가 반환한 결과가 시스템을 압도했다. 그때부터 MCP의 책임을 “기능을 연결하는 서버”가 아니라 “외부 클라이언트가 다음 행동을 결정할 수 있게 하는 경계”로 다시 정의했다.

이 글에서는 그 경계를 세 가지로 나눈다.

1. **의미 경계:** 빈 결과와 실패를 서로 다른 상태로 구분한다.
2. **범위 경계:** 지원하지 않는 입력과 외부 도구의 출력을 제한한다.
3. **수명 경계:** 세션·프로세스·캐시가 언제 시작되고 끝나는지 관리한다.

이 문제들은 MCP에만 존재하는 것은 아니다. 일반 API와 외부 연동에도 나타난다. 다만 MCP에서는 호출자가 사람이 아니라 LLM 기반 클라이언트일 수 있다. 모호한 빈 응답이나 과도한 출력이 잘못된 다음 행동으로 바로 이어질 수 있다는 점에서, 상태와 책임의 경계를 더 분명히 해야 했다.

본문에서 `status`는 도구 실행 상태, `grounding`은 검색한 법령 근거를 결과에 연결하는 작업, `TTL(Time To Live, 유효 시간)`은 캐시가 살아 있는 시간, `LRU(Least Recently Used, 최근 사용 빈도가 낮은 항목부터 제거하는 방식)`는 캐시 축출 정책으로 설명한다. `client`는 MCP를 호출하는 외부 AI나 애플리케이션을 뜻한다.

## 1. 의미 경계: 빈 응답은 운영 가능한 상태가 아니었다

실제 계약서를 파이프라인에 넣었을 때 가장 먼저 마주친 문제는 기능 오류보다 응답 의미의 모호함이었다.

```text
사용자: 계약서를 검토해 주세요.
MCP: results=[]
```

이 결과만으로는 다음 상황을 구분할 수 없다.

- 문서에서 조항을 하나도 파싱하지 못했는가?
- 표준 코퍼스(corpus, 검색 대상 문서 모음)가 없는가?
- 검색 후보가 없는가?
- 법령 근거만 조회되지 않은 것인가?
- 정말 반환할 검토 후보가 없는가?

외부 클라이언트가 이 차이를 알 수 없으면 빈 목록을 “문제 없음”으로 표시할 가능성이 생긴다. 따라서 목록보다 먼저 실행 상태를 계약으로 만들었다.

### 실행 상태와 조항 상태는 다른 계층이다

처음부터 모든 상태를 하나의 `status` 목록으로 생각하면 응답 구조가 다시 모호해진다. 실제로는 세 계층을 나누어야 한다.

| 계층 | 예시 | 질문 |
| --- | --- | --- |
| 도구 실행 상태 | `OK`, `EMPTY_DOCUMENT`, `CORPUS_UNAVAILABLE`, `INVALID_CONFIG` | 도구 호출 자체가 처리되었는가? |
| 조항별 검색 상태 | `NONE`, `EXTRA`, `NO_MATCH`, `MISSING` | 각 조항을 표준과 어떻게 비교했는가? |
| 외부 grounding 상태 | `OK`, `NO_RESULT`, `INVALID_INPUT` | 이번 근거 조회가 어떤 결과였는가? |

예를 들어 전체 검토 응답은 실행 상태가 `OK`일 수 있고, 그 안의 개별 조항은 `NO_MATCH`일 수 있다. 반대로 실행 상태가 `EMPTY_DOCUMENT`이면 조항별 결과 자체가 존재하지 않는다. 따라서 상위 응답의 상태와 항목별 상태를 같은 의미로 읽으면 안 된다.

### 조항 상태는 서로 다른 질문에 답한다

`review_contract`의 결과에서 사용하는 상태는 법률적 결론이 아니라 검색 파이프라인에서 일어난 일을 설명한다.

| 상태 | 의미 | 비교 방향 |
| --- | --- | --- |
| `NONE` | 대응하는 표준조항 후보를 찾음 | 사용자 조항 → 표준조항 |
| `EXTRA` | 가까운 후보는 있으나 충분한 대응으로 보기 어려움 | 사용자 조항 → 표준조항 |
| `NO_MATCH` | 검색 후보 자체를 찾지 못함 | 사용자 조항 → 표준조항 |
| `MISSING` | 표준조항이 계약서 전체에서 대응되지 않음 | 표준조항 → 계약서 전체 |

`NONE`이라는 이름은 한 가지 기술 부채를 남겼다. 기존 enum(열거형)과 소비자 호환성 때문에 유지했지만, 현재 의미는 “법적으로 아무 문제 없음”이 아니라 “threshold(판정 임계값)를 넘은 대응 후보가 있음”에 가깝다. 새 계약을 설계한다면 `MATCHED_CANDIDATE`처럼 더 직접적인 이름을 검토할 것이다. 외부 클라이언트는 이 값을 안전성이나 적법성으로 번역해서는 안 된다.

`NO_MATCH`와 `MISSING`은 방향이 반대다. 계약서에 있는 조항의 비교 대상을 못 찾은 것과, 표준에는 있지만 계약서 전체에서 보이지 않는 것을 같은 실패로 합치면 외부 클라이언트가 후속 행동을 결정할 수 없다.

법령 조회도 같은 원칙을 적용한다.

```json
{
  "status": "NO_RESULT",
  "grounding": [],
  "message": "명시된 조문을 조회하지 못했습니다."
}
```

`grounding=[]`는 관련 법령이 없거나 계약이 안전하다는 뜻이 아니다. 이번 호출에서 명시된 근거를 조회하지 못했다는 뜻이다.

문서 파싱 결과도 빈 배열로 끝내지 않는다.

```json
{
  "status": "EMPTY_DOCUMENT",
  "contract_type": null,
  "results": [],
  "message": "조항을 찾을 수 없습니다. 스캔 PDF이거나 '제N조' 형식이 없는 문서일 가능성이 있습니다."
}
```

변경 전과 변경 후를 비교하면 다음과 같다.

```text
변경 전
문서 공백·검색 실패·코퍼스 부재 → 빈 배열

변경 후
문서 공백       → EMPTY_DOCUMENT
후보 없음       → NO_MATCH
근거 없음       → NO_RESULT
코퍼스 사용 불가 → CORPUS_UNAVAILABLE
```

운영 도구에서 중요한 것은 실패를 없애는 것이 아니라 실패의 종류를 숨기지 않는 것이다.

## 2. 범위 경계: 지원하지 않는 문서를 먼저 판별해야 했다

두 번째 문제는 `review_contract`가 어떤 문서든 곧바로 지원 표준계약서와 비교하려는 흐름이었다. 파일이 파싱되었다는 사실과 현재 코퍼스가 그 문서를 지원한다는 사실은 다르다.

지원 범위 밖 문서를 그대로 검토하면 다음과 같은 오해가 발생한다.

- 농축어업이나 선원 문서가 SW 표준계약서와 비교된다.
- 인접 도메인 문서가 특정 계약유형으로 자동 확정된다.
- 파일명에 의존한 유형 판별이 실제 본문과 어긋난다.
- 외부 클라이언트가 “검토 결과가 나왔으므로 지원되는 문서”라고 해석한다.

기존 `review_contract`의 입력·출력 계약을 바꾸는 대신, 검토 전에 범위를 확인하는 별도 도구 `assess_contract_scope`를 추가했다.

```text
assess_contract_scope
        ↓
IN_SCOPE
CONTRACT_TYPE_UNCERTAIN
OUT_OF_SCOPE
EMPTY_DOCUMENT
        ↓
외부 클라이언트가 사용자에게 확인
        ↓
review_contract
```

이 판별은 파일명이나 LLM(Large Language Model, 대규모 언어 모델)에 의존하지 않고 파싱된 조항의 결정론적 앵커(anchor, 분류를 유도하는 표식)와 계약유형 표식을 사용한다.

### 상태가 곧 사용자 행동을 결정한다

```json
{
  "status": "CONTRACT_TYPE_UNCERTAIN",
  "suggested_contract_type": "SI_SUBCONTRACT",
  "candidates": [
    {"contract_type": "SI_SUBCONTRACT", "score": 4},
    {"contract_type": "SM_SUBCONTRACT", "score": 3}
  ],
  "matched_clause_count": 8,
  "exclusion_markers": ["근로시간"],
  "message": "계약유형 근거가 충분하지 않습니다. 사용자가 유형을 선택한 뒤 검토를 계속할 수 있습니다."
}
```

상태별 책임은 다르다.

| 상태 | MCP가 제공하는 것 | 외부 클라이언트의 다음 행동 |
| --- | --- | --- |
| `IN_SCOPE` | 제안 계약유형과 근거 점수 | 유형을 보여주고 사용자 확인 요청 |
| `CONTRACT_TYPE_UNCERTAIN` | 가능한 후보와 불확실성 | 자동 확정하지 않고 사용자 선택 요청 |
| `OUT_OF_SCOPE` | 지원 코퍼스와 공통 근거가 부족하다는 상태 | 지원 범위 밖임을 경고하고 진행 여부 확인 |
| `EMPTY_DOCUMENT` | 파싱할 조항이 없다는 상태 | 파일 형식·스캔 상태 확인 |

`CONTRACT_TYPE_UNCERTAIN`은 검토 차단 상태가 아니다. 사용자가 계약유형을 명시하면 기존 검토 흐름을 계속 호출할 수 있다. 반대로 `OUT_OF_SCOPE`에서는 MCP가 자동으로 `review_contract`를 이어서 호출하지 않는다. 현재 정책은 사용자가 지원 범위 밖이라는 경고를 확인하고 계약유형을 명시적으로 재확인한 경우에만 표준 대비 비교를 허용하는 것이다. 이때 결과는 “지원 문서에 대한 검토”가 아니라, 사용자가 선택한 표준 코퍼스와의 제한적 비교로 표시해야 한다. 일반 문서 분석으로 자동 라우팅하는 기능은 아직 없다.

### 외부 도구의 출력도 시스템 자원이었다

세 번째 문제는 외부 법령 MCP(Model Context Protocol 서버)의 출력량이었다. 법령 조회가 성공하면 원문을 많이 붙일수록 좋은 결과라고 생각하기 쉽다. 하지만 포괄적인 검색과 조문 파싱 오류가 겹치자 정상 문서 한 건에 법령 근거가 폭증했다.

```text
수정 전
총 법령 근거: 82,080개
결과당 평균 법령 수: 448.5개
평균 처리시간: 66.1초
```

이것은 법령 검색이 풍부해진 결과가 아니라, 외부 도구의 출력 경계를 MCP가 통제하지 못한 결과였다.

### 폭증의 원인

- 조문번호가 없는 `민법 도급` 같은 포괄 질의를 외부로 보냈다.
- 본문 안의 `제N조` 인용을 실제 조문 헤더로 오인했다.
- 법령 전문을 결과에 그대로 붙였다.
- 동일한 `(법령명, 조번호)`를 안정적으로 중복 제거하지 않았다.

그래서 정적 grounding(미리 정한 카테고리별 근거 조회)은 조문번호가 명확한 경우만 허용하도록 좁혔다.

```python
if query_str is None or not has_article_number(query_str):
    return []

laws = parse_article_headers(raw_text)
laws = deduplicate(laws, key=("law_name", "article_number"))
return laws[:3]
```

조문 헤더도 줄 시작에 있는 경우만 실제 조문으로 본다. 본문 문장 중간의 법령 인용을 새 결과로 만들지 않는다. 결과가 없으면 외부 도구가 실패한 것처럼 숨기지 않고 `NO_RESULT` 경로로 전달한다.

```text
변경 전
카테고리 → 포괄 검색 → 법령 전문 → 무제한 결과

변경 후
카테고리
  → 조문번호가 있는 매핑만 허용
  → 줄 시작 조문 헤더만 파싱
  → 중복 제거
  → 최대 3건
  → 결과 없음은 NO_RESULT
```

결과는 다음처럼 달라졌다.

| 지표 | 변경 전 | 변경 후 |
| --- | ---: | ---: |
| 총 법령 근거 | 82,080 | 101 |
| 결과당 평균 법령 수 | 448.5 | 1.0 |
| 평균 처리시간 | 66.1초 | 19.1초 |

이 수치는 법률 검색의 정확도를 증명하지 않는다. 외부 도구의 출력을 1차 MCP가 감당 가능한 범위로 제한했다는 운영 지표다.

반대로 상한을 둔 뒤 필요한 근거를 잃지 않았는지는 별도의 질문이다. 현재는 정적 매핑의 기대 조문이 결과에 포함되는지와 결과 수·중복을 확인했지만, 모든 카테고리에서 “최대 3건이면 충분한가”를 검증하는 독립적인 grounding recall(필요한 근거를 놓치지 않는 비율) 평가는 아직 없다. 출력 폭증은 해결했지만, 상한 3건의 충분성은 후속 평가 대상으로 남겨 두었다.

## 3. 수명 경계: 세션·캐시·수명주기를 앱의 경계로 올렸다

외부 법령 도구를 호출할 때마다 stdio(표준 입출력) 자식 프로세스를 만들고 종료하던 초기 구조는 기능적으로는 동작했지만 운영하기 어려웠다.

```text
요청 1 → 프로세스 생성 → 도구 호출 → 종료
요청 2 → 프로세스 생성 → 도구 호출 → 종료
요청 3 → 프로세스 생성 → 도구 호출 → 종료
```

이 구조에서는 프로세스 생성 비용이 반복되고, 외부 도구가 가진 세션 캐시도 재사용할 수 없다. 종료 시점도 각각의 호출 코드에 흩어진다.

현재는 앱 수명주기(lifespan, 애플리케이션 시작과 종료를 관리하는 구간)에 세션을 묶는다.

```python
@asynccontextmanager
async def _lifespan(_app: FastMCP):
    koreanLaw.start()
    try:
        yield {}
    finally:
        koreanLaw.close()
```

```text
앱 시작
  └─ KoreanLawMCPClient 세션 시작

요청 1 ─┐
요청 2 ─┼─ 같은 세션·클라이언트 공유
요청 3 ─┘

앱 종료
  └─ 세션 종료·자식 프로세스 정리
```

`create_app()`은 MCP composition root(애플리케이션 구성 요소를 조립하는 시작점)가 되어 WorkShield 도구와 법률 도구 프록시를 등록한다.

```python
def create_app() -> FastMCP:
    mcp = FastMCP("WorkShield", lifespan=_lifespan)
    WorkShieldTools(mcp)
    KoreanLawWrapper(mcp, client=koreanLaw)
    return mcp
```

### 캐시는 성능보다 수명 정책이었다

모든 결과를 오래 저장하지 않았다. 재사용해도 되는 데이터와 요청이 끝나면 폐기해야 하는 데이터를 나눴다.

| 캐시 | 대상 | 수명 |
| --- | --- | --- |
| 요청 캐시 | 한 번의 `review_contract` 안에서 반복되는 카테고리 grounding | 요청 종료 시 폐기 |
| 프로세스 캐시 | 정적 `(Category, ContractType)` grounding | TTL 24시간, 최대 64건 |

프로세스 캐시에는 성공 결과와 `NO_RESULT`만 저장한다. `NO_RESULT`는 영구적인 실패가 아니라 현재 질의에서 근거를 찾지 못했다는 음성 결과(negative result)이므로 24시간 동안만 캐시한다. 외부 데이터가 갱신되거나 매핑이 바뀌면 TTL(Time To Live, 유효 시간) 만료 뒤 다시 조회할 수 있다. 통신 오류는 결과 부재와 다른 상태이므로 캐시하지 않아 다음 요청이 즉시 재시도할 수 있게 한다. 사용자 조항 원문을 자유 질의한 결과도 정적 캐시에 넣지 않는다.

동시에 같은 키를 요청하는 경우에는 single-flight(동일 키의 중복 작업을 하나로 합치는 방식)로 한 요청만 외부 도구를 호출하게 한다. 현재 구현은 단일 프로세스 안에서만 중복을 제거한다. 여러 워커(worker, 병렬 요청 처리 프로세스)가 각각 실행되는 배포라면 공유 락이나 외부 캐시가 추가로 필요하다. 캐시는 단순한 딕셔너리가 아니라 외부 데이터의 수명과 재사용 범위를 정하는 정책이 되었다.

## 서버와 외부 클라이언트의 책임을 나눴다

현재 앱의 구성은 다음과 같다.

```text
외부 MCP 클라이언트
        ↓
src/app.py
  FastMCP 앱·수명주기·공용 세션
        ↓
src/server/
  ├─ WorkShieldTools      계약 검토·표준 리소스
  └─ KoreanLawWrapper     외부 법령 MCP 프록시
        ↓
core / pipe / adapter
판정 규칙 / 검토 조립 / 외부 I/O
```

1차 MCP가 책임지는 범위는 다음과 같다.

- 입력 형식과 필수값 검증
- `NO_MATCH`, `NO_RESULT`, `EMPTY_DOCUMENT` 같은 명시 상태 반환
- 지원 범위의 결정론적 사전 판별
- 검색·매칭·정적 grounding
- 외부 도구의 출력 상한과 오류 경계
- 세션·캐시·수명주기 관리

외부 클라이언트가 책임지는 범위는 다르다.

- `CONTRACT_TYPE_UNCERTAIN`에서 사용자에게 계약유형을 선택받기
- `OUT_OF_SCOPE`를 지원 범위 밖 경고로 표시하기
- `NONE`, `EXTRA`, `NO_MATCH`, `MISSING`을 법률 결론으로 번역하지 않기
- `NO_RESULT`나 `EMPTY_DOCUMENT` 뒤의 다음 행동 안내
- 2차 LLM을 사용할 경우 서버 밖에서 비교 근거를 해석하기

1차 MCP는 “이 계약이 위법하다”거나 “이 조항이 불리하다”고 말하지 않는다. 외부 클라이언트도 빈 결과를 근거로 그런 결론을 추론해서는 안 된다. 서버는 후보와 상태를 제공하고, 클라이언트는 그 상태를 사용자 경험과 후속 선택으로 연결한다.

## 평가는 모델 점수가 아니라 운영 질문으로 나눴다

운영 가능한 MCP를 평가할 때 모델 점수 하나만 보지 않았다. 외부 클라이언트가 실제로 필요한 질문으로 나눴다.

| 질문 | 확인 방법 |
| --- | --- |
| 실패를 클라이언트가 구분할 수 있는가? | 상태별 응답 스키마와 메시지 확인 |
| 지원 범위 밖 문서를 식별하는가? | scope golden(정답 범위 모음) 상태 분류 |
| 외부 출력이 제한되는가? | 결과 수·중복·최대 상한 확인 |
| 세션을 재사용하는가? | 프로세스 생성·종료 횟수 확인 |
| 캐시가 안전하게 재사용되는가? | TTL·LRU·오류 미캐시 테스트 |
| 근거 폭증이 줄었는가? | 결과당 법령 수와 E2E 처리시간 확인 |
| 법률 결론으로 오해되지 않는가? | 상태 설명·메시지·클라이언트 표시 규칙 검토 |

82,080개에서 101개로 줄었다는 수치도 모델의 정확도 점수가 아니다. 출력 상한과 파싱 규칙이 운영 문제를 줄였는지 확인하는 질문의 답이다.

## 시스템의 책임 경계

1차 MCP가 책임지는 것은 다음까지다.

- 외부 입력을 파싱하고 명시 상태로 반환한다.
- 지원 코퍼스와 비교할 수 있는 범위를 결정론적으로 안내한다.
- 외부 도구 호출을 조문·개수·수명 측면에서 제한한다.
- 원문과 근거를 보존하면서 클라이언트가 사용할 후보를 제공한다.

1차 MCP가 책임지지 않는 것은 다음이다.

- 조항의 의미가 법적으로 같은지 확정하는 일
- 기간·대가·권리 주체의 유불리를 판정하는 일
- `grounding=[]`를 안전성이나 적법성으로 해석하는 일
- 지원 범위 밖 문서의 법적 성격을 대신 결정하는 일

경계를 코드와 응답 포맷에 함께 넣었기 때문에, 1차 MCP는 결정론적 도구로 남고 필요한 의미 해석은 서버 밖의 별도 계층으로 분리할 수 있다.

## 내가 배운 설계 원칙

- MCP의 출력은 데이터 묶음이 아니라 외부 클라이언트와의 상태 계약이다.
- 빈 목록보다 명시 상태가 안전하다.
- 지원 범위 판별은 법적 결론이 아니라 다음 행동을 안내하는 경계다.
- 외부 도구의 출력량도 MCP가 책임지는 운영 자원이다.
- 조문 상한과 중복 제거는 검색 품질과 별개인 신뢰성 장치다.
- 세션·캐시·수명주기는 성능 최적화가 아니라 데이터와 프로세스의 수명 정책이다.
- 정적 grounding과 동적 법률 리서치는 같은 캐시·같은 책임으로 다루지 않는다.
- 1차 MCP는 후보와 근거를 제공하고, 외부 클라이언트는 사용자 확인과 후속 해석을 담당한다.
- 도구가 무엇을 할 수 있는지보다, 실패했을 때 외부 클라이언트가 무엇을 해야 하는지 분명히 알려주는 것이 더 중요하다.

MCP는 도구를 외부에 공개하는 방식이 아니라, 시스템의 책임 경계를 외부 호출자가 이해할 수 있는 계약으로 만드는 방식에 가깝다. 운영 가능한 도구는 모든 질문에 답하는 도구가 아니라, 답할 수 없는 상황을 상태와 다음 행동으로 정확히 전달하는 도구다.
