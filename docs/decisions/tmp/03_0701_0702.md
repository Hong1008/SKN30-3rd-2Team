## 매칭 파이프라인·결과 의미

### D35. 매칭 스케일 결함을 리랭커 필수 경로로 수정
- **결정:** `search → rerank → sigmoid 정규화 → select_best_match`를 매칭의 필수 흐름으로 확정하고, RRF `fusion_score`를 threshold와 직접 비교하지 않는다.
- **이유:** RRF 점수의 최대값은 약 0.033인데 0.5 threshold와 비교되어 전 조항이 NO_MATCH, 전 표준조항이 MISSING이 되는 실결함이 발생했다. 테스트 fake가 높은 rerank 점수를 하드코딩해 결함을 가렸다.
- **영향:** reranker를 `review_contract`의 필수 주입으로 확장하고, raw logit은 sigmoid 후 비교한다.
- **상태:** 채택; 동결에 가까운 시그니처 확장은 승인 후 적용 (07-01)

### D36. NO_MATCH와 EXTRA를 검색 결과 유무로 구분
- **결정:** 검색 후보가 없을 때만 NO_MATCH, 후보가 있으나 임계값 미달이면 EXTRA 판정 경로로 보낸다.
- **이유:** 후보 존재 여부와 매칭 성립 여부를 혼동해 EXTRA가 도달 불가능했던 결함을 복원한다.
- **상태:** 채택 (07-01)

### D37. grounding과 고도화 배선의 범위
- **결정:** grounding은 CHANGED/MISSING에만 붙이고 GENERAL에는 적용하지 않는다. 서브청크 roll-up과 독소 역방향 검색은 배선하되 graph는 옵션 seam으로 둔다.
- **이유:** NONE까지 grounding하는 낭비와 스펙 불일치를 없애며, 독소는 표준 매칭과 무관하게 EXTRA에서 발견될 수 있다.
- **후속:** Graph 구현체 및 coverage의 완전 배선.
- **상태:** 채택 (07-01)

### D38. 서브청크에는 contract_type만 저장
- **결정:** 서브청크 메타데이터에 contract_type은 추가하고 category/source 복사는 보류한다.
- **이유:** contract_type은 검색 정확도를 위한 안전한 파티션 키다. 반면 부모 category 복사는 자식 항의 의미 차이와 소비처 부재로 드리프트 위험이 있다.
- **상태:** 일부 채택 / category 보류 (07-01)

### D39. 배치 검색 우선, 배치 리랭킹은 준비만
- **결정:** `Retriever.search_many`로 컬렉션당 임베딩·검색을 한 번으로 묶는다. `rerank_many`는 구현하지만 프로파일링상 병목이 확인되기 전까지 파이프에 적용하지 않는다.
- **이유:** 검색은 조항 수만큼 네트워크/임베딩 왕복이 발생하는 주 병목이고, 리랭킹 후보 수는 작다. 배치 리랭킹은 점수 불변의 드롭인 옵션으로 준비한다.
- **상태:** 채택 / 적용 보류 (07-01)

### D40. coverage는 별도 작업으로 격리
- **결정:** 항 삭제 탐지는 사용자 항 분할·표준 서브청크 로드·M×N 점수 행렬을 포함하므로 H 작업 카드로 분리한다. 결과는 새 필드 없이 NONE→CHANGED로 상향한다.
- **이유:** 단순 배선으로 표기하면 실제 미구현 범위를 숨기게 된다. 유사도는 새 embedder 주입 대신 주입된 reranker를 재사용하는 방향이다.
- **상태:** 설계 확정, 구현 대기 (07-01)

## 평가 설계·데이터 정합성

### D41. 합성 조항 평가와 실계약 E2E 평가를 분리
- **결정:** 트랙 A는 합성 조항 기반 검색·분류·독소·ablation, 트랙 B는 실계약 문서 기반 E2E·MISSING으로 나눈다. 이번 범위는 트랙 A 문서·설계까지다.
- **이유:** MISSING은 계약 단위 속성이어서 조항 단위 합성셋에서 측정할 수 없지만, 검색 ablation은 함정이 밀집한 합성셋에 적합하다.
- **상태:** 채택; 트랙 B 보류 (07-01)

### D42. 골든 라벨은 동결 enum에 맞춰 정정
- **결정:** enum 밖 독소 라벨 두 개를 기존 `UNILATERAL_CANCELLATION`으로 재매핑하고 enum 확장은 하지 않는다. trap에는 `contradiction`을 정식 포함한다.
- **이유:** 탐지 원천이 없는 라벨은 평가를 불가능하게 하며, 사례의 의미는 기존 코퍼스와 맞는다. 실제 사용 중인 함정 유형을 스키마에 반영한다.
- **상태:** 채택 (07-01)

### D43. eval driver는 리랭킹을 포함한 4변형으로 재작성
- **결정:** bm25/dense/hybrid/hybrid_rerank를 비교하고, 조항 단위 review 결과로 이탈·독소 P/R을 집계한다.
- **이유:** 기존 driver는 잘못된 import, 단일 컬렉션, rerank 누락으로 실제 매칭 불변식에 맞지 않았다.
- **상태:** 설계 확정, 구현 후속 (07-01)

## 운영 모델·배포

### D44. RunPod Hub 워커 대신 자체 서버리스 워커
- **결정:** rerank 직렬화 버그가 있는 Hub 워커, Pod 전환, vLLM 대안을 기각하고 로컬 임베딩·리랭커 구현을 재사용하는 자체 RunPod 워커를 구축한다.
- **이유:** rerank는 매칭의 필수 경로다. 자체 워커는 로컬/운영 코드 불일치를 줄이고, Pod는 재고·지연의 근본 문제를 해결하지 못한다.
- **상태:** 채택 및 배포 (07-02)

### D45. 모델 가중치는 이미지에 굽고 ghcr.io로 배포
- **결정:** 불안정한 `--model-reference`/네트워크 볼륨 대신 Docker 빌드 시 모델을 이미지에 내장하고 GHCR public 이미지를 사용한다.
- **이유:** 런타임 볼륨 경로와 재다운로드에 의존하지 않고 콜드스타트 위험을 낮춘다.
- **상태:** 채택 (07-02)

### D46. 서버리스 운영은 workers-min 토글, 클라이언트는 180초
- **결정:** 평상시 workers-min 0, 작업 전 1로 전환하고 API timeout을 60초에서 180초로 올린다.
- **이유:** 비용 없이 유지하면서 콜드스타트를 운영적으로 피하고, 워커 기동 전에 클라이언트가 실패하는 문제를 막는다.
- **상태:** 채택; 토글 자동화는 후속 (07-02)

### D47. API 리랭킹을 다중 질의 1회 호출로 변경
- **결정:** 자체 워커에 `queries`/`docs_per_query` 배치 스키마를 추가하고 `ApiReranker.rerank_many`가 질의별 N회가 아닌 한 번의 호출을 사용하게 한다.
- **이유:** 워커를 소유하므로 기존 per-query API 제약이 사라졌고, 정렬·top-k 책임은 호출측에 유지한 채 네트워크 왕복을 줄일 수 있다.
- **상태:** 채택 및 실제 엔드포인트 검증 (07-02)
