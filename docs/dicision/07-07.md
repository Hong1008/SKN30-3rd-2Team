## 2. `core/deviation.py` 실측 버그 2건 수정

v3 케이스를 실제 함수(`calculate_text_similarity`·`detect_critical_changes`)에 태워 의도한 라벨과
실제 동작이 맞는지 검증하는 과정에서 발견·수정.

- **`_NEGATION_RE` — "아니 된다" 미탐지**: "된"(되+ㄴ)과 "되"는 다른 유니코드 문자라 `아니되`
  패턴이 "~하여서는 아니 된다"(계약서에서 가장 흔한 금지 표현)를 못 잡고 있었음. `아니된` 패턴
  추가로 수정.
- **`_PARTY_RE` — 숫자 뒤 조사 "을" 오탐**: "10000분의 1을 곱하여"처럼 숫자 뒤에 오는 목적격
  조사 "을"이 당사자 "을(乙)"로 오탐됨(기존 가드는 앞이 한글일 때만 조사로 인정, 숫자는 미고려).
  `(?<![가-힣])` → `(?<![가-힣0-9])`로 수정.
- **테스트 추가**: `tests/core/test_deviation.py`에 두 버그 재현 테스트 2건 추가. 전체
  `uv run pytest tests/core/ tests/pipe/ tests/eval/` 106 passed 확인.
- **세 번째 갭 발견 시 패치 중단**: "~할 수 없다"(모달 불가능, "없다/없이") 전혀 미커버 확인 —
  이 시점에서 사용자가 "정규식으로 이런 걸 판별한다는 명제 자체"를 재검토하자고 제안, 개별 패치를
  멈추고 §3 아키텍처 재설계로 전환.

## 3. 명제 재검토 — 규칙기반 판정 계층은 왜 구조적으로 실패하는가

정규식 3건(아니된다·숫자+을·없다) 전부가 "개별 버그"가 아니라 **두 가지 서로 다른 구조적 실패
유형**의 사례임을 확인:

1. **열거 불완전성**: 한국어 부정어 표현(아니하다/아니되다/않다/못하다/없다/말다/한자 부정접두사)과
   숫자 표기(아라비아 vs 한글 숫자, "오 년")를 정규식으로 완전히 열거하는 건 원리상 끝나지 않는다.
2. **마커 없는 의미 반전**: `v3-sw-11`(공동소유→도급인 단독 영구귀속)처럼 부정어·숫자·당사자
   집합 자체가 안 바뀌는 순수 의미 반전은 어떤 정규식 정교화로도 원천 탐지 불가 — "바뀐 사실"이
   표면 토큰이 아니라 개념 간 관계에 있기 때문. 당사자 검출도 집합 비교로는 **주어-목적어 역할
   반전**(누가 의무자인가)을 표현 못 한다는 한계를 `v3-sw-09` 설계 시도 중 확인.

리랭커 기반 커버리지 체크(관련성 점수)도 "의미 동치성"과는 다른 신호라는 07-03 §2 진단과 같은
계열의 문제로 재확인됨.

## 4. 1차/2차 판정 경계 확정 — 규칙계층 완전 제거

§3 분석을 근거로 아래 4개 사안을 사용자 사인오프로 확정(AGENTS.md §2 동결 계약 변경):

- **규칙기반 치명변경 탐지(`detect_critical_changes`, `_NEGATION_RE`/`_NUMBER_RE`/`_PARTY_RE`,
  `CRITICAL_NEGATION`/`CRITICAL_NUMBER`/`CRITICAL_PARTY`) 완전 제거.** 축소·보강이 아니라 삭제 —
  2차 LLM이 정렬된 원문 쌍을 직접 읽고 판단하며, 신뢰 못 할 규칙 힌트로 프롬프트에 노이즈를 얹지
  않는다.
- **difflib 폴백(커버리지 불가 시 글자일치율로 NONE/CHANGED 가르던 로직)도 동일 원리로 제거.**
  커버리지 체크 불가 시 **무조건 CHANGED**(안전측, 2차로 위임)로 단순화.
- **2차(LLM) 호출 위치: 서버 밖(클라이언트/데모 레이어).** MCP 서버 안에 별도 2차 도구를 새로
  만들지 않는다 — `review_contract`는 계속 LLM 무호출 유지, AGENTS.md 절대규칙 #1을 문자 그대로
  준수.
- **`DeviationResult`에 새 필드 추가 없음.** 2차가 필요한 정보(`user_clause`·`matched_standard.text`·
  `uncovered_sub_chunk_ids`·`confidence`)는 이미 존재 — 최소주의 채택. `Deviation` enum 값은
  무변경, **`NONE`의 의미만 "확정"에서 "1차 잠정치(2차 확인 전)"로 재정의**.

**새 판정표** (매칭 성공 조항 기준):

| 커버리지 결과 | 이전 | 이후 |
| --- | --- | --- |
| 미달 | CHANGED(확정) | CHANGED(확정, 불변) |
| 충족 + 치명변경 규칙 발견 | CHANGED(확정) | *(규칙 삭제로 경로 소멸)* |
| 충족 + 치명변경 없음 | NONE(확정) | **NONE(잠정 — 2차 확인 전)** |
| 체크 불가 | difflib 폴백 | **CHANGED(확정, 안전측)** |

## 6. Grounder 계약유형별 법령 그라운딩 결함 발견 → task O 생성

사용자가 "data/에 SI/SM 하도급 계약서가 섞이면서 그라운딩 쿼리도 유형별로 달라야 하지 않냐"고
제기, 코드로 실측 확인:

- `Grounder.get_grounding(self, category: Category)` — Protocol 자체에 `contract_type` 인자 없음.
- `CATEGORY_QUERIES`(`korean_law_grounder.py`)는 유형 무관 고정 딕셔너리, 매핑 10건 전부 민법·
  저작권법·부정경쟁방지법·근로기준법 — 코드 전체에 "하도급법" 언급 0건.
- Category 15종 중 4종(`CONTRACT_PERIOD`·`DELIVERY_INSPECTION`·`WARRANTY`·`SUBCONTRACTING`)은
  아예 미매핑 → `"민법 도급"` 폴백. 실측: SI/SM 표준 코퍼스에서 **`SUBCONTRACTING`(재하도급)이
  최빈값(12건)인데 완전 미매핑** — 가장자리 케이스가 아니라 다수 조항에 영향.
- 호출부 3곳(`review_pipe._grounding_for`, `server.py`의 `classify_clause`·`review_contract`·단독
  `get_grounding` 도구) 전부 `contract_type`이 스코프에 있는데도 안 넘김.
- 이 결함은 `run_eval.py`가 `NullGrounder`(no-op)를 쓰므로 **어떤 result.md에도 드러난 적 없음**
  — 실제 MCP 서버/데모 경로에서만 나타나는 숨은 결함.
- [Z_Lever.md](../tasks/Z_Lever.md)에 `grounding_ref`(하도급법 제13조) 아이디어가 있었으나, 항 단위
  "법률 레버" 판정이라는 더 큰 설계(§3의 규칙기반 한계와 같은 축 + NLI)에 묶여 있었고 그 설계
  전체가 이미 "오버엔지니어링"으로 기각돼, 이 좁은 문제는 단독으로 다뤄진 적 없었음.

**확정**: `get_grounding`에 `contract_type: Optional[ContractType] = None` 가법 추가(하위호환),
SW_FREELANCE 쿼리는 불변, SI/SM 전용 override 테이블 신설(우선 조회 → 없으면 기존 공용값 폴백).
사용자 제공 초안 3건(`PAYMENT`→하도급법 제13조, `DELIVERY_INSPECTION`→제9조, `CONFIDENTIALITY`→
제12조의3) 반영. 나머지 조문(특히 최빈값인 `SUBCONTRACTING`)은 법률 검토가 필요해 이 세션에서
추측으로 채우지 않고 [O_grounding_contract_type.md](../tasks/O_grounding_contract_type.md)의
오픈 항목으로 남김.

- **J~N 카드 구현** — 규칙계층 삭제(J)가 최우선(K·M의 전제), L(골든 SI/SM 확장)·N(독소 코퍼스)은 독립 병행 가능.
- **O 카드 구현** — Protocol 확장 + 초안 3건 반영, 나머지 카테고리(`SUBCONTRACTING` 최우선) 조문은 법률 검토 필요.
- **M 카드의 질문 4건** — 해결책 없이 남김: 1차 단독 채점 지표 재정의, 2차 LLM 출력 검증이 규칙 #5(LLM-judge 금지)와 충돌 안 하는지, 트랙 B 축퇴 경보 재검토, `degeneracy_alerts` 문구 재검토.
- **이번 세션 전체를 별도 결정 로그로 남길지 여부** — 남김(본 문서).
