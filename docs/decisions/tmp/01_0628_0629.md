# 결정 카드 — 2026-06-28~29

> 목적: 블로그 회고를 위한 사실 중심의 임시 분류이며, 사후 평가나 인사이트는 포함하지 않는다.

## 제품 원칙·협업 방식

### D01. 단계적 병렬 협업과 모듈 소유권
- **결정:** 리드는 총괄·MCP 서버·데모를 맡고, 팀원은 normalize/index/review/eval을 모듈 단위로 분담한다. 단계마다 구현·리뷰 후 다음 단계로 진행한다.
- **이유:** 동결 계약과 fake 기반 테스트를 공통 경계로 삼으면 비전공자를 포함한 병렬 작업과 독립 검증이 가능하다.
- **영향:** A·C·D는 즉시 병렬 착수하고, 실제 데이터 통합은 A→B→C 순서를 따른다.
- **상태:** 채택 (06-28)

### D02. TDD를 팀 협업 규격으로 사용
- **결정:** 테스트를 모듈 규격으로 먼저 고정하고, 미구현 영역에는 skip 스펙 테스트와 스켈레톤을 제공한다.
- **이유:** 구현자가 통과 기준과 시그니처를 명확히 알고 작업할 수 있다.
- **영향:** core 네 모듈의 기존 동작은 회귀 기준으로 유지한다.
- **상태:** 채택 (06-28)

## 아키텍처·계약 경계

### D03. 헥사고날 구조와 동결 계약
- **결정:** `contracts`를 동결 계약으로 두고, 순수 `core`·외부 I/O `adapter`·조립 `pipe`를 분리한다. core는 adapter를 직접 import하지 않고 port를 주입받는다.
- **이유:** 계약에만 의존하는 병렬 개발과 fake 주입 테스트를 가능하게 하며, core의 순수성을 보장한다.
- **상태:** 채택 (06-28)

### D04. 계약 변경은 가법 확장과 사람 승인으로 관리
- **결정:** `Deviation` enum 신규값은 기각하고, element 수준 결과는 `granularity`, `change_kind`, `ElementResult` 같은 가법 확장으로 한 묶음 승인한다.
- **이유:** enum 변경은 모든 소비자 분기를 깨뜨릴 수 있고, 빈 응답·분기 누락 위험을 높인다. 가법 확장도 동결 변경이므로 사전 사람 승인이 필요하다.
- **기각/보류:** `ELEMENT_MISSING` 등 enum 신설 기각. G 구현은 승인 전 코드 수정 보류.
- **상태:** 채택 / 승인 대기 (06-29)

### D05. 포트 구현체의 배치 원칙
- **결정:** 단일 포트 구현은 `adapter/`, 여러 어댑터·변환 조합 구현은 `contracts/implement/`에 둔다. 동결 영역은 순수하게 유지한다.
- **이유:** 포트 시그니처는 안정적으로 유지하면서 구현 조합의 변경을 허용한다.
- **사례:** VectorManager/Bgem3Embedder는 adapter, KordocParser/KoreanLawGrounder는 implement.
- **상태:** 채택 (06-28)

## 데이터·도메인 모델·분류

### D06. 정답 데이터는 Git, 인덱스는 재생성
- **결정:** 정규화 JSON·migration SQL·변환 Markdown만 Git으로 관리하고 SQLite·Chroma는 `just build-db`로 재생성한다.
- **이유:** 바이너리 머지 충돌·용량을 피하고, 데이터 라벨 변경을 PR diff로 검토하기 위해서다.
- **기각:** SQLite/Chroma 바이너리 커밋.
- **상태:** 채택 (06-28)

### D07. 스키마·출력의 초기 불변 조건
- **결정:** 표준조항·조항관계·독소패턴 3개 테이블을 초기부터 동결하고, `NO_MATCH`를 포함한 명시 응답을 유지한다.
- **이유:** 고도화 도입 때의 마이그레이션을 줄이고, 매칭 실패를 빈 응답으로 숨기지 않기 위해서다.
- **상태:** 채택 (06-28)

### D08. Category를 앵커 임베딩 분류로 전환
- **결정:** 키워드 하드코딩을 폐기하고 `Category.anchors`와 임베딩 유사도 argmax를 사용한다. 임계값 미달은 fallback하지 않고 명시적으로 실패한다.
- **이유:** “보수”가 “하자보수”를 PAYMENT로 잘못 분류하는 사례처럼, pre-filter 오분류는 검색 자체를 막는다. 이 지점은 정확도보다 재현율이 중요하다.
- **영향:** Category를 보강·정리하고 `build_category_vectors()`를 build-db 단계에 연결한다.
- **상태:** 채택, 구현 후속 (06-29)

### D09. 하도급 표준계약서의 MVP 지원 유지
- **결정:** SI/SM 등 하도급 표준계약서를 MVP 범위에서 제외하지 않고 적재·지원한다.
- **이유:** 대기업·중견기업과 도급하는 프리랜서의 보호라는 제품 미션에 필요한 계약 유형이기 때문이다.
- **영향:** 하도급 5종 서식 분할과 관련 ContractType 확장을 진행한다.
- **상태:** 채택 (06-29)

## 검색·매칭·판정

### D10. 청킹과 도메인 해석의 분리
- **결정:** Markdown 분할은 adapter, `제N조` 해석은 normalize 파이프에 둔다. 숫자 접두 CLI 파일은 래퍼만 맡긴다.
- **이유:** 도메인 무관 분할 기능을 표준 코퍼스와 사용자 업로드가 공유하고, import 가능한 로직 모듈을 유지하기 위해서다.
- **상태:** 채택 (06-28)

### D11. element 정렬은 임계값 기반 다대다 매칭
- **결정:** 헝가리안 알고리즘 기반 “다대다 정렬”은 폐기하고, 임계값 이상 연결을 허용하는 다대다 매칭을 채택한다.
- **이유:** 헝가리안은 1:1이므로 흔한 1:N·N:1 문장 구조에서 잔여를 MISSING/EXTRA로 오판할 수 있다. 작은 코퍼스에서는 최적성보다 재현성과 단순성을 우선한다.
- **보류:** 1:1 할당 뒤 잔여를 흡수하는 복합 패스.
- **상태:** 채택 (06-29)

### D12. NLI의 역할을 보조 신호로 한정
- **결정:** KLUE-NLI는 보조 신호로만 사용하고 최종 심각도·불리 판정은 당사자·양태·수치 기반 룰로 낸다.
- **이유:** contradiction을 불리함으로 바로 매핑하는 것은 제품의 1차 판정 경계를 넘을 수 있다.
- **후속:** ablation으로 기여가 확인될 때만 활성화 여부를 판단한다.
- **상태:** 채택 / 조건부 보류 (06-29)

### D13. 수혜자 개념과 심각도 계산의 명시화
- **결정:** 모호한 `protects` 대신 `beneficiary`를 사용하고, 심각도는 `(required, beneficiary, deviation/change_kind)` 결정표로 산출한다.
- **이유:** 수혜자와 보호 의도를 분리해야 “불리한 변경”의 정의가 일관되고, 결정론·재현성 요구를 충족한다.
- **후속:** 등급 경계와 가중치는 골든셋으로 보정한다.
- **상태:** 채택 (06-29)

## 평가·재현성

### D14. 1차 평가는 결정론적 IR 지표로 한정
- **결정:** 1차에서 RAGAS/LLM-judge를 사용하지 않고 Recall@k·MRR·precision/recall과 BM25/dense/hybrid/reranker ablation을 사용한다.
- **이유:** 1차는 생성물이 아닌 검색·분류 과제이며, LLM-judge는 결정론 평가 원칙에 맞지 않는다.
- **상태:** 채택 (06-28)

### D15. 런타임 LLM 배제와 재현 가능한 분해
- **결정:** LLM은 오프라인 초안 후 사람이 검수해 정답 데이터로 동결하는 경우에만 허용하며, 런타임에서는 호출하지 않는다. kss는 규칙 기반이라고 표현하지 않고 버전 고정으로 재현성을 확보한다.
- **이유:** 런타임 1차 판정의 결정론성을 지키고, 도구의 실제 특성을 정확히 기술하기 위해서다.
- **상태:** 채택 (06-29)

### D16. 대표 골든셋은 보유 코퍼스 범위에 맞춤
- **결정:** 대표 element 파괴 케이스를 코퍼스가 있는 SW_FREELANCE로 옮기고, SW_SUBCONTRACT 평가는 코퍼스 확보 전까지 보류한다.
- **이유:** 표준조항이 없는 계약 유형을 대표 평가로 삼으면 평가 설계가 성립하지 않는다.
- **상태:** 채택 / 일부 보류 (06-29)

## 인프라·운영 보조 결정

### D17. 법령 연동과 임베딩 계산의 명시적 표준화
- **결정:** 법제처 키 환경변수는 `LAW_OC`로 통일한다. 임베딩 유사도는 L2 정규화 벡터의 내적으로 계산한다. Grounder 법령 매핑은 실측으로 정정한다.
- **이유:** 연동 설정의 혼선을 줄이고, 외부 라이브러리 없이 일관된 코사인 유사도를 사용하며, 법령 참조 정합성을 확보하기 위해서다.
- **상태:** 채택 (06-28)
