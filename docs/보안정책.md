# WorkShield 보안·데이터 처리 정책

> 상태: 초안 · 적용 대상: MCP 서버, 데모, RunPod 임베딩·리랭커 워커 및 향후 2차 LLM 워커
>
> 이 문서는 기술·운영 기준이다. 개인정보보호법, 고객 계약, 국외 이전 등 법적 적용 여부는
> 서비스 운영 주체가 별도로 검토·승인한다.

## 1. 원칙

- 계약서와 그 조항은 기본적으로 **민감한 고객 데이터**로 취급한다.
- 1차 검토는 표준 대비 검색·매칭·규칙 분류만 수행하며, 외부 LLM 호출을 포함하지 않는다.
- 외부 GPU/LLM 처리는 로컬 처리와 동일하지 않다. 데이터가 해당 클라우드 워커로 전송된다는
  사실을 전제로 데이터 분류와 고객·운영 정책에 따라 사용 여부를 결정한다.
- 결과는 법률적 결론이 아니라 표준 대비 **검토 후보**와 참고 법령으로 표현한다.

## 2. 데이터 처리 경계

| 구성 | 처리 데이터 | 외부 전송 | 기본 용도 |
| --- | --- | --- | --- |
| 로컬 Parser·SQLite·Chroma·로컬 모델 | 계약 원문·조항·표준 코퍼스 | 없음 | 개발, 외부 반출 제한 문서 |
| RunPod 임베더·리랭커 | 조항 텍스트·검색 질의·후보 표준조항 | RunPod 워커 | 운영 1차 검색·재정렬 |
| korean-law-mcp | 법령 검색어·조문 번호 | 외부 법령 서비스 | 법령 본문 참고자료 수집 |
| 향후 2차 LLM | 선택된 검토 결과·조항·법령 근거 | LLM 워커 | 사람이 읽을 설명·추가 조사 보조 |

RunPod에서 자체 Docker 이미지를 운영하면 애플리케이션 코드와 모델은 통제할 수 있지만,
요청 입력은 RunPod Serverless 작업으로 전송된다. Serverless endpoint는 API 키가 있는 HTTP
요청을 받아 워커에서 처리한다. [RunPod 요청 방식](https://docs.runpod.io/serverless/endpoints/send-requests)

## 3. RunPod 사용 기준

### 3.1 허용 조건

- 운영 주체가 해당 계약서의 외부 클라우드 처리를 허용했다고 판단한 경우에만 `APP_ENV=prod`를 사용한다.
- 외부 반출 제한, 비밀유지 특약, 민감 개인정보가 포함된 문서는 로컬 모델 또는 별도 승인된
  전용 환경에서 처리한다.
- RunPod endpoint는 서비스 전용으로 분리하고, API 키는 **Restricted** 권한과 해당 endpoint의
  최소 권한만 부여한다. [RunPod API 키 권한](https://docs.runpod.io/get-started/api-keys)
- 운영 키는 `.env` 또는 배포 환경의 시크릿 저장소에서만 주입하며 저장소·이미지·로그에 포함하지 않는다.

### 3.2 컨테이너·저장소

- 자체 Docker 이미지에는 필요한 모델·라이브러리만 포함하고, 디버그 도구·불필요한 서비스·시크릿을 넣지 않는다.
- 네트워크 볼륨은 모델 캐시 등 필요한 영속 데이터에만 사용한다. 계약 원문·추론 요청·LLM 프롬프트는
  기본적으로 저장하지 않는다.
- 네트워크 볼륨은 Serverless 워커에 마운트되는 영속 저장소이므로 접근 권한과 보존 기간을 관리한다.
  [RunPod 네트워크 볼륨](https://docs.runpod.io/storage/network-volumes)
- 이미지, 모델 가중치, 의존성의 버전과 출처를 기록하고 취약점 점검·재빌드 절차를 운영한다.

## 4. 2차 LLM 운영 기준

- 2차 LLM은 1차 결과를 해석·정리하는 보조 계층이다. 1차의 결정론적 분류·평가 지표를 대체하지 않는다.
- 전달 범위는 필요한 조항, 대응 표준, 법령 근거로 최소화한다. 계약 전문을 기본 입력으로 보내지 않는다.
- 주민등록번호, 계좌번호, 서명·도장 이미지, 전화번호·주소·이메일 등 직접 식별 정보는 가능한 한
  전송 전 마스킹한다.
- LLM 프롬프트·응답·도구 호출 로그에 원문을 기본 저장하지 않는다. 품질 분석이 필요하면 비식별화된
  샘플과 보존 기간이 명시된 별도 저장소를 사용한다.
- 2차 LLM이 MCP 도구를 호출할 때는 사용자·세션 권한, 호출 대상 도구 allowlist, 요청 크기 제한,
  호출 횟수 제한을 적용한다.

## 5. 로그·접근·전송 통제

- API 키, 인증 헤더, 계약 원문, 전체 프롬프트, 파일의 base64 내용은 로그에 남기지 않는다.
- 운영 로그에는 요청 ID, 처리 시간, 상태, 조항 수, 이탈 분포, 오류 코드 등 최소 진단 정보만 남긴다.
- MCP 서버와 외부 endpoint 통신은 인증된 HTTPS를 사용한다. endpoint URL만으로 접근을 허용하지 않는다.
- 서비스 계정·개발자·운영자의 권한을 분리하고, 키 유출·인력 변경 시 즉시 폐기·교체한다.
- 업로드 임시 파일은 요청 종료 후 삭제하고, 실패·재시작 경로도 정기적으로 점검한다.

## 6. 평가·통합 테스트

- 단위 테스트와 기본 통합 테스트는 fake Parser/Retriever/Reranker/Grounder를 사용해 외부 전송 없이 수행한다.
- 네트워크 스모크 테스트는 도구·리소스 등록, 법령 조회처럼 계약 원문을 보내지 않는 요청으로 제한한다.
- `src/eval/golden_b/raw/`의 실계약 E2E는 RunPod 전송이 허용된 환경에서만 실행한다.
- E2E 산출물에는 원문 대신 파일 해시, 조항 수, 이탈 분포, 표준조항 ID, 법령명·조번호와 같은
  최소 메타데이터만 저장한다.
- 법령 grounding 결과는 `(법령명, 조번호)` 기준 중복 제거, 비어 있지 않은 본문, `NO_RESULT` 상태를
  검증 항목으로 둔다.

## 7. 사고 대응

1. API 키·계약 원문·민감 로그의 노출을 발견하면 즉시 해당 키를 폐기하고 endpoint 접근을 중지한다.
2. 관련 로그·배포 버전·영향 범위를 보존해 조사하되, 추가 민감정보를 복제하지 않는다.
3. 고객·운영 정책과 적용 법규에 따른 통지·재발 방지 절차를 서비스 운영 주체가 수행한다.
4. 조치 후 권한, 로그 정책, 이미지·시크릿 관리, 테스트 데이터의 보존 상태를 재점검한다.
