# RAG 품질은 질의와 코퍼스의 표현 계약에서 시작된다

> 프로젝트 저장소: [WorkShield — SKN30 3차 프로젝트](https://github.com/SKNETWORKS-FAMILY-AICAMP/SKN30-3rd-2Team)  
> 이 글은 위 프로젝트에서 표준계약서 대비 검토 후보를 찾는 RAG MCP를 구현하며 실제로 겪은 검색 품질 문제와 표현 계약의 설계 과정을 정리한 글이다. 법률적 판단이나 조항의 위법성을 설명하지 않고, 질의·코퍼스·평가 데이터가 어떤 표현으로 검색되어야 하는지를 다룬다.

프로젝트 발전 과정은 [잘 찾는 도구에 신뢰 기준을 세우다](https://velog.io/@hong1008/ai-camp-3rd-project-3)에서 다뤘다. 이 글은 그중 “데이터를 늘렸는데도 검색되지 않았던” 사건을 검색 입력 인터페이스의 관점에서 다시 살펴본다.

핵심은 데이터의 양이 부족했던 것이 아니었다. 추가한 데이터와 실제 질의가 서로 다른 검색 언어를 사용하고 있었다.

본문에서 query는 질의, corpus는 검색 대상 문서 모음(코퍼스), reranker는 후보 재정렬기, threshold는 판정 임계값으로 표현한다. BM25는 단어 출현 정보를 이용하는 검색 방식이고, grounding은 검색한 근거를 결과에 연결하는 작업이다. recall은 정답을 놓치지 않은 비율인 재현율, precision은 검색 결과 중 정답 비율인 정밀도다. TP(true positive, 참 양성)는 실제 독소를 독소로 찾은 사례이고, F1은 정밀도와 재현율을 함께 요약한 점수다. 이 글의 “활성 검색 대상”은 현재 비교에 사용하는 데이터이고, archive는 보관 영역을 뜻한다.

## 코퍼스를 40건에서 90건으로 늘렸는데도 TP가 0이었다

독소 패턴 코퍼스의 검색 recall을 높이기 위해 기존 40건에 하도급 계약서의 당사자 표현을 반영한 변형을 추가했다. 코퍼스는 90건으로 늘었고, 데이터베이스와 벡터 인덱스도 다시 만들었다.

단일 문장을 직접 검색하면 새 패턴이 정상적으로 높은 점수로 나왔다. 그런데 골든셋 전체를 실제 평가 경로로 실행하면 SI·SM 계약의 독소 참 양성은 여전히 0건이었다.

처음에는 다음을 의심했다.

- 새 데이터가 인덱스에 들어가지 않았는가?
- 독소 threshold가 너무 높은가?
- 계약유형 필터가 새 패턴을 제거했는가?

하지만 데이터와 인덱스는 정상이었다. 문제는 검색 대상의 내용이 아니라 질의와 코퍼스가 서로 다른 표면형으로 입력되고 있다는 데 있었다.

## 헤더 하나가 reranker 점수를 바꿨다

골든셋의 사용자 조항은 Markdown 조항 헤더를 포함하고 있었다.

```text
### 제43조(경업금지 및 인력 유출 방지)
본문...
```

반면 독소 패턴 코퍼스는 제목이나 헤더가 없는 평문이었다.

```text
본문...
```

동일한 내용의 문서쌍을 두고 헤더만 바꿔 reranker 점수를 비교했다.

| 계약유형 | 헤더 포함 질의 | 헤더 제거 질의 |
| --- | ---: | ---: |
| SI_SUBCONTRACT | 0.00036 | 0.9979 |
| SW_FREELANCE | 0.009 | 0.9999 |

이 비교는 다음 조건을 고정한 단일 변수 비교였다.

- 임베딩 모델과 reranker 모델은 동일했다.
- 비교한 후보 문서와 후보 순서는 동일했다.
- 질의와 코퍼스의 내용·숫자·당사자 표현은 바꾸지 않고 헤더 표현만 달리했다.
- 검색 후보를 고르는 단계가 아니라 동일한 질의·후보 쌍의 reranker 점수를 비교했다.
- 점수는 이 프로젝트의 `BgeReranker`가 반환하는 0~1 범위의 정규화 점수였다.

헤더를 제거하자 같은 독소 패턴이 즉시 상위 후보가 됐다. 모델이 독소 개념을 모르는 것이 아니었다. 질의와 코퍼스가 같은 내용을 서로 다른 형식으로 표현하고 있었고, 그 형식 차이가 reranker(후보 재정렬기) 입력의 큰 차이가 됐다.

표준조항 검색이 상대적으로 정상인 이유도 여기서 설명됐다.

| 입력 | 저장 표현 | 검색 표현 | 결과 |
| --- | --- | --- | --- |
| 표준조항 | `### 제N조(제목)` 포함 | 같은 헤더 포함 | 대칭 |
| 사용자 조항 | `### 제N조(제목)` 포함 | 같은 헤더 포함 | 대칭 |
| 독소 패턴 | 본문 평문 | 헤더 포함 질의와 비교 | 비대칭 |

검색 품질 문제를 모델 교체나 threshold 조정으로 해결하려 했지만, 먼저 확인했어야 할 것은 입력 표현의 대칭이었다.

## 독소 검색에만 패치를 붙이지 않기로 했다

가장 쉬운 수정은 독소 검색 직전에만 헤더를 제거하는 것이었다.

```python
# 특정 검색 경로에만 적용하는 국소 패치
toxic_query = remove_markdown_header(user_clause)
```

이 방식은 당장의 독소 사례를 고칠 수 있지만, 표현 계약을 더 불안정하게 만든다.

- 표준조항·서브청크·독소패턴마다 다른 전처리 규칙이 생긴다.
- 런타임 review와 평가용 직접 검색이 다른 입력을 사용하게 된다.
- 새 컬렉션이 추가될 때 어느 경로에서 어떤 정규화를 호출해야 하는지 다시 판단해야 한다.
- 검색 결과가 데이터셋별 예외 코드에 의존한다.

그래서 문제를 “독소 검색의 헤더 제거”가 아니라 “검색에 들어가는 텍스트의 공통 표현 계약”으로 다시 정의했다.

검색 경로마다 다른 정규화를 붙이면 성능은 고칠 수 있어도 재현성은 잃는다.

## 검색용 정규화는 구조적 표식만 제거한다

WorkShield에는 검색 전용 순수 함수 `normalize_for_search()`를 두었다. 이 함수가 하는 일은 검색을 방해하는 구조적 표식을 제거하는 것이다.

- `### 제N조(제목)`에서 Markdown 기호·조번호·괄호 제거
- 제목은 본문 앞에 남김
- 줄 시작의 `①`, `1.`, `1)`, `- 1)` 같은 항·호 기호 제거
- 본문의 실제 어휘와 숫자는 유지

여기서 “헤더 제거”는 본문 전체를 삭제한다는 뜻이 아니다. Markdown 기호와 조번호·괄호 같은 구조 표식은 제거하지만, 헤더 안의 제목은 본문 앞에 남긴다.

예를 들어 입력은 다음처럼 바뀐다.

```text
변경 전:
### 제43조(경업금지 및 인력 유출 방지)
① 수급사업자는 ...
② 원사업자는 ...

변경 후:
경업금지 및 인력 유출 방지
수급사업자는 ...
원사업자는 ...
```

구현의 핵심은 모두 지우는 것이 아니다. 제목은 의미 검색에 유용하므로 보존하고, 숫자는 기간·금액·비율을 나타낼 수 있으므로 보존한다. 코퍼스마다 유무가 달라지는 구조 표식만 제거한다.

```python
def normalize_for_search(text: str) -> str:
    """검색·임베딩 입력 전용 표면형을 정규화합니다."""
    match = _CLAUSE_HEADER_RE.match(text)
    if match:
        title = match.group(1).strip()
        rest = text[match.end():]
        text = f"{title}\n{rest}" if title else rest
    return _LEADING_ENUM_STRIP_RE.sub("", text).strip()
```

실제 정규식은 줄 시작을 기준으로 여러 줄의 항·호 기호를 처리한다.

```python
_LEADING_ENUM_STRIP_RE = re.compile(
    r"^[ \t]*-?[ \t]*(?:[0-9]+[.)]|[①-⑳])[ \t]*",
    re.MULTILINE,
)
```

`re.MULTILINE`이 있으므로 첫 줄만 바뀌는 것이 아니다. `①`, `1.`, `2)`, `- 3)`이 각 줄의 시작에 있으면 모두 제거된다. 반대로 본문 중간의 숫자나 연도처럼 줄 시작 항·호가 아닌 값은 이 패턴의 대상이 아니다.

이 함수는 외부 모델이나 DB를 알지 않는 순수 함수다. 따라서 색인·런타임·평가가 같은 규칙을 공유할 수 있다.

## 원본 문서와 검색 표현을 분리했다

정규화된 텍스트를 원본 데이터에 덮어쓰지 않았다. 검색 표현은 임베딩·BM25의 입력이고, 원본은 응답·근거·진단에 사용해야 하기 때문이다.

데이터 흐름은 다음과 같다.

```text
원본 문서
    ├─ 응답·grounding·진단용 원문
    └─ 검색용 파생 표현
         ├─ Chroma 색인
         ├─ BM25 색인
         ├─ 런타임 질의
         └─ 평가 직접 검색
```

색인 시에는 정규화된 사본만 사용한다.

```python
vector.upsert_documents(
    collection_name,
    documents=[normalize_for_search(row["text"]) for row in rows],
    ids=[row["clause_id"] for row in rows],
    metadatas=[...],
)
```

런타임 질의도 같은 함수를 통과시킨다.

```python
clause_texts = [
    normalize_for_search(clause.text)
    for clause in clauses
]
```

반면 결과를 사용자에게 보여줄 때는 원본을 사용한다.

```json
{
  "user_clause": "### 제43조(경업금지 및 인력 유출 방지)\n① ...",
  "matched_standard": {
    "text": "### 제43조(경업금지 및 인력 유출 방지)\n① ..."
  }
}
```

검색 표현은 비교를 위한 내부 사본이고, 원본 표현은 사용자가 확인할 근거다. 둘을 섞으면 검색 성능을 위해 바꾼 텍스트가 최종 결과에 나타나거나, 반대로 표시용 원문이 검색 품질을 오염시킬 수 있다.

## 색인·런타임·평가가 같은 함수를 써야 했다

표현 계약을 함수 하나로 만들었다고 문제가 끝나지는 않았다. 호출 경로 중 하나라도 빠지면 다시 비대칭이 생긴다.

실제로 `run_eval.py`의 A-1 검색 경로는 `review_contract`를 거치지 않고 검색기를 직접 호출했다. 코퍼스는 이미 정규화되어 있는데 평가 질의만 원문을 사용하고 있었다.

여기서 이 결과는 앞에서 본 독소 TP 평가와 다른 축이다. 앞의 TP 0은 독소패턴 후보를 찾고 독소 상태를 예측하는 평가였고, 아래 Recall@5는 SI 표준조항 검색 후보가 정답을 포함하는지 보는 A-1 검색 평가다. 같은 정규화 변경에서 관찰됐지만 동일한 지표나 동일한 데이터셋은 아니다.

SI 계약의 `hybrid_rerank` Recall@5를 비교하면 다음과 같았다.

| 평가 질의 | 재현율(Recall@5) |
| --- | ---: |
| 원문 질의 | 0.867 |
| 정규화 질의 | 0.900 |

따라서 세 경로를 명시적으로 맞췄다.

```text
pipe/build_index.py   ── normalize_for_search ── 색인
pipe/review_pipe.py   ── normalize_for_search ── 런타임 검색
eval/run_eval.py      ── normalize_for_search ── 평가 검색
```

### 변경 전

```text
원본 코퍼스 ─────────────→ 색인
헤더 포함 사용자 질의 ──→ 검색
헤더 없는 독소 코퍼스 ──→ 비대칭 비교
평가 질의 ──────────────→ 별도 규칙
```

### 변경 후

```text
원본 코퍼스 ── 정규화 ──→ 색인
원본 사용자 질의 ─ 정규화 ─→ 검색
원본 평가 질의 ── 정규화 ─→ 평가
원본 텍스트 ────────────→ 응답·근거
```

변경은 모델이나 threshold를 바꾼 것이 아니었다. 같은 질문을 같은 표현 계약으로 전달하도록 입력 경계를 고친 것이다.

## 정규화 이후에도 모든 문제가 해결되지는 않았다

정규화는 검색 입력의 비대칭을 줄였지만, 검색 품질의 모든 원인을 제거하지는 않았다.

- 독소 recall은 개선됐지만 점수 분포가 바뀌면서 precision이 악화된 구간이 있었다.
- 검색 후보를 잘 찾는 것과 독소인지 의미적으로 판정하는 것은 여전히 다른 질문이다.
- 숫자·기간·당사자 표현을 과도하게 지우면 중요한 신호가 사라질 수 있다.
- 새로운 코퍼스 포맷이 추가되면 표현 계약이 다시 검증되어야 한다.

즉, 정규화는 “정답을 찾는 함수”가 아니라 “질의와 코퍼스가 비교 가능한 표면형을 갖게 하는 함수”다.

## 평가는 모델 점수가 아니라 질문 단위로 나눴다

정규화 전후의 reranker 점수 하나만 보고 품질을 판단하지 않았다. 다음 질문을 분리했다.

| 질문 | 평가 방법 |
| --- | --- |
| 질의와 코퍼스가 같은 표현 계약을 따르는가? | 헤더 유무 등 동일 문서쌍 점수 비교 |
| 정답 표준조항을 후보에 넣었는가? | 재현율(`Recall@k`), 평균 역순위(`MRR`) |
| 정규화가 매칭 경계를 바꿨는가? | NONE/EXTRA 혼동행렬 |
| 독소 검색 후보를 찾았는가? | 계약유형별 정밀도(precision)·재현율(recall)·F1 점수 |
| 평가 경로가 런타임과 같은 입력을 쓰는가? | 색인·review·eval 경로 회귀 검증 |

모델 점수가 높아졌다는 사실은 입력 표현이 개선됐다는 단서일 수 있지만, 그 자체가 시스템 품질의 최종 지표는 아니다. 어떤 질문을 개선했는지 분리해서 측정해야 한다.

## 시스템의 책임 경계

검색용 표현 정규화가 책임지는 범위는 명확하다.

- 질의와 코퍼스의 구조적 표현 차이를 줄인다.
- 색인·런타임·평가가 같은 입력 규칙을 사용하게 한다.
- 원본을 보존하면서 검색용 파생 표현을 만든다.

정규화가 책임지지 않는 것도 명확하다.

- 두 조항의 의미가 같은지 확정하지 않는다.
- 높은 reranker 점수를 법률적 판단으로 해석하지 않는다.
- 기간·대가·권리 주체의 유불리를 판정하지 않는다.

이 경계를 분리했기 때문에 1차 MCP는 검색 후보와 원문 근거를 제공하고, 필요한 의미 해석은 별도의 후속 계층이 담당할 수 있다.

## 내가 배운 설계 원칙

- 모델을 바꾸기 전에 질의와 코퍼스의 표현이 대칭인지 확인한다.
- 검색 전처리는 컬렉션별 예외가 아니라 공통 표현 계약으로 만든다.
- 원본 문서와 검색용 파생 표현을 분리한다.
- 색인·런타임·평가가 같은 정규화 함수를 공유하는지 확인한다.
- 제목·숫자·본문처럼 검색에 필요한 신호와 제거할 구조 표식을 구분한다.
- 정규화 효과는 모델 점수가 아니라 검색·매칭·독소·회귀라는 질문별 지표로 평가한다.
- 표현 계약을 맞추는 일은 의미 동치성을 판정하는 일과 다르다.

RAG 품질은 모델의 크기나 코퍼스의 양에서만 시작하지 않는다. 질의와 코퍼스가 같은 표현 규칙으로 비교되고, 원본과 검색 표현의 책임이 분리될 때 비로소 검색 결과를 재현 가능한 품질 문제로 다룰 수 있다.

검색되지 않는 데이터를 더 추가하기 전에, 기존 데이터와 질의가 같은 표현 계약으로 비교되고 있는지 먼저 확인해야 한다. 모델과 threshold는 그다음 문제였다.
