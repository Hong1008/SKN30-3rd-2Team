## 독소 코퍼스·텍스트 정규화

### D60. 독소 패턴은 계약유형 제약 대신 어휘 변형으로 보강
- **결정:** ToxicPattern에 `contract_types`를 추가하지 않고, 기존 40건에 하도급 당사자 어휘 변형 40건과 `INDEFINITE_CONFIDENTIALITY` 10건을 더해 90건으로 확장한다.
- **이유:** 독소 개념은 모든 지원 계약유형에 성립하며 문제는 개념 범위가 아닌 당사자 어휘 불일치다. 유형 필터는 오히려 후보를 없앨 수 있다.
- **상태:** 구현·재빌드 완료 (07-09)

### D61. 독소 무검출의 근본 원인은 헤더 포맷 비대칭
- **결정:** 코퍼스 증식 후 recall이 개선되지 않은 원인을 N 카드와 분리해 텍스트 정규화 P 카드로 관리한다.
- **이유:** 헤더 포함 사용자 쿼리와 헤더 없는 독소 코퍼스의 리랭킹 점수가 약 0.99에서 0.0003대로 붕괴했다. 표준조항은 포맷이 대칭이라 이탈 탐지가 정상이었다.
- **상태:** 범위 분리; 대칭 정규화 구현은 병행 작업으로 확인 (07-09)

### D62. 평가 직접 검색에도 동일 정규화를 적용
- **결정:** `run_eval.py`의 A-1 직접 검색 질의에 `normalize_for_search`를 적용한다.
- **이유:** review 경로는 정규화돼 있었지만 A-1은 원문 질의로 정규화된 코퍼스를 조회해 비대칭을 재발시켰다.
- **상태:** 즉시 수정·회귀 확인 (07-09)

## 데이터 원천·실험 통제

### D63. 활성 표준판은 계약유형별 최신 1개, 과거판은 archive
- **결정:** SI/SM 2022 표준과 서브청크는 archive로 옮기고, MVP 검색 풀은 최신 SI/SM 2025와 보유 SW 2020판만 사용한다. 활성 루트의 단일 version 불변식을 migration에 둔다.
- **이유:** version 미스코핑으로 신·구 표준이 한 검색 풀에 공존해 EXTRA가 과거판에 잘못 매칭됐다. 과거 시점 비교는 별도 계약 변경 승인 과제로 남긴다.
- **상태:** Q 완료 (07-10)

### D64. 튜닝은 고정 held-out으로 검증하고 기본값 유지
- **결정:** RRF 5:5와 `match_threshold=0.50`, `toxic_threshold=0.60`을 유지한다.
- **이유:** RRF 대안은 우세하지 않았고, tuning에서 좋아진 threshold 후보가 고정 held-out에서 재현되지 않았다. 추가 튜닝으로 기본값을 바꾸지 않는다.
- **상태:** 채택 (07-10)

### D65. 제품 표면은 ‘검토 후보’로 제한
- **결정:** 최신 표준 대비 사람 검토를 보조하는 데모·파일럿만 허용하고, 독소/이탈의 자동 위험 판정 상용화는 보류한다.
- **이유:** 독소 precision 0.205, held-out 불안정성, 이탈 FN이 자동 결론을 지지하지 못한다.
- **상태:** 범위 확정 (07-10)

## MCP 조립·법률 도구

### D66. 앱 composition root와 법률 보조 도구 DI
- **결정:** `app.py::create_app()`이 MCP composition root가 되어 기존 WorkShield 도구와 법률 MCP 9개 도구를 주입·등록한다.
- **이유:** 2차 LLM은 법률 리서치 도구를 조합할 수 있지만, 1차 파이프라인은 이를 해석·판정에 사용하지 않는 경계를 유지한다.
- **상태:** 채택·등록 계약 검증 (07-10)

### D67. 지원 범위를 결정론적으로 사전 판별
- **결정:** 기존 review 계약은 바꾸지 않고 `assess_contract_scope`를 추가해 IN_SCOPE/CONTRACT_TYPE_UNCERTAIN/OUT_OF_SCOPE를 앵커 규칙으로 반환한다.
- **이유:** 파일명·LLM 없이 지원 범위 경고를 제공하고, 불확실 문서는 사용자가 유형을 명시하면 기존 검토 흐름을 이용하게 한다.
- **상태:** 채택 (07-10)

### D68. 법령 MCP는 장기 세션·제한 캐시·조문 한정 grounding
- **결정:** 앱 lifespan에서 법령 MCP stdio 세션 하나를 공유하고, 요청 캐시와 TTL LRU 캐시를 둔다. 정적 grounding은 조문번호가 명시된 매핑만 호출하며 결과는 최대 3건으로 제한한다.
- **이유:** 매 호출 프로세스 생성으로 캐시가 사라지고, 포괄 질의·본문 인용 오인으로 근거가 82,080건까지 폭증했다.
- **검증 기록:** 정상 E2E의 근거 수 82,080→101, 평균 처리시간 66.1초→19.1초.
- **상태:** 채택·E2E 검증 (07-10)
