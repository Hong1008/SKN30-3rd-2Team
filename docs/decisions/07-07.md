## 2. `core/deviation.py` 실측 버그 2건 수정

v3 케이스를 실제 함수(`calculate_text_similarity`·`detect_critical_changes`)에 태워 의도한 라벨과
실제 동작이 맞는지 검증하는 과정에서 발견·수정.

- **`_NEGATION_RE` — "아니 된다" 미탐지**: "된"(되+ㄴ)과 "되"는 다른 유니코드 문자라 `아니되`
  패턴이 "~하여서는 아니 된다"(계약서에서 가장 흔한 금지 표현)를 못 잡고 있었음. `아니된` 패턴
  추가로 수정.
- **`_PARTY_RE` — 숫자 뒤 조사 "을" 오탐**: "10000분의 1을 곱하여"처럼 숫자 뒤에 오는 목적격
  조사 "을"이 당사자 "을(乙)"로 오탐됨(기존 가드는 앞이 한글일 때만 조사로 인정, 숫자는 미고려).
  `(?<![가-힣])` → `(?<![가-힣0-9])`로 수정.
- **테스트 추가**: `tests/core/test_deviation.py`에 두 버그 재현 테스트 2건 추가. 전체
  `uv run pytest tests/core/ tests/pipe/ tests/eval/` 106 passed 확인.
- **세 번째 갭 발견 시 패치 중단**: "~할 수 없다"(모달 불가능, "없다/없이") 전혀 미커버 확인 —
  이 시점에서 사용자가 "정규식으로 이런 걸 판별한다는 명제 자체"를 재검토하자고 제안, 개별 패치를
  멈추고 §3 아키텍처 재설계로 전환.

## 3. 명제 재검토 — 규칙기반 판정 계층은 왜 구조적으로 실패하는가

정규식 3건(아니된다·숫자+을·없다) 전부가 "개별 버그"가 아니라 **두 가지 서로 다른 구조적 실패
유형**의 사례임을 확인:

1. **열거 불완전성**: 한국어 부정어 표현(아니하다/아니되다/않다/못하다/없다/말다/한자 부정접두사)과
   숫자 표기(아라비아 vs 한글 숫자, "오 년")를 정규식으로 완전히 열거하는 건 원리상 끝나지 않는다.
2. **마커 없는 의미 반전**: `v3-sw-11`(공동소유→도급인 단독 영구귀속)처럼 부정어·숫자·당사자
   집합 자체가 안 바뀌는 순수 의미 반전은 어떤 정규식 정교화로도 원천 탐지 불가 — "바뀐 사실"이
   표면 토큰이 아니라 개념 간 관계에 있기 때문. 당사자 검출도 집합 비교로는 **주어-목적어 역할
   반전**(누가 의무자인가)을 표현 못 한다는 한계를 `v3-sw-09` 설계 시도 중 확인.

리랭커 기반 커버리지 체크(관련성 점수)도 "의미 동치성"과는 다른 신호라는 07-03 §2 진단과 같은
계열의 문제로 재확인됨.

## 4. 1차/2차 판정 경계 확정 — 규칙계층 완전 제거

§3 분석을 근거로 아래 4개 사안을 사용자 사인오프로 확정(AGENTS.md §2 동결 계약 변경):

- **규칙기반 치명변경 탐지(`detect_critical_changes`, `_NEGATION_RE`/`_NUMBER_RE`/`_PARTY_RE`,
  `CRITICAL_NEGATION`/`CRITICAL_NUMBER`/`CRITICAL_PARTY`) 완전 제거.** 축소·보강이 아니라 삭제 —
  2차 LLM이 정렬된 원문 쌍을 직접 읽고 판단하며, 신뢰 못 할 규칙 힌트로 프롬프트에 노이즈를 얹지
  않는다.
- **difflib 폴백(커버리지 불가 시 글자일치율로 NONE/CHANGED 가르던 로직)도 동일 원리로 제거.**
  커버리지 체크 불가 시 **무조건 CHANGED**(안전측, 2차로 위임)로 단순화.

  준수.
- **`DeviationResult`에 새 필드 추가 없음.** 2차가 필요한 정보(`user_clause`·`matched_standard.text`·

  무변경, **`NONE`의 의미만 "확정"에서 "1차 잠정치(2차 확인 전)"로 재정의**.

## 6. Grounder 계약유형별 법령 그라운딩 결함 발견 → task O 생성

사용자가 "data/에 SI/SM 하도급 계약서가 섞이면서 그라운딩 쿼리도 유형별로 달라야 하지 않냐"고
제기, 코드로 실측 확인:

- `Grounder.get_grounding(self, category: Category)` — Protocol 자체에 `contract_type` 인자 없음.
- `CATEGORY_QUERIES`(`korean_law_grounder.py`)는 유형 무관 고정 딕셔너리, 매핑 10건 전부 민법·
  저작권법·부정경쟁방지법·근로기준법 — 코드 전체에 "하도급법" 언급 0건.
- Category 15종 중 4종(`CONTRACT_PERIOD`·`DELIVERY_INSPECTION`·`WARRANTY`·`SUBCONTRACTING`)은
  아예 미매핑 → `"민법 도급"` 폴백. 실측: SI/SM 표준 코퍼스에서 **`SUBCONTRACTING`(재하도급)이
  최빈값(12건)인데 완전 미매핑** — 가장자리 케이스가 아니라 다수 조항에 영향.
- 호출부 3곳(`review_pipe._grounding_for`, `server.py`의 `classify_clause`·`review_contract`·단독
  `get_grounding` 도구) 전부 `contract_type`이 스코프에 있는데도 안 넘김.
- 이 결함은 `run_eval.py`가 `NullGrounder`(no-op)를 쓰므로 **어떤 result.md에도 드러난 적 없음**
  — 실제 MCP 서버/데모 경로에서만 나타나는 숨은 결함.
- [Z_Lever.md](../tasks/Z_Lever.md)에 `grounding_ref`(하도급법 제13조) 아이디어가 있었으나, 항 단위
  "법률 레버" 판정이라는 더 큰 설계(§3의 규칙기반 한계와 같은 축 + NLI)에 묶여 있었고 그 설계
  전체가 이미 "오버엔지니어링"으로 기각돼, 이 좁은 문제는 단독으로 다뤄진 적 없었음.

**확정**: `get_grounding`에 `contract_type: Optional[ContractType] = None` 가법 추가(하위호환),
SW_FREELANCE 쿼리는 불변, SI/SM 전용 override 테이블 신설(우선 조회 → 없으면 기존 공용값 폴백).
사용자 제공 초안 3건(`PAYMENT`→하도급법 제13조, `DELIVERY_INSPECTION`→제9조, `CONFIDENTIALITY`→
제12조의3) 반영. 나머지 조문(특히 최빈값인 `SUBCONTRACTING`)은 법률 검토가 필요해 이 세션에서
추측으로 채우지 않고 [O_grounding_contract_type.md](../tasks/O_grounding_contract_type.md)의
오픈 항목으로 남김.

## 7. (2026-07-08 후속) J 구현 — CHANGED 폐지·서브청크 커버리지 전체 삭제로 재확장

§4에서 확정한 원안(커버리지 미달→CHANGED 확정 / 충족→잠정 NONE)을 사용자가 J 카드 구현 계획을
검토하던 중, `classify_clause_deviation`을 서버 밖 `classify_clause` 단독 툴도 호출한다는 사실이
드러나며 범위가 재검토됨. 논의 끝에 원안보다 훨씬 크게 단순화하기로 확장 결정(사용자 직접 구현,
Claude는 검증):

- **`Deviation.CHANGED` 완전 폐지.** 남는 값은 `MISSING`/`EXTRA`/`NONE`/`NO_MATCH` 4종.
- **서브청크 의미 커버리지 2패스(`check_coverage`, `all_standard_sub_chunks` 커버리지 입력,
  `coverage_threshold`, `use_coverage`) 전체 삭제.** (서브청크 검색·`roll_up_sub_chunks` 매칭 보강
  자체는 커버리지와 무관해 유지)
- **새 판정**: 매칭 성공(`score >= match_threshold`) → `NONE`(잠정). 미달 → `EXTRA`. 근거: "내용이
  얼마나 다른지"는 규칙·리랭커로 판별 불가하다는 §3 결론을 EXTRA/NONE 경계에도 그대로 적용 — 1차는
  "매칭 여부"만, 나머지는 전부 2차(LLM) 몫.
- **골든셋 재라벨링**: 기존 `gold_deviation: "CHANGED"`(106건, 7개 파일) → `"NONE"`으로 일괄
  치환(EXTRA 아님 — CHANGED는 "매칭은 됐지만 내용이 다름"이었으므로 신 설계에서 정확한 대응은
  잠정 NONE). `validate_golden.py`의 `paraphrase+NONE 금지` 규칙(구 철학)도 함께 폐지.

## 8. `select_best_match` 임계값 미적용 확장 — EXTRA도 `matched_standard` 보유

§7 직후 "EXTRA는 왜 2차가 설명할 근거(비교 대상)가 없냐"는 문제 제기(사용자) → 검토 끝에
`core/matching.py::select_best_match`가 **임계값 판정을 하지 않고 항상 최고점 후보를 반환**하도록
변경(Claude 구현). `classify_clause_deviation(matched_standard, score, match_threshold)`로 시그니처
확장해 EXTRA/NONE 판정을 여기서 확정.

- **부작용 발견·수정**: `review_pipe.py`가 `matched_standard is not None`이면 무조건 MISSING 탐지용
  "커버됨"으로 기록하던 로직이, 이제 임계 미달 EXTRA 근접후보까지 오염시킬 뻔함 →
  `deviation == Deviation.NONE`일 때만 기록하도록 수정(회귀 테스트 추가:
  `test_임계미달_근접후보는_MISSING_탐지를_오염시키지_않음`).
- 효과: K(2차 LLM)가 별도 `match_clause` 재호출 없이 `DeviationResult.matched_standard`만으로
  EXTRA도 설명 가능해짐(§11 K 문서 개정에 반영).

## 11. `Category.contract_types` 필드 신설 — korean-law-mcp 실측 근거 기반 스코핑

§10 조사 중 재발견된 문제(`label_category`가 `contract_type`을 몰라 근로계약 전용 카테고리
`WORKING_HOURS`가 SI/SM 코퍼스에 실제로 새어 들어가 있음, 1~2건씩 총 6건 실측)를 놓고 "O의 결함과
같은 문제인가"·"정규화가 grounding 시점을 알 필요가 있는가"를 사용자와 검토 → **별개 결함이며,
정규화 시점엔 불필요(grounding 시점엔 이미 contract_type을 앎)**로 결론(§9와 독립). 이후 사용자가
`contract_types` 필드 추가를 요청, 진행 전 "SW_FREELANCE와 SI/SM을 같은 그룹으로 묶는 게 타당한가"
재검증 요청 → SW_FREELANCE 자체의 기존 라벨(`DELIVERY_INSPECTION`/`WARRANTY`/`SUBCONTRACTING`
9건 중 4건만 정확, `art13 "장소"→SUBCONTRACTING` 등 명백한 오분류 발견)도 오염돼 있어 "실측
데이터로 실측 데이터를 검증하는 순환"이 되는 걸 확인 후, korean-law-mcp로 실제 법조문을 조회해
근거를 확보:

| 카테고리 | `contract_types` | 근거(실측 조회) |
| --- | --- | --- |
| `WORKING_HOURS`/`HOLIDAY_LEAVE` | `SW_EMPLOYMENT`만 | 근로기준법 특유 개념 |
| `DELIVERY_INSPECTION`/`WARRANTY`/`SUBCONTRACTING` | `SW_FREELANCE`+`SI`+`SM` | 민법 제667조(수급인의 담보책임)가 도급 일반에 적용 확인 |
| `INDUSTRIAL_SAFETY` | `SI`+`SM`+`SW_EMPLOYMENT` (SW_FREELANCE 제외) | 산업안전보건법 제63조 — "관계수급인 **근로자**" 보호 전제라 근로자 없는 개인 프리랜서는 구조상 해당 안 됨(특수형태근로종사자 열거에도 미포함) |
| `INFO_SECURITY` | `SM`만 | 정보통신망법 제45조는 유형 무관 일반조항이나, 현재 표준계약서 템플릿엔 SM에만 실제 조항 존재(실증적 제한, 법적 필연 아님 — 확장 가능) |
| 나머지 9종 + 신규 무제한 카테고리 | 전체 공통 | — |

구현: `Category.__new__`에 `contract_types` 필드 추가. **버그 발견·수정**: 그룹 공유용 헬퍼
`_SUBCONTRACT_FAMILY`를 `Category` 클래스 바디 안에 선언했더니 Python Enum이 이를 진짜 멤버로
오인(단순 밑줄 접두사는 sunder/dunder가 아니라 자동 제외 안 됨) → 모듈 레벨 상수로 이동해 수정.
`label_category(title, text, contract_type)`로 확장해 후보를 스코핑, `get_grounding`엔 무효조합
경고 로그, `list_categories(contract_type=)` 필터링 추가. 실제 임베딩 모델로 통합테스트
(`-m integration`, `tests/pipe/test_normalize.py` 16건) 전량 통과로 스코핑 후에도 정확도 유지 확인.

## 12. O 잔여 미결정 해소 — 신규 카테고리 + `TERMINATION`/`LIABILITY`/`WARRANTY` grounding 확정

§11에서 발견한 SI/SM 실제 조항명("부당한 위탁취소 등의 금지")을 단서로 korean-law-mcp 재조회:

- `WARRANTY`(공용): 민법 제667조 — §11에서 이미 확인한 근거를 grounding 쿼리에도 반영.
- `TERMINATION` SI/SM 오버라이드: 하도급법 제8조(부당한 위탁취소의 금지 등) — SI/SM 표준계약서
  실제 조항명과 정확히 일치, 민법 제673조(도급인 임의해제권)와는 반대 방향(원사업자의 부당 취소로
  부터 수급사업자 보호)이라 SW_FREELANCE와 근거 분리.
- `LIABILITY` SI/SM 오버라이드: 하도급법 제35조(손해배상 책임, 위반 시 3~5배 가중).
- `INDUSTRIAL_SAFETY`(공용): 산업안전보건법 제63조. `INFO_SECURITY`(공용): 정보통신망법 제45조.
- `SUBCONTRACTING`·`CONTRACT_PERIOD`는 재조사했으나 대응하는 단일 하도급법 조문을 찾지 못해 공용
  폴백 유지. `SCOPE_SOW`·`DISPUTE`는 재검토 결과 SI/SM 오버라이드 불필요로 확정.
- `docs/tasks/O_grounding_contract_type.md`·`docs/archive/tasks/K_second_stage_llm.md` 갱신(K는 §7·§8
  반영 위해 전면 개정 — "배경"·"확정된 설계"가 원안과 달라져 과거 버전 참고 금지 경고 추가, 2차
  대상에 EXTRA 포함 확정, grounding은 2차가 기존 `get_grounding` 도구를 직접 호출해야 한다는 결론
  반영, `match_clause` 재활용안이 §8로 인해 불필요해진 경위 기록).
- 검증: `uv run pytest tests/ -m "not integration"` 95 passed.
