# 의사결정 로그 — 2026-07-03

> [분류기 근본원인 분석 + 판정 아키텍처 재검토] `review_pipe`/`core`/`eval` 전체를 훑어 이탈(deviation)
> 분류기가 고장난 근본 원인을 규명하고, "판정을 LLM 없이 결정론으로만 할 것인가"라는 명제 자체를
> 재검토한 세션. 맥락: v2 골든 재평가에서 이탈 특이도가 threshold를 5회 스윕해도 "전부 CHANGED"
> ↔ "전부 NONE" 두 축퇴 모드 사이만 오갔다([v2_fail/](../../src/eval/golden/v2_fail/)). 코드 버그가
> 아니라 (1) 골든셋 자체의 구조적 결함, (2) NONE/CHANGED 판정에 필요한 "의미 동치성" 신호가
> 결정론 스택에 없다는 두 겹의 원인이 드러나 판정 아키텍처를 어디까지 결정론으로 유지할지 논의.
> 이번 세션은 **분석·명제 합의까지** — 코드 변경 없음, 구현은 후속 세션.

## 0. 세션 결론 (한 줄)

이탈 분류 실패의 1차 원인은 **v2 골든셋이 NONE 도달을 구조적으로 불가능하게 만들어 놓은 것**
(80% 케이스가 단문 user_clause를 다항 표준조항에 NONE으로 매핑 — [v1_review.md](../../src/eval/golden/v1_review.md)
§8.3에서 이미 경고됐던 문제가 v2 제작에서 미반영). 2차 원인은 **NONE↔CHANGED 판정이 본질적으로
의미 동치성(entailment) 문제인데, 결정론 신호(글자 일치율·리랭커 관련성 점수) 어디에도 그 정보가
없어** 게이트·임계값을 아무리 쌓아도 판별력이 안 생긴다는 것. 이에 따라 **판정 계층의 경계를
재정의**한다 — 검색·MISSING/NO_MATCH/EXTRA·규칙 기반 치명변경 플래그는 **결정론 유지**,
NONE↔CHANGED 최종 동치 판정만 **2차(LLM)로 이관**하는 방향에 합의. 동결 계약 변경 사안이라
정식 반영은 별도 사인오프 후.

## 1. 근본 원인 규명 — v2 골든셋의 구조적 결함 (코드 버그 아님)

- **진단:** v2 골든 NONE 라벨 44건 중 35건(80%)이 60~90자 단문 `user_clause`를 서브청크 2개
  이상(최대 28개, 2,898자)짜리 표준조항에 매핑. 현재 NONE 게이트(`check_coverage`,
  [review_pipe.py:362](../../src/pipe/review_pipe.py#L362))는 "표준의 모든 항이 사용자 조항에
  커버돼야 NONE"이라, 단문이 다항 표준을 전부 커버하는 것은 **논리적으로 불가능** — 어떤
  임계값에서도 이 35건은 NONE이 나올 수 없다.
- **선행 경고 미반영 확인:** [v1_review.md](../../src/eval/golden/v1_review.md) §8.3이 정확히
  이 현상(단문 골든 26/33이 다항 표준 매핑 → 커버리지 게이트 계측 부적합)을 진단했고, §6
  체크리스트 최우선 항목이 "v2 골든의 user_clause를 조(條) 전체 텍스트로"였는데 v2 제작에서
  **이행되지 않았다**(체크박스 미완료).
- **부가 라벨 품질 문제(실측, [v2_sw_freelance.json](../../src/eval/golden/v2_sw_freelance.json)):**
  g08은 "계약기간·자동연장" 조항을 "용어의 정의"(art2)에 NONE으로 오매핑. g01·g03 등 강한
  paraphrase가 NONE으로 라벨돼, v1에서 사람이 이미 확정한 엄격안("문구가 바뀌면 CHANGED",
  v1_review §6)과 정면 모순.
- **결론: v2_fail/의 5회 threshold 스윕이 전부 축퇴 모드 사이만 오간 것은 임계값 튜닝 미숙이
  아니라, 골든 데이터 자체가 모순이라 어떤 값도 양쪽을 동시에 만족시킬 수 없었기 때문.**

## 2. 구조적 원인 — 관련성 점수로 동치성을 판정하려는 과제 설정

- **진단: NONE↔CHANGED는 "내용이 보존됐는가(의미 동치성)"를 묻는 문제인데, 결정론 스택이 가진
  신호 두 가지 다 동치성을 재지 못한다.**
  - `difflib` 글자 일치율: 독립 작성된 실계약에서 천장이 0.545 (Track B 실측, v1_review §2 원인 A).
  - 리랭커 로짓(sigmoid 정규화): **주제 관련성**이지 동치성이 아님. "공동소유→일방 영구귀속" 같은
    독소 변조는 원 표준조항과 주제가 완전히 같아 오히려 최고점을 받는다.
- **연쇄 실패:** `detect_critical_changes`(부정어·숫자·당사자 규칙, [deviation.py](../../src/core/deviation.py))가
  이 공백을 메우려는 마지막 방어선인데, `min_pair_ratio=0.5` 가드([deviation.py:116](../../src/core/deviation.py#L116))
  때문에 **말바꿈이 강할수록 정렬 일치율이 떨어져 검사 자체가 스킵**된다 — 정작 잡아야 할
  케이스에서 방어선이 꺼지는 역설. 독소 탐지(recall 0~80%, [v2_after_feature_improvement_result.md](../../src/eval/golden/v2_after_feature_improvement_result.md))도
  동일 뿌리(리랭커 매칭 = 관련성 검색이지 독소 여부 판정이 아님).
- **결론: 신호 자체에 없는 정보는 게이트를 몇 겹 쌓아도 생기지 않는다.** 현재 판정 계층
  (커버리지 → 치명변경 → difflib 폴백 → 독소, 임계값 5개)은 "쓸데없이 복잡"이 아니라
  **신호 부족을 게이트 복잡성으로 보상하려 한 결과**이며, 그 보상이 구조적으로 불가능함이
  이번 분석·v2_fail 실측으로 확인됨.

## 3. 판정 아키텍처 재검토 — RAG 경계는 어디인가 *(명제 합의, 구현 보류)*

- **문제 제기(사용자):** "분류를 LLM 없이 하는 게 옳은가, 이건 RAG 범위를 벗어난 게 아닌가 —
  현 시스템은 검색만 정확히 수행하고, 그 결과를 근거로 2차 LLM이 판정하는 게 맞지 않나."
- **검토 결과: 방향은 맞으나 전면 이관이 아니라 라벨별 분해가 정확한 교정.**

  | 라벨 | 본질 | 필요 신호 | 결정론으로 가능한가 |
  | --- | --- | --- | --- |
  | MISSING | 표준 − 매칭됨 (집합 차) | 검색 결과 | ✅ 가능 — 유지 |
  | NO_MATCH | 후보 0건 | 검색 유무 | ✅ 가능 — 유지 |
  | EXTRA | 대응 표준 없음 | 검색 점수 임계 | 🔶 대체로 가능 — 유지 |
  | **NONE ↔ CHANGED** | **의미 동치성(entailment)** | **의미 판단** | ❌ 원천 불가 — 2차 이관 검토 |

- **RAG 명제 정정:** MCP 서버는 이미 LLM 클라이언트에게 호출되는 구조라 "LLM이 개입하냐"가
  아니라 "서버가 최종 라벨까지 확정해 던지느냐, 근거 패키지만 넘기고 동치 판정은 호출자에
  맡기느냐"가 실제 쟁점. 현재는 전자라 오히려 **정통 RAG(검색→생성)에서 G가 잘려나간 상태**였다는
  진단에 합의.
- **채택 방향(설계 스케치, 미확정):**
  - **1차/MCP 서버(결정론 유지):** 검색 + 항↔항 정렬 + 규칙 플래그(치명변경·독소 후보 등) +
    MISSING/NO_MATCH/EXTRA 라벨까지 산출. 산출물은 최종 라벨이 아니라 **근거 패키지**로 재정의.
  - **2차/LLM(서버 밖, 데모 등 클라이언트 레이어):** `(사용자 조항, 매칭 표준조항, 항 정렬,
    규칙 플래그)`를 grounding으로 받아 NONE↔CHANGED **동치성만** structured output으로 판정.
- **미해결 — 다음 세션 숙제로 이관:**
  1. **eval 결정론(AGENTS.md 규칙 #5, LLM-judge 금지)과의 정합.** 프로덕션 판정이 LLM이면
     그 출력을 무엇으로 재현 가능하게 채점할지 — eval은 검색 지표 + 규칙 플래그 검출률만
     결정론으로 재고, LLM 동치 판정은 별도로(예: 고정 스냅샷 회귀) 검증하는 분리안이 유력하나
     **미확정**.
  2. **동결 계약 변경 범위.** `review_contract` 반환 계약(`DeviationResult.deviation`)의 의미가
     "최종 판정"에서 "1차 후보 판정(근거 포함)"으로 바뀌면 AGENTS.md §2 동결 계약 변경 사안 —
     사람 사인오프 필요.
  3. **core 규칙 코드의 역할 재정의.** `detect_critical_changes` 등은 폐기가 아니라 "최종
     판사"에서 "LLM에 주는 근거 플래그 + 가드레일(리랭커/LLM이 부정어 플립을 뭉개는 것 방지)"로
     격상하는 방향이 유력하나 세부 설계 미착수.

## 4. 독소 탐지 recall 붕괴 — 코퍼스 당사자 어휘 편중 (실측, 신규 규명)

- **진단: 독소 코퍼스가 SW 프리랜서 어휘로만 작성돼 하도급(SI/SM)에는 구조적으로 무력하다.**
  [toxic_patterns.json](../../data/03_normalized/toxic_patterns.json) 40건(8개념 × 5예문)의 당사자
  용어를 실측하니 **도급인 60회·수급인 55회, 원사업자 0회·수급사업자 0회**. 반면 골든의 SI/SM
  독소 케이스 20건은 **전부 원사업자/수급사업자**를 쓰고 도급인/수급인은 0건, SW 독소 6건은 그
  반대(전부 도급인/수급인). 코퍼스 어휘가 SW에만 정렬돼 있다.
- **recall이 어휘 일치도와 정확히 일치(실측):**

  | 유형 | 골든 독소 케이스 어휘 | 코퍼스 일치 | 독소 recall |
  | --- | --- | --- | --- |
  | SW_FREELANCE | 도급인/수급인 | ✅ 일치 | **1.000** (6/6) |
  | SI_SUBCONTRACT | 원사업자/수급사업자 | ❌ 불일치 | 0.429 (3/7) |
  | SM_SUBCONTRACT | 원사업자/수급사업자 | ❌ 불일치 | **0.077** (1/13) |

- **원인 분리:** 리랭커는 독소 *개념*이 같아도(예: 일방 변경·대금 지연) 당사자 용어가 다르면
  점수를 낮게 준다 → 하도급 조항이 코퍼스에 안 물린다. 이는 [v1_review.md](../../src/eval/golden/v1_review.md)
  §2 원인 B가 지목한 "sigmoid 바닥" 임계값 문제와는 **별개의 2차 원인**이며, 임계값을 어떻게
  조정해도 해소되지 않는다(어휘가 안 맞으면 애초에 후보로 안 뜬다).
- **부가 관찰:** (1) 개념 8종 선정 자체는 타당(IP 무상귀속·과도 경업금지·대금 지연·일방 변경·
  과도 손배·일방 해석·일방 해지·무상 추가업무). (2) 개념당 5예문은 검색 기반 탐지엔 얇다.
  (3) `NONCOMPETE_EXCESS`가 `CONFIDENTIALITY` 카테고리에 배치돼 있음(경업금지 ≠ 비밀유지, 애매).
- **채택 방향(미확정, 데이터 변경이라 사인오프 필요):** 효과 순으로
  1. **매칭 전 당사자 용어 정규화**(원사업자↔도급인, 수급사업자↔수급인을 역할 토큰으로 치환) —
     코퍼스를 그대로 두고 코드로 해결, 유형 무관하게 일반화. **1순위 권장.**
  2. **코퍼스에 하도급 어휘 변형 추가**(40 → ~80건) — 확실하나 데이터 증식.
  3. 개념당 예문 확대(5 → 10+).
  - `data/03_normalized/*.json`은 git 관리 정답 데이터라 이번 세션에선 변경하지 않음(분석만).

## 후속(미결) 항목

- **v3 골든셋 재제작** — `user_clause`를 조(條) 전체 텍스트로 전환(v1_review §6 최우선 항목
  재이행), NONE 라벨 엄격안 기준 전수 재검(g08류 오매핑 포함). 이거 없이는 어떤 임계값 튜닝도
  분류기 품질을 재고 있지 않음 — **다음 스텝의 필수 선행조건.**
- **1차/2차 판정 경계 정식 설계** — §3 스케치를 구현 스펙으로: `review_contract` 반환 계약 변경
  범위, grounding 패키지 스키마, LLM 호출 위치(데모 vs 별도 2차 MCP 도구) 확정. 동결 계약
  변경이라 사인오프 필요.
- **eval의 LLM 판정 검증 방식** — 결정론 규칙 #5와 어떻게 공존할지 설계(스냅샷 회귀 등 후보만
  거론, 미확정).
- **`min_pair_ratio` 가드 재검토** — 말바꿈이 강할수록 `detect_critical_changes` 검사가 스킵되는
  역설 수정 (강한 말바꿈 = paraphrase 함정의 핵심 케이스인데 방어선이 꺼짐).
- **독소 탐지 recall 개선** — 임계값 튜닝이 아니라 **코퍼스 당사자 어휘 편중** 해소가 핵심
  (§4 실측 규명: SW 어휘 전용 → SI/SM recall 0.077~0.43). 1순위는 매칭 전 당사자 용어 정규화,
  차선은 하도급 어휘 변형 추가. 데이터 변경이라 사인오프 필요.
